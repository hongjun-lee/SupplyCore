## 一、TL;DR — 给财务总监 5 min 介绍版

> SupplyCores 已为财务部 React 化 **10 页核心交易系统**（采购计划 / 三单匹配 / 资金计划 / 付款申请 / 5 类招标 / 需求计划 / 对账 / 入库审核）— **全部使用 mock 演示数据**

> **业务价值兑现**：财务方 1 对接人 + 反馈窗口 + 1 endpoint/周 deadline → **2-3 月内**核心交易系统全切真实数据 → **月结全手工 → 系统自动化**

> **关键阻塞链**：F-01 三单匹配 mock → 付款审核手工 → 月结手工 → **每月底财务部 3-5 天加班手工核对**

> **协调成本**：财务总监级 1 人对接 + 每周 1h 反馈会 + 紧急 1 日内响应 — **2-3 月投入 = 1 年回报**（月结时间从 5d → 1d / 财务部 2 人月时间释放）

## 二、10 页深度业务价值分析（按详设域分组）

### 2.1 三单匹配 + 付款核心（F 域 2 页 / 最高优先）

#### F-01 three-way-match（三单匹配）

**业务流**：采购订单 PO + 入库单 GRN + 发票 INV → 三方一致 → 付款审核通过

**mock 现状**：F-01 Controller 骨架已建（19j 19k）/ React UI 完整 / SeedData 演示 3 单匹配 / **业务规则未实施**：
- 数量公差（10% 默认 / 业务方可配置）
- 单价容差（5% 默认）
- 异常分支（数量超 / 价差大 / 入库时间倒挂）
- 工作流路由（按金额分级审批）

**mock → real 工时**：**L3 1-1.5 PD**（强业务规则 / 异常分支多 / 业务方反馈复杂度极高）

**业务方反馈需要**：
- 实际公差阈值（数量 / 单价）
- 异常分支处理流程（直接 reject / 重审 / 上级审批）
- 工作流路由规则（金额分级 / 多级会签）
- 报表口径（按 supplier / 按 period / 按 PO type）

**阻塞业务流**：
- 付款审核手工 → 月结手工
- F-02 付款申请依赖 F-01 通过

#### F-02 payment-request（付款申请）

**业务流**：F-01 三单匹配通过 → 财务发起付款申请 → 多级审批 → 银行付款指令

**mock 现状**：form-heavy + timeline UI 完整 / **业务规则未实施**：
- 付款方式（电汇 / 票据 / 信用证）
- 付款金额计算（部分付款 / 预付款抵扣）
- 审批流（金额分级）
- 银行对接（手工录入银行流水）

**mock → real 工时**：**L3 1-1.5 PD**（form-heavy / timeline / 多级审批）

**业务方反馈需要**：
- 付款审批金额阈值（如 ≤10W / 10W-50W / >50W 三级）
- 预付款抵扣规则
- 银行对接是否在 19q-19s 范围内（如否，留预留接口）

---

### 2.2 资金 + 对账（C 域 + INV_RECON / 2 页）

#### C-04 funding-plan（资金计划）

**业务流**：月度资金需求预算 → 实际付款执行 → 差异分析

**mock 现状**：React UI + Mock 月度计划数据 / **业务规则未实施**：
- 资金计划口径（部门 / 项目 / 供应商）
- 实际付款关联（F-02 完成 → 资金计划扣减）
- 差异预警（计划 vs 实际 ±X% 触发）

**mock → real 工时**：**L2 0.5-1 PD**（数据流相对简单 / 业务规则中等）

**业务方反馈需要**：
- 资金计划填报频率（月 / 季 / 半年）
- 实际付款关联粒度（按 PO / 按合同 / 按供应商）
- 差异预警阈值

#### reconciliation（三对一致对账 / INV_RECON）

**业务流**：月度物理库存 vs 系统库存 vs 财务库存 三方对账

**mock 现状**：React UI + Mock 对账数据 / **业务规则未实施**：
- 对账规则（差异容忍度 / 自动调账 / 人工调账）
- 物理盘点对接（与 S-06/S-07/mobile-stocktake 联动）
- 财务库存口径（成本核算）

**mock → real 工时**：**L2 0.5-1 PD**

**业务方反馈需要**：
- 月度对账时间（月底 / 月初哪天）
- 差异调账权限（仓库 vs 财务）
- 成本核算口径

---

### 2.3 采购计划链（P 域 / 2 页）

#### P-01 requirement-list（需求计划）

**业务流**：业务部门提交物料需求 → 采购员审核 → 形成采购订单

**mock 现状**：列表 UI + Mock 需求数据 / **业务规则未实施**：
- 需求审批流（部门预算检查 / 物料分类审批人）
- 需求合并（多部门同物料合并采购）
- 需求紧急度（紧急 / 普通 / 储备）

**mock → real 工时**：**L2 0.5-1 PD**

#### P-02/P-03 purchase-planning（采购计划编排）

**业务流**：需求计划 + 库存 → 采购编排 → 招标启动

**mock 现状**：编排 dashboard + Mock / **业务规则未实施**：
- 采购编排算法（按物料 / 按供应商 / 按交期）
- 招标启动条件（金额 ≥ X 触发招标 / 否则直接采购）
- 预算检查（财务部预算扣减）

**mock → real 工时**：**L2 0.5-1 PD**

---

### 2.4 招标 5 类（T 域 / 5 页 / 累计 ~3-4 PD）

T-01 公开招标 / T-02 邀请招标 / T-03 询比价 / T-04 单一来源 / T-05 竞争性谈判

**mock 现状**：5 个独立 React 页 / Mock 招标数据 / **业务规则未实施**：
- 5 类招标差异化流程
- 招标公告 / 投标人资格审核 / 评标 / 中标公示
- 与 supplier-performance（M-13）联动评分

**mock → real 工时**：**L2 0.5-1 PD 每页 × 5 = 2.5-5 PD**

**业务方反馈需要**：
- 5 类招标实际使用频率（哪类最常用 / 哪类极少 → 优先级排序）
- 评标规则（综合评分 / 最低价中标 / 技术分数权重）
- 中标后是否需要走 P-02 采购编排再下 PO

---

### 2.5 入库审核（S-05 / 1 页 / 财务关联）

#### purchase-receipt（入库审核 4 件事原子事务）

**业务流**：仓库验收入库 → 财务核对 → 触发应付款生成 → 库存调整

**mock 现状**：React UI + Mock 入库数据 / **4 件事原子事务未实施**：
- 入库单生成
- 应付款生成
- 库存数量调整
- 物料成本核算

**mock → real 工时**：**L3 1-1.5 PD**（事务核心 / 4 件事原子保证）

**业务方反馈需要**：
- 4 件事失败回滚策略
- 财务核对是否同步发生（vs 异步对账）
- 入库与采购订单关联（部分入库 / 多次入库）
