# 分批上线与基础数据采集计划 V0.2

**项目：** 阜矿物资供应管理系统 / SupplyCore
**日期：** 2026-05-17（V0.2 cici 反馈 6 点修订）
**定位：** 上线准备与业务推进计划
**核心策略：** 不一次性大上线，先上线基础数据采集与治理能力，再分批放开库存、采购、合同审批、财务与 NC 联调。

**V0.2 修订要点（cici 反馈）**：
- **#1 时间口径分层**：明确"第 1 阶段 1-2 周（数据治理）+ 第 2 阶段 3-4 周（库存试点）+ 第 3 阶段 5-7 周（采购合同）+ 第 4 阶段后续（财务 NC）"，避免对外汇报口径混淆（§零）
- **#2 组织与人员模板**：弱化填报，强化 Nova 同步主导，模板仅保留 3 类补充场景（详 [`数据采集模板/01-组织与人员模板-V0.2.md`](数据采集模板/01-组织与人员模板-V0.2.md)）
- **#5 试点单位推荐依据**：5 维度评分替换 dashboard mock HeightPct（§二第 2 批）
- **#6 文档入仓 + 跨仓链接修正**：原 V0.1 untracked / V0.2 入仓 + 链接同步指向 V0.2

## 一、总体判断

当前系统主体框架、核心服务、报表大屏、NC 凭证导出主线、部署 runbook 和质量治理已经具备阶段性基础，但正式上线的主要风险不在“有没有代码”，而在：

1. 基础数据是否完整、准确、可追溯；
2. 业务方是否确认字段、流程、职责和例外处理；
3. Mock / 演示数据是否逐步替换为真实业务数据；
4. NC 真接通是否具备业务与技术双方条件；
5. 上线后是否有问题台账、回退方案和责任机制。

因此，后续工作建议从“继续堆功能”转为“上线准备 + 数据治理 + 试点运行”。

## 二、分批上线总体路线

### 第 0 批：上线准备与数据模板定版（3～5 天）

**目标：** 明确第一批上线边界，把基础数据采集模板、责任人、校验规则准备好。

**主要工作：**

- 确认第一批上线范围：基础数据、数据采集、导入校验、查询维护、问题台账；
- 梳理基础数据清单与字段模板；
- 明确数据来源：Nova 平台、物资系统维护、NC/财务系统对照、人工采集；
- 明确各类数据责任部门与责任人；
- 准备 Excel/导入模板、字段说明、示例数据、错误反馈模板；
- 建立“数据问题台账”：缺失、重复、编码冲突、字段不规范、权威源不一致等。

**验收口径：**

- 模板齐全；
- 字段说明齐全；
- 责任人明确；
- 至少选定 1～2 个厂矿/部门作为第一批试点对象。

---

### 第 1 批：基础数据采集与试运行（1～2 周）

**目标：** 先把数据收上来、洗干净、能导入、能校验、能追责。

**建议先上线的数据范围：**

1. 组织与人员基础数据
   - 组织架构、部门、班组；
   - 人员、岗位、角色、权限范围；
   - 来源优先对接 Nova 平台，本地只做同步缓存与业务扩展字段。

2. 物资主数据
   - 物料分类；
   - 物料编码；
   - 物料名称、规格型号、计量单位；
   - 是否高敏感物资、是否设备备件、是否可采购、是否启用。

3. 供应商档案
   - 供应商名称、统一社会信用代码；
   - 联系信息、开户信息、资质信息；
   - 黑名单/准入状态；
   - 与 NC 供应商编码的对应关系。

4. 仓储基础数据
   - 仓库、库区、货位；
   - 仓库所属单位；
   - 仓库管理员；
   - 是否启用批次、序列号、有效期等管理。

5. 财务与 NC 映射数据
   - 部门编码映射；
   - 供应商编码映射；
   - 科目映射；
   - 业务单据类型与 NC 接口编码映射；
   - SC 编号与 NC 编号双号制规则。

6. 期初库存/库存余额
   - 物料、仓库、货位、数量、金额；
   - 批次、入库日期、供应商、项目/成本中心；
   - 盘点差异说明。

**系统侧需要配合完成：**

- 基础数据导入；
- 导入前校验；
- 导入错误下载；
- 重复数据识别；
- 编码冲突提示；
- 数据版本留痕；
- 数据质量看板。

**验收口径：**

- 试点单位基础数据导入成功率 ≥ 95%；
- 必填字段完整率 ≥ 98%；
- 编码重复率、供应商重复率、库存异常项形成台账；
- 数据责任部门确认“可作为后续业务上线基础”。

---

### 第 2 批：库存查询与基础单据试点（第 3～4 周）

**目标：** 在基础数据稳定后，先放开低风险、高价值的库存查询与基础流转能力。

**建议上线范围：**

- 库存查询；
- 库存流水；
- 到货/入库基础登记；
- 出库/领料基础登记；
- 调拨、退料等低风险流转；
- 库存报表与大屏基础指标；
- 数据质量看板持续运行。

**试点原则（V0.2 cici 反馈 #5 修订 5 维度评分依据）：**

V0.1 deprecated：仅说"选 1-2 个厂矿试点"，未明确选择依据，容易被业务方质疑。

V0.2 5 维度评分（替换 dashboard mock HeightPct）：

| 维度 | 说明 | 权重 |
|---|---|---|
| ① 数据基础好不好 | 现有物料 / 库存 / 供应商台账完整度 + 数字化程度 | 25% |
| ② 配合度高不高 | 业务方对接积极度 + IT 历史合作 | 20% |
| ③ 业务规模中等不太极端 | 月单量 + 物料种类 + 仓库数（不要最大也不要最小）| 20% |
| ④ 仓储/财务流程是否完整 | 是否有规范流程 + 财务月结配合度 | 20% |
| ⑤ 出问题影响面是否可控 | 影响半径 + 替代方案 + 回退难度 | 15% |

试点单位由 cici 与 PO 按 5 维度评分后正式指定（推荐艾友 + 本部待评分确认）。

试点过程：
- 先“系统记录 + 线下复核”并行；
- 暂不强行替代原有流程；
- 每周汇总问题清单。

**验收口径：**

- 试点仓库库存查询可用；
- 主要库存流转单据可记录、可查询、可追溯；
- 与线下台账差异可解释；
- 具备扩大试点条件。

---

### 第 3 批：采购计划、合同与审批流程试运行（第 5～7 周）

**目标：** 在主数据和库存数据基本稳定后，再进入业务流程闭环。

**建议上线范围：**

- 需求计划；
- 采购计划；
- 采购任务/采购订单；
- 合同管理；
- 审批流；
- 招采过程留痕；
- 供应商履约记录。

**关键前置条件：**

- 物料、供应商、仓库、组织人员、权限范围已经稳定；
- 审批角色与组织角色已确认；
- 业务方已确认核心单据字段；
- 试点单位可接受线上线下并行。

**验收口径：**

- 采购计划到合同审批的核心链路跑通；
- 审批节点、责任人、驳回/撤回流程可用；
- 单据状态可追踪；
- 关键操作有审计留痕。

---

### 第 4 批：财务、NC 联调与正式上线准备（第 8～10 周）

**目标：** 在业务数据和流程稳定后，再启动财务与 NC 真联调，避免脏数据冲击财务系统。

**建议上线范围：**

- 三单匹配；
- 付款申请；
- 凭证生成；
- SC/NC 双号制；
- NC 真凭证号自动回写；
- 推送失败重试；
- 对账与异常处理；
- NC 接口监控。

**启动条件：**

- 财务侧确认 F-3 12 列 Excel 与凭证字段；
- 业务方确认 G-12 NC 真号回写流程；
- NC 侧提供可联调环境、接口地址、认证方式、错误码说明；
- 当前凭证导出主线通过第二次 demo 验收。

**验收口径：**

- 至少 5 类核心 InterfaceCode 能够完成推送/回写/失败重试；
- NC 返回号能自动写回业务单据；
- 失败记录可查、可重推、可关闭；
- 财务与物资双方确认对账口径。

## 四、建议成立 4 条工作线

### A 线：基础数据与数据治理

- 负责人建议：物资公司牵头，网信办统筹；
- 参与：各厂矿、仓储、财务、供应商管理人员；
- 产出：模板、数据台账、导入结果、问题清单、数据质量报告。

### B 线：系统上线准备

- 负责人建议：技术团队牵头，网信办把关；
- 产出：部署 runbook、环境清单、权限配置、备份恢复、灰度上线 checklist。

### C 线：业务流程与 demo 验收

- 负责人建议：业务 PO 牵头；
- 产出：demo checklist、业务反馈单、字段确认表、流程确认表。

### D 线：NC/财务接口准备

- 负责人建议：财务 + NC 管理方 + 技术团队；
- 产出：接口地址、认证配置、错误码、联调计划、对账规则、回退方案。

## 五、近期不建议做的事

1. 不建议直接全流程上线；
2. 不建议在基础数据未清洗前强推采购/财务闭环；
3. 不建议业务方未确认字段时继续大量新增页面；
4. 不建议提前把 NC 真接通作为第一批上线范围；
5. 不建议只做演示，不做数据责任和问题台账。

## 六、建议汇报口径

> 下一阶段建议从“功能开发推进”切换到“上线准备和数据治理推进”。第一批不做大范围业务闭环，而是先上线基础数据采集、导入校验、数据质量管理和问题台账能力。用 1～2 周把组织、人员、物料、供应商、仓库、货位、NC 映射、期初库存等基础数据收集并清洗到位，再分批上线库存、采购、合同审批和财务 NC 联调，降低一次性上线风险。
