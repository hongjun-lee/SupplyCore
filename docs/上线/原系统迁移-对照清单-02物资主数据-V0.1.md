## 一、本清单是什么

02 物资主数据的迁移对照清单 — 你按下面的新表结构 + SQL 样例，对照原系统出一段 SELECT SQL 给我们即可。

**完整方案** / 协作模式 / 为什么用 SQL 不用 Excel / staging 流程 / 双轨编码 / 落库验证 — 见主方案 [`原系统迁移方案-V0.1.md`](./原系统迁移方案-V0.1.md)。本清单只列 02 的具体内容。

> 💡 **重要 — 你不需要管的事**（避免过度清洗）：
>
> 阜矿这次迁移是借机做一次**数据治理**，业务部门会在新系统里对数据**重新归类、调整属性**。所以：
>
> - ❌ 不要纠结**物料分类是否合理**（**业务部门会在 02 物料迁移之前先单独确认分类树并锁定** / 见 §三.1 提示 / 详见主方案 §八-A.8）
> - ❌ 不要纠结**高敏感 / 备件 / 可采购属性是否最新**（业务方会逐条核对）
> - ❌ 不要纠结**安全库存数值是否正确**（业务方会重新设定）
> - ❌ 不要试图**合并重复物料 / 删除废弃物料**（业务方在 staging 里决定）
> - ❌ 不要**补 NC 物料编码**（财务和 NC 管理方人工对照）
>
> ✅ **你只需要按原系统的现状原样抽出来**，业务方在新系统的 staging 区会做调整。
> ✅ **特别注意 1**：02 物料 SQL 启动前，**业务方已锁定新分类树 + 提供"原→新 分类编码映射 Excel"**。你抽物料时**输出 `original_category_code`（原系统的原分类编码）即可**，SupplyCore 接收后会 JOIN 业务方映射 Excel 拿到新分类。
> ✅ **特别注意 2**：**`material_code` 不需要你的 SQL 输出**（七轮调整 + 九轮修正）—— SupplyCore 会按集团《物资编码规范》V1.8 自动生成 `material_code = XX-XX-XXXXXX`（例 `ZH-01-000001` 支护材料-锚杆-第 1 号 / 大类拼音首字母 + 分类 2 位数字 + 流水 6 位）。你 SQL 里只需要输出 `legacy_code`（原物料编码）+ `original_category_code`（原分类编码）+ 其他字段。

## 三、新表结构清单（02 物资主数据 / 共 3 张表）

### 3.1 表一：物料分类（MaterialCategory）

> ⚠️ **物料分类的特殊工作流（与其他表不同 / 你也要参与映射）**：
>
> **项目组按集团《物资编码规范》V1.8 已经规划好分类**（14 大类 + 95 二类）。完整清单 +
> 你要做的映射工作详见独立文档：
>
> 👉 [`物料分类映射指南-V0.1.md`](./物料分类映射指南-V0.1.md)（**先做这个 / 02 物料 SQL 的前置依赖**）
>
> **关键工作步骤**（详见映射指南）：
> 1. **第一步（前置）**：你从原系统抽出所有分类编码 + 对照映射指南 §二 14 大类 + 95 二类清单 → 输出"原 → 新分类编码映射 Excel"（M-18 表的 6 列）
> 2. **第二步（本清单 §五.2 表二）**：02 物料 SQL 抽数时，输出 `legacy_code` + `original_category_code` + 其他字段
> 3. **SupplyCore 自动做**：JOIN M-18 映射表 → original_category_code → new_category_id → 按 V1.8 规范自动生成 `material_code = XX-XX-XXXXXX`
>
> **你不需要在 02 物料 SQL 里做 JOIN** — SupplyCore 内部用 M-18 表关联。你只管按映射指南填好 Excel + 抽好物料数据。
>
> 业务方核对路径 + 工程师归类提报机制（找不到归类时的边界处理）见
> [`物料分类基线-V0.1.md`](./物料分类基线-V0.1.md) §四。

**用途**：物料的分类层级（一级 / 二级 / 三级 / 四级），例如"电气设备 → 电缆电线 → 电力电缆"。

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 长度 | 必填 | 字段用途备注 |
|---|---|---|---|---|---|---|
| 1 | `category_code` | 分类编码 | 字符串 | 16 | ✅ | 唯一 / 例 `01` 电气设备 / `01-01` 电缆电线 |
| 2 | `category_name` | 分类名称 | 字符串 | 64 | ✅ | 中文显示名 |
| 3 | `parent_category_code` | 上级分类编码 | 字符串 | 16 | ⚠️ | 一级分类留空 / 其他填上级 code |
| 4 | `level` | 分类级别 | 整数 | - | ✅ | 1=一级 / 2=二级 / 3=三级 / 4=四级 |
| 5 | `is_high_sensitive` | 是否高敏感物资 | 布尔 | - | ✅ | 火工品 / 易燃易爆等填 true，其他填 false |
| 6 | `is_active` | 是否启用 | 布尔 | - | ⚠️ | 默认 true，停用的物料分类填 false |

**示例数据**（让你大致感受预期长什么样）：

| category_code | category_name | parent_category_code | level | is_high_sensitive |
|---|---|---|---|---|
| 01 | 电气设备 | | 1 | false |
| 01-01 | 电缆电线 | 01 | 2 | false |
| 02 | 火工品 | | 1 | true |
| 02-01 | 乳化炸药 | 02 | 2 | true |

### 3.2 表二：物料主数据（Material）⭐ 核心表

**用途**：每个物料的编码 / 名称 / 规格 / 计量单位 / 分类 / NC 对照编码 / 是否高敏感等核心信息。

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 长度 | 必填 | 字段用途备注 |
|---|---|---|---|---|---|---|
| 1 | ~~`material_code`~~ | ~~新系统物料编码~~ | - | - | ❌ | **SupplyCore 自动生成**（七轮调整）/ 你 SQL **不需要输出** |
| 2 | `legacy_code` | **原系统物料编码** ⭐ | 字符串 | 32 | ✅ | **直接取原系统的物料编码字段** / 用于双轨追溯 + 老查询兼容 |
| 3 | `material_name` | 物料名称 | 字符串 | 128 | ✅ | 例 `电力电缆` `安全帽` |
| 4 | `specification` | 规格型号 | 字符串 | 256 | ✅ | 例 `YJV22-3×95+1×50` |
| 5 | `unit` | 计量单位 | 字符串 | 16 | ✅ | 例 `米` `千克` `件` `卷` / 必须在表三 Unit 字典里 |
| 6 | `original_category_code` ⚠️ | **原系统的原分类编码** | 字符串 | 16 | ✅ | **七轮调整**：你 SQL 输出原分类编码，SupplyCore JOIN 业务方映射 Excel 转成 new_category_code |
| 7 | `nc_material_code` | NC 系统物料编码 | 字符串 | 32 | ⚠️ | **原系统如果存了 NC 编码，直接抽** / 没有则留空 |
| 8 | `is_high_sensitive` | 是否高敏感物资 | 布尔 | - | ⚠️ | 火工品 / 易燃易爆等 |
| 9 | `is_equipment_spare` | 是否设备备件 | 布尔 | - | ⚠️ | 设备的备件 / 维修件 |
| 10 | `is_purchasable` | 是否可采购 | 布尔 | - | ⚠️ | 默认 true |
| 11 | `is_active` | 是否启用 | 布尔 | - | ⚠️ | 默认 true / 已停用的填 false |
| 12 | `safety_stock` | 安全库存 | 小数 | - | ❌ | 没有就留空 |
| 13 | `remark` | 备注 | 字符串 | 256 | ❌ |  |

**关于双轨编码 + material_code 生成**（V0.1 七轮调整 / 请重点理解）：

| 字段 | 谁产生 | 内容 |
|---|---|---|
| `legacy_code` | **你的 SQL 输出** | 原系统的物料编码（直接抽）|
| `original_category_code` | **你的 SQL 输出** | 原系统的原分类编码（直接抽）|
| `category_code`（新）| **SupplyCore 内部** | JOIN 业务方映射 Excel：`original_category_code → new_category_code`（规范 V1.8 的 4 位分类码 如 `ZH-01`）|
| `material_code`（新）| **SupplyCore 内部** | 按规范 V1.8 自动生成：`XX-XX-XXXXXX`（例 `ZH-01-000001`）|

**为什么这么设计**：业务方在阶段 0 重新规划了大类-小类编码，原 `legacy_code` 的分类前缀已经对不上新分类了。所以 `material_code` 不能简单 = `legacy_code`，必须按新分类规则重新生成。这部分逻辑 SupplyCore 内部处理，**不用你操心**。

所以你的 SQL 简化为：

```sql
SELECT
  原系统物料编码字段 AS legacy_code,             -- 双轨追溯用
  原系统分类编码字段 AS original_category_code,  -- 给 SupplyCore JOIN 业务方映射用
  原系统物料名称字段 AS material_name,
  ...
```

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 必填 | 字段用途备注 |
|---|---|---|---|---|---|
| 1 | `unit_code` | 单位编码 | 字符串 | ✅ | 唯一 / 例 `M` `KG` `T` `件` `卷` |
| 2 | `unit_name` | 单位中文名 | 字符串 | ✅ | 例 `米` `千克` `吨` |
| 3 | `unit_type` | 单位类型 | 枚举 | ✅ | `长度` / `重量` / `体积` / `数量` / `其他` |

| M | 米 | 长度 | M | 1 |

## 四、SQL 输出规范（重要 — 请按这个格式写）

### 4.1 核心约定

**每个 SELECT 字段都用 `AS` 起新表英文字段为别名**，这样我们接收结果集后零再加工。

✅ 正确示例：

```sql
SELECT
  m.MAT_CODE          AS legacy_code,
  m.MAT_CODE          AS material_code,
  m.MAT_NAME          AS material_name,
  m.SPEC              AS specification,
  ...
FROM 物料表 m
WHERE 组织字段 IN ('001.007.001', '001.007.002', '001.007.018');
```

❌ 不要这样：

```sql
SELECT MAT_CODE, MAT_NAME, SPEC FROM 物料表;   -- 字段名是原系统的，我们要再做映射，容易出错
```

### 4.2 SQL 样例（你照这个模板改）

**表一：物料分类**

```sql
SELECT
  分类编码字段          AS category_code,
  分类名称字段          AS category_name,
  上级分类编码字段       AS parent_category_code,
  分类级别字段          AS level,
  是否高敏感字段        AS is_high_sensitive,
  CASE WHEN 启用状态字段 = '启用值' THEN 1 ELSE 0 END AS is_active
FROM 物料分类表
WHERE 启用状态字段 IN ('启用值', '停用值');   -- 全量抽 / 启用 + 停用都要
```

**表二：物料主数据**（试点 3 单位过滤 / V0.1 七轮调整：material_code 不输出）

```sql
SELECT DISTINCT
  m.原物料编码字段       AS legacy_code,              -- 双轨追溯
  m.物料名称字段        AS material_name,
  m.规格型号字段        AS specification,
  m.计量单位字段        AS unit,
  m.原分类编码字段       AS original_category_code,   -- ⚠️ 七轮调整：输出原分类编码 / SupplyCore JOIN 业务方映射转新分类
  m.NC编码字段          AS nc_material_code,         -- 如果原系统没存 NC 编码，输出 NULL
  m.高敏感字段          AS is_high_sensitive,
  m.备件标识字段        AS is_equipment_spare,
  m.可采购字段          AS is_purchasable,
  CASE WHEN m.启用状态 = '启用值' THEN 1 ELSE 0 END AS is_active,
  m.安全库存字段        AS safety_stock,
  m.备注字段           AS remark
  -- ⚠️ 七轮+九轮调整：material_code 不需要你输出（SupplyCore 按规范 V1.8 自动生成 XX-XX-XXXXXX，例 ZH-01-000001）
FROM 物料主数据表 m
JOIN 库存表 i ON i.物料编码字段 = m.原物料编码字段
WHERE i.组织字段 IN ('001.007.001', '001.007.002', '001.007.018');
```

> **关于 JOIN 库存表**：这是为了过滤"试点 3 单位实际有库存或用过的物料"，避免抽出大量历史无用物料。
> 如果你的原系统物料表本身没有组织字段、需要从其他业务表反查 → JOIN 哪张表你最熟，按方便的写。

**表三：计量单位字典**

```sql
SELECT
  单位编码字段          AS unit_code,
  单位中文名字段        AS unit_name,
  单位类型字段          AS unit_type,
  基础单位字段          AS base_unit_code,
  换算系数字段          AS conversion_factor
FROM 计量单位字典表;   -- 全量
```

## 六、新系统会做的校验（提前告知 / 不影响你出 SQL）

我们拿到数据后会做这些校验，**有问题不是退回给你重写 SQL，而是我们和业务方一起处理**。所以你不用提前清洗：

| 校验项 | 失败处理 |
|---|---|
| material_code / legacy_code 重复 | 记台账 / 业务方人工确认是否合并 |
| 必填字段缺失（name / unit / category_code）| 记台账 / 业务方补 |
| nc_material_code 缺失 | 记 NC 映射台账 / 我们和财务 + NC 管理方对照 |
| 计量单位不在 Unit 字典 | 记台账 / 业务方补字典 |
| 高敏感属性与分类不一致 | 警告 / 业务方确认 |

## 七、关于 Excel 备选

如果某张表实在不方便写 SQL（例如要跨多个数据库 JOIN / 字段散在十几张表），可以单张表退回到 Excel：

1. 按"§三 新表结构清单"的字段顺序，导出一份 Excel
2. 第一行是字段英文名（material_code / material_name ...）
3. 数据从第二行开始
4. 在 §五 字段对照说明里标注"该表走 Excel 备选"

但**优先 SQL** — 后续业务在跑、数据要更新、要做新旧并行对账，SQL 都比 Excel 省事。
