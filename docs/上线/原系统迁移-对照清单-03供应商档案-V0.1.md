## 一、本清单是什么

03 供应商档案的迁移对照清单 — 跟 02 同一套模式（5 张表 / 你写 5 段 SELECT SQL 给我们）。

**完整方案 / 协作模式 / 为什么用 SQL 不用 Excel / staging 流程** — 见主方案 [`原系统迁移方案-V0.1.md`](./原系统迁移方案-V0.1.md)。本清单只列 03 的差异点 + 5 张表的字段。

> 03 跟 02 唯一显著差异：**资质有效期会随时间变化**（资质到期要预警），所以"SQL 可重复执行"在 03 上比 02 更重要 — 我们会周期性 re-run 拿最新的 `expire_date` 状态。

> 💡 **重要 — 你不需要管的事**（避免过度清洗）：
>
> 阜矿这次迁移是借机做一次**数据治理**，业务部门会在新系统里对供应商数据**重新审视、调整状态**。所以：
>
> - ❌ 不要纠结**准入状态是否最新**（业务方会重新审视黑名单 / 暂停 / 待审核）
> - ❌ 不要**清理黑名单 / 暂停的供应商**（业务方在新系统决定是否解禁 / 维持）
> - ❌ 不要试图**合并重复供应商**（业务方在 staging 决定怎么合）
> - ❌ 不要**清理过期资质**（业务方会重新审视 / 临期预警）
> - ❌ 不要纠结**联系人 / 开户信息是否最新**（业务方核对）
> - ❌ 不要**补 NC 供应商编码**（如果原系统没存，整张表 NcSupplierMapping 跳过即可）
>
> ✅ **你只需要按原系统的现状原样抽出来**，业务方在新系统的 staging 区会做调整。

## 二、抽取范围说明（全集团 / 与 02 略有不同）

> **V0.1 十一轮（网信办 2026-05-19）**：抽取范围为阜新矿业集团全部单位（OrgCode 前缀 `001.007.*`），不再限试点 3 单位。

**供应商通常不按 OrgCode 分**（一个供应商可能服务多个矿），所以过滤方式有两种，**任选其一**：

### 方式 1：抽全集团实际用过的供应商（推荐 / 过滤历史无用供应商）

```sql
-- 通过业务单据（采购订单 / 入库单 / 合同 等）反查
WHERE supplier_code IN (
  SELECT DISTINCT 供应商编码字段
  FROM 业务单据表
  WHERE 组织字段 LIKE '001.007.%'
)
```

全集团过滤：`组织字段 LIKE '001.007.%'`（阜新矿业集团全部单位 / 不再枚举试点 3 单位）

### 方式 2：全量抽（如果方式 1 反查太麻烦）

直接抽所有有效供应商，反正供应商总数也不大（预估 200-500 家）。

**告诉我们你用哪个方式即可**，两种我们都接得住。

## 三、新表结构清单（03 供应商档案 / 共 5 张表）

### 3.1 表一：供应商基础信息（Supplier）⭐ 核心表

**用途**：每个供应商的基本信息 / 营业执照 / 法人 / 经营范围 / 准入状态。

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 长度 | 必填 | 字段用途备注 |
|---|---|---|---|---|---|---|
| 1 | `supplier_code` | 新系统供应商编码 | 字符串 | 32 | ✅ | **第 1 阶段直接 = legacy_code**（沿用原编码）|
| 2 | `legacy_code` | **原系统供应商编码** ⭐ | 字符串 | 32 | ✅ | 直接取原系统的供应商编码字段 |
| 3 | `supplier_name` | 供应商名称 | 字符串 | 256 | ✅ | 应与营业执照一致 |
| 4 | `unified_credit_code` | 统一社会信用代码 | 字符串 | 18 | ✅ | 18 位 / 唯一 |
| 5 | `legal_representative` | 法定代表人 | 字符串 | 64 | ✅ |  |
| 6 | `registered_capital` | 注册资本（万元）| 小数 | - | ⚠️ |  |
| 7 | `registered_address` | 注册地址 | 字符串 | 256 | ⚠️ |  |
| 8 | `business_scope` | 经营范围 | 字符串 | 1024 | ⚠️ |  |
| 9 | `supplier_type` | 供应商类型 | 枚举 | 32 | ✅ | `制造商` / `经销商` / `代理商` / `服务商` / `其他` |
| 10 | `access_status` | 准入状态 | 枚举 | 16 | ✅ | `待审核` / `已准入` / `暂停` / `黑名单` |
| 11 | `access_date` | 准入日期 | 日期 | - | ⚠️ |  |
| 12 | `blacklist_reason` | 黑名单原因 | 字符串 | 256 | ⚠️ | 仅黑名单时填 |
| 13 | `is_active` | 是否启用 | 布尔 | - | ⚠️ | 默认 true / 停用填 false |
| 14 | `remark` | 备注 | 字符串 | 512 | ❌ |  |

| 2 | `contact_name` | 联系人姓名 | 字符串 | ✅ |  |
| 3 | `contact_phone` | 联系电话 | 字符串 | ✅ | 手机号 |
| 4 | `contact_email` | 邮箱 | 字符串 | ⚠️ |  |

### 3.3 表三：供应商开户信息（SupplierBank）

**用途**：每个供应商的银行账户（一供应商可多账户）。

### 3.4 表四：供应商资质（SupplierQualification）⭐ 关键

**用途**：供应商的资质证书（营业执照 / 行业资质 / ISO 等），新系统会按 expire_date 做临期预警。

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 必填 | 字段用途备注 |

| 1 | `supplier_code` | 供应商编码 | 字符串 | ✅ |  |
| 2 | `qualification_type` | 资质类型 | 枚举 | ✅ | `营业执照` / `税务登记` / `行业资质` / `ISO` / `安全许可` / `其他` |
| 3 | `qualification_no` | 证书编号 | 字符串 | ✅ |  |
| 4 | `issue_date` | 颁发日期 | 日期 | ⚠️ |  |
| 5 | `expire_date` | 到期日期 | 日期 | ⚠️ | **重要** — 新系统每天 cron 检查 / 30 天临期预警 |
| 6 | `issuing_authority` | 颁发机构 | 字符串 | ⚠️ |  |
| 7 | `attachment_url` | 扫描件 URL | 字符串 | ❌ | 原系统如果存了文件路径，抽过来；没有就空 |

### 3.5 表五：NC 供应商映射（NcSupplierMapping）⭐ 关键

**用途**：SupplyCore 供应商编码 → NC 系统供应商编码的对照关系（双号制核心）。

| 序号 | 新表字段（英文）| 中文含义 | 类型 | 必填 | 字段用途备注 |
|---|---|---|---|---|---|
| 1 | `supplier_code` | SupplyCore 供应商编码 | 字符串 | ✅ | 引用表一 |
| 2 | `nc_supplier_code` | NC 系统供应商编码 | 字符串 | ✅ | **原系统如果存了 NC 编码，直接抽** |
| 3 | `mapping_status` | 映射状态 | 枚举 | ✅ | `已映射` / `待确认` / `冲突` |
| 4 | `mapping_notes` | 备注 | 字符串 | ❌ | 冲突原因 / 历史合并等 |

**如果原系统没存 NC 编码**：把表五**整张表跳过**即可（出 4 张表就行），我们会通过 05 NC 映射模板由财务和 NC 管理方人工对照补。

## 四、SQL 输出规范 + 样例

### 4.1 核心约定（同 02）

每个 SELECT 字段都用 `AS` 起新表英文字段为别名。

### 4.2 SQL 样例

**表一：供应商基础信息**（方式 1 — 反查全集团 `001.007.%` 用过的供应商）

```sql
SELECT
  s.原供应商编码字段     AS legacy_code,
  s.原供应商编码字段     AS supplier_code,
  s.供应商名称字段       AS supplier_name,
  s.统一信用代码字段     AS unified_credit_code,
  s.法人字段            AS legal_representative,
  s.注册资本字段         AS registered_capital,
  s.注册地址字段         AS registered_address,
  s.经营范围字段         AS business_scope,
  s.供应商类型字段       AS supplier_type,
  s.准入状态字段         AS access_status,
  s.准入日期字段         AS access_date,
  s.黑名单原因字段       AS blacklist_reason,
  CASE WHEN s.启用状态 = '启用值' THEN 1 ELSE 0 END AS is_active,
  s.备注字段            AS remark
FROM 供应商表 s
WHERE s.原供应商编码字段 IN (
  SELECT DISTINCT 供应商编码字段
  FROM 采购订单表  -- 或入库单表 / 合同表 / 哪张业务表方便用哪张
  WHERE 组织字段 LIKE '001.007.%'
);
```

如果方式 2（全量抽）更方便，去掉 WHERE 子查询即可。

**表二：联系人**

```sql
SELECT
  c.原供应商编码字段     AS supplier_code,
  c.联系人姓名字段       AS contact_name,
  c.联系电话字段         AS contact_phone,
  c.邮箱字段            AS contact_email,
  c.职位字段            AS position,
  CASE
    WHEN ROW_NUMBER() OVER (PARTITION BY c.原供应商编码字段 ORDER BY c.id) = 1 THEN 1
    ELSE 0
  END                  AS is_primary  -- 原系统没标主次时，每供应商第一条标 true
FROM 供应商联系人表 c
WHERE c.原供应商编码字段 IN ( /* 同表一过滤逻辑 */ );
```

> 如果原系统的窗口函数语法不一样，或者已经标记了主联系人 → 按你的实际写法。

**表三：开户信息**

```sql
SELECT
  b.原供应商编码字段     AS supplier_code,
  b.开户银行字段        AS bank_name,
  b.分支行字段          AS bank_branch,
  b.账号字段            AS account_no,
  b.户名字段            AS account_name,
  b.SWIFT字段           AS swift_code,
  CASE
    WHEN ROW_NUMBER() OVER (PARTITION BY b.原供应商编码字段 ORDER BY b.id) = 1 THEN 1
    ELSE 0
  END                  AS is_primary
FROM 供应商开户信息表 b
WHERE b.原供应商编码字段 IN ( /* 同表一过滤逻辑 */ );
```

**表四：资质**

```sql
SELECT
  q.原供应商编码字段     AS supplier_code,
  q.资质类型字段        AS qualification_type,
  q.证书编号字段        AS qualification_no,
  q.颁发日期字段        AS issue_date,
  q.到期日期字段        AS expire_date,
  q.颁发机构字段        AS issuing_authority,
  q.附件路径字段        AS attachment_url
FROM 供应商资质表 q
WHERE q.原供应商编码字段 IN ( /* 同表一过滤逻辑 */ );
```

**表五：NC 供应商映射**（原系统有 NC 编码才需要）

```sql
SELECT
  s.原供应商编码字段     AS supplier_code,
  s.NC供应商编码字段     AS nc_supplier_code,
  '已映射'              AS mapping_status,   -- 原系统已存对照的，默认 已映射
  ''                    AS mapping_notes
FROM 供应商表 s
WHERE s.NC供应商编码字段 IS NOT NULL
  AND s.原供应商编码字段 IN ( /* 同表一过滤逻辑 */ );
```

## 六、新系统会做的校验（提前告知）

| 校验项 | 失败处理 |
|---|---|
| supplier_code / legacy_code 重复 | 记台账 / 业务方人工确认 |

| nc_supplier_code 缺失 | 记 NC 映射台账 / 财务 + NC 管理方对照 |
