## 一、本清单是什么

06 期初库存的迁移对照清单 — 跟 02 / 03 / 04 同一套模式（1 张大表 / 20 字段 / **全集团十万行级**）。

**这是 6 类基础数据里数据量最大的一张表，SQL 是唯一可行路径**（人工填 Excel 不现实）。

**完整方案 / 协作模式 / staging 流程** — 见主方案 [`原系统迁移方案-V0.1.md`](./原系统迁移方案-V0.1.md)。本清单只列 06 的具体内容（**§二 期初库存 5 个特殊点请先看**）。

> 💡 **重要 — 你不需要管的事**（避免过度清洗）：
>
> 阜矿这次迁移是借机做一次**数据治理**，期初库存会由业务部门在新系统做**盘点确认 + 财务对账**。所以：
>
> - ❌ 不要纠结 **stock_state 状态是否准确**（业务方盘点后会逐条确认正常 / 待检 / 隔离 / 报废）
> - ❌ 不要**清理零库存或负库存**（SQL 里加 `数量 > 0` 过滤即可，不要为此做更多清洗）
> - ❌ 不要**补单价 / 金额**（财务后续按 P 平均成本 / 加权 / 移动平均计算）
> - ❌ 不要**清理报废库存**（业务方在 staging 里识别报废 + 调整 stock_state）
> - ❌ 不要**修正盘点差异**（adjustment_note 由仓储后续补 / 你留空即可）
> - ❌ 不要纠结**入库日期 / 供应商 / 批次** 是否齐全（缺失的字段空着就行，业务方核对）
>
> ✅ **你只需要按原系统的当前库存状态原样抽出来**，业务方在新系统的 staging 区会做调整。

## 二、期初库存的几个特殊点（请先看这个）

### 2.1 SQL 可以先出，但导入要"后置"

期初库存表里每一行都引用：
- 物料识别码（`material_code` 字段可填原系统旧物料编码 → 关联 02 的 `legacy_code` → 新 `material_code`）→ 必须在 02 物资主数据先落库
- 仓库 / 库区 / 货位（warehouse_code / zone_code / location_code）→ 必须在 04 仓储先落库
- 供应商编码（supplier_code，可空）→ 03 供应商落库后填

**给你的建议**：
- **SQL 现在就可以写**（不用等 02/03/04）
- **跑 SQL 抽数也可以现在做**（拿到结果集存着即可）
- **真正灌进新系统**等 02/03/04 落库后再说

### 2.2 财务字段不强求（原系统可能没全）

期初库存有 `unit_price` 单价 / `total_amount` 金额 两个财务字段：

- 原系统**有**：直接抽
- 原系统**没有 / 不全**：留空 / 财务后续按 P（平均成本 / 加权 / 移动平均）补全

**不要为了让数据"完整"而拍脑袋填一个单价**。留空比错值好。

### 2.3 盘点状态 / 来源 字段是新概念

`stock_state` 盘点状态（正常 / 待检 / 隔离 / 报废）和 `source` 数据来源是新系统才有的概念。原系统通常没有这些字段：

- `stock_state` 默认全部填 **`正常`** 即可
- `source` 默认全部填 **`上一系统迁移`** 即可
- 后续物理盘点发现差异由仓储补 `adjustment_note`

### 2.4 批次 / 序列号 / 有效期 看仓库设置

如果原系统有这些字段就抽，没有就留空 — 我们按仓库的 `enable_batch / enable_serial / enable_expiry` 开关（04 表一）判断哪些是必填。

### 2.5 高敏感物资（火工品 / 易燃易爆）的特殊处理

火工品 / 易燃易爆物资的期初库存：
- 我们会严格校验：必须有 batch_no + expiry_date
- 缺这两个字段的 → 不会拒绝导入，但会记台账 + 严重度=极高 / 业务方必处理

**你能抽就抽，不能就空 — 我们不会让你为这个特殊处理**。

```sql
JOIN 仓库表 w ON w.原仓库编码字段 = i.原仓库编码字段
```

## 四、新表结构清单（06 期初库存 / 1 张大表 / 20 字段）

> ⚠️ **V0.1 复评修正 — 06 模板字段名与导入实现统一**：
>
> 因 02 物资主数据调整：新 `material_code` 按集团《物资编码规范》V1.8 自动生成（数据库/API 存 `ZH01000001`，页面展示 `ZH-01-000001`），与 06 SQL 抽出的"原物料编码"不一致。
>
> 但为与 06 数据采集模板和 `InitialStockImportHandler` 保持一致，06 SQL 仍输出字段名 **`material_code`**；字段值可以填新系统编码，也可以直接填原系统旧物料编码（即 02 的 `legacy_code`）。SupplyCore 导入时会先按 `Material.material_code` 查找，未命中再按 `Material.legacy_code` 查找，并最终落到新的 `material_code`。
>
> 也就是说：**字段名用 `material_code`，字段值可用旧码**；不要再另起 `legacy_material_code` 字段，避免模板、代码和工程师清单不一致。

**用途**：当前每个仓库 / 货位上各物料的实时库存（数量 / 金额 / 批次 / 供应商 / 入库时间）。

## 五、SQL 输出规范 + 样例

### 5.1 核心约定（同 02）

每个 SELECT 字段都用 `AS` 起新表英文字段为别名。

### 5.2 SQL 样例

```sql
SELECT
  i.原物料编码字段           AS material_code,           -- 可直接输出旧物料编码；SupplyCore 按 material_code / legacy_code 双路径反查
  m.物料名称字段            AS material_name,            -- JOIN 物料表取
  i.原仓库编码字段           AS warehouse_code,
  i.库区编码字段            AS zone_code,                -- 没有库区概念就 NULL
  i.货位编码字段            AS location_code,             -- 没有货位概念就 NULL
  i.批次号字段              AS batch_no,
  i.序列号字段              AS serial_no,
  i.数量字段                AS quantity,
  m.计量单位字段            AS unit,                      -- JOIN 物料表取（与物料 unit 一致）
  i.单价字段                AS unit_price,                -- 没有就 NULL
  i.金额字段                AS total_amount,              -- 没有就 NULL
  i.入库日期字段            AS inbound_date,
  i.有效期字段              AS expiry_date,
  i.供应商编码字段           AS supplier_code,
  i.项目编码字段            AS project_code,
  i.成本中心字段            AS cost_center_code,
  '正常'                   AS stock_state,                -- 全部默认正常
  NULL                     AS adjustment_note,            -- 留空
  CURRENT_DATE             AS report_date,                -- SQL 跑数当天日期
  '上一系统迁移'            AS source                      -- 全部固定值
FROM 库存表 i
JOIN 物料表 m ON m.原物料编码字段 = i.原物料编码字段
JOIN 仓库表 w ON w.原仓库编码字段 = i.原仓库编码字段
WHERE w.组织字段 LIKE '001.007.%'
  AND i.数量字段 > 0;   -- 只抽有库存的，零库存的不要
```

> `CURRENT_DATE` 在不同数据库语法可能不一样（Oracle 用 `SYSDATE` / SQL Server 用 `GETDATE()`），按你的实际方言写。

## 七、新系统会做的校验（提前告知 / 严格但宽容）

| 校验项 | 失败处理 |
|---|---|
| material_code 在 02 Material.material_code / legacy_code 均查不到 | 错误 / 等 02 落库后通过新编码或 legacy_code 关联补 |
| warehouse_code 在 04 仓库查不到 | 错误 / 等 04 落库后回填 |
| quantity ≤ 0 | 错误 / 不导入（SQL 里已经过滤了） |
| unit 与物料的 unit 不一致 | 警告 / 业务方核对 |
| 仓库启批次 but batch_no 空 | 警告 / 记台账 / 业务方补 |
| 高敏感物资缺批次 / 有效期 | **严重度极高** / 记台账 / 业务方必处理 |
| total_amount ≠ quantity × unit_price | 警告 / 财务核对 |
| 库存对账总额与原系统不符 | 抽样核查 |

**重要原则**：**校验失败不是退回让你重写 SQL**，是新系统这边记台账由业务方处理。你不需要为了"数据干净"过度清洗。

## 八、抽数后的对账（关键）

期初库存的对账是**最严格的**，建议你抽数完成后顺便跑两段对账 SQL 给我们：

```sql
-- 对账 1：按仓库汇总数量
SELECT 仓库编码, 物料编码, SUM(数量) AS 数量合计
FROM 库存表
WHERE 仓库.组织字段 LIKE '001.007.%'   -- 全集团
GROUP BY 仓库编码, 物料编码;

-- 对账 2：按仓库汇总金额（如果有单价）
SELECT 仓库编码, SUM(数量 × 单价) AS 金额合计
FROM 库存表
WHERE 仓库.组织字段 LIKE '001.007.%'   -- 全集团
GROUP BY 仓库编码;
```

这两段对账的结果集，我们会和落库后从 SupplyCore 算的同样汇总做 diff，偏差 > 1% 必须解释。
