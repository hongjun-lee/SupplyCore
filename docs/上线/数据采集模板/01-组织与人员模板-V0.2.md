# 01 组织与人员数据采集模板 V0.2（2026-05-17 / cici 反馈 #2 弱化填报 强化 Nova 同步）

**目的**：第 1 批基础数据采集 / 组织架构 + 人员 + 岗位 + 角色 + 权限范围
**责任部门**：物资公司（确认）+ 各厂矿（**仅补业务联系人和数据责任人**）
**核心策略**（V0.2 修订）：**Nova 平台同步为主，模板填报为辅**
**校验**：Nova 同步差异检测 + 业务字段补全校验 + 异常申报

---

## 零、V0.2 修订（cici 反馈 #2）

### 0.1 V0.1 deprecated 反模式

原 V0.1 让各单位填报组织 + 人员 + 角色 + 权限的完整三 Sheet 模板（含 org_code / employee_no / role / scope 等）— **与集团统一组织人员主数据打架**。

具体问题：
1. 组织架构和员工台账已经在 Nova 集团统一维护，二次填报必然版本不一致
2. 各单位手填 org_code 与 Nova 实际编码冲突 → 同步上线时 90% 行需人工对照修正
3. 角色 / 权限属于系统职责（应由 Nova 同步 + 业务 PO 在系统内配置），不应让各单位"填"

### 0.2 V0.2 修订策略：Nova 同步主导 / 模板仅 3 类补充

**3 类有限补充场景**：

| 场景 | 数据范围 | 责任 | 工作量 |
|---|---|---|---|
| **A 补业务联系人** | 各厂矿对接物资 / 财务 / 仓储的接口人电话邮箱 | 各厂矿 | 0.5 天 |
| **B 确认数据责任人** | 各模板（02-06）的填报责任人 + 处理责任人 | 物资公司 | 0.5 天 |
| **C 申报 Nova 缺失或错误** | 组织缺失节点 / 员工不在 Nova / 调岗未同步 | 物资公司 → Nova 团队 | 1 天 |

**总工作量**：1-2 天（V0.1 是 3-5 天，因为大段填报 deprecated）。

### 0.3 V0.2 实施流程

```
Nova 同步先行 ──→ 系统读取组织 + 人员 + 默认角色（自动）
                       │
                       ▼
              系统侧 diff 报表（缺失 / 不一致 / 调岗）
                       │
                       ▼
            模板 A / B / C 补充（仅 1-2 天）
                       │
                       ▼
              业务 PO 在系统内配置 scope / 特殊权限
```

## 一、组织架构（Sheet 1: Organization）— **V0.2 弱化 / Nova 同步主导**

> **V0.2 重要说明**：本 Sheet 仅作为 Nova 同步**异常申报**用，不要求各单位重新填报。各单位应在 Nova 平台维护组织数据，本系统通过 NovaOrganizationSyncContributor 自动同步。**仅当**发现 Nova 缺失 / 错误时，用本 Sheet 申报。

### 1.1 字段清单

| # | 字段（中文）| 字段（英文）| 类型 | 长度 | 必填 | 说明 / 校验 |
|---|---|---|---|---|---|---|
| 1 | 组织编码 | org_code | string | 32 | ✅ | 唯一 / 字母数字 / 例 `FK001` |
| 2 | 组织名称 | org_name | string | 128 | ✅ | 例 `阜矿集团本部` |
| 3 | 组织类型 | org_type | enum | 16 | ✅ | 集团 / 厂矿 / 部门 / 班组 / 仓储 |
| 4 | 上级组织编码 | parent_org_code | string | 32 | ⚠️ | 顶层组织留空 / 否则必须在本表存在 |
| 5 | 组织级别 | org_level | int | - | ⚠️ | 1=集团 / 2=厂矿 / 3=部门 / 4=班组 |
| 6 | 是否启用 | is_active | bool | - | ⚠️ | 默认 true |
| 7 | 描述 | description | string | 256 | ❌ | 备注 |

### 1.2 示例数据

| org_code | org_name | org_type | parent_org_code | org_level | is_active |
|---|---|---|---|---|---|
| FK000 | 阜矿集团 | 集团 | | 1 | true |
| FK001 | 本部 | 厂矿 | FK000 | 2 | true |
| FK002 | 艾友矿 | 厂矿 | FK000 | 2 | true |
| FK002-WL | 物流仓库 | 仓储 | FK002 | 3 | true |

### 1.3 校验规则

- `org_code` 唯一性（跨整表）
- `parent_org_code` 必须在本表存在（除顶层）
- `org_type` 必须为枚举值之一
- `org_level` 必须与 `parent_org_code` 对应级别相关（顶层 1 / 下一级 2 / ...）

## 二、人员（Sheet 2: User）— **V0.2 仅补业务联系人**

> **V0.2 重要说明**：员工台账由 Nova 统一维护，本 Sheet 仅用于"补业务联系人 + 确认数据责任人"两类场景。**不要重新填**整个员工表。系统通过 NovaUserSyncContributor 自动同步基础信息。

### 2.1 字段清单

| # | 字段（中文）| 字段（英文）| 类型 | 长度 | 必填 | 说明 / 校验 |
|---|---|---|---|---|---|---|
| 1 | 工号 | employee_no | string | 32 | ✅ | 唯一 / 字母数字 / 例 `FK0001` |
| 2 | 姓名 | name | string | 64 | ✅ |  |
| 3 | 手机号 | phone | string | 16 | ⚠️ | 13/14/15/17/18/19 开头 11 位数字 |
| 4 | 邮箱 | email | string | 128 | ⚠️ | RFC 5322 / 与公司邮箱一致 |
| 5 | 所属组织编码 | org_code | string | 32 | ✅ | 必须在 Organization sheet 中 |
| 6 | 岗位 | position | string | 64 | ✅ | 例 `仓库管理员` / `物资专员` |
| 7 | 角色 | role | enum | 32 | ✅ | admin / 物资 / 财务 / 仓储 / 审批 / 浏览 |
| 8 | 权限范围 | scope | string | 256 | ⚠️ | JSON / 例 `{"org_codes":["FK001","FK002"]}` |
| 9 | 入职日期 | hire_date | date | - | ❌ | YYYY-MM-DD |
| 10 | 状态 | status | enum | 16 | ⚠️ | 在职 / 离职 / 调动 / 试用 |
| 11 | 备注 | remark | string | 256 | ❌ |  |

### 2.2 示例数据

| employee_no | name | phone | email | org_code | position | role | scope | status |

| FK0001 | 李建颖 | 13900000001 | lijianying@fk.com | FK000 | 财务对接 | 财务 | {"org_codes":["FK000"]} | 在职 |
| FK0002 | 汤云龙 | 13900000002 | tangyunlong@fk.com | FK001 | 物资对接 | 物资 | {"org_codes":["FK001"]} | 在职 |

### 2.3 校验规则

- `employee_no` 唯一性
- `org_code` 必须在 Organization sheet 中
- `role` 必须为枚举值
- `scope` 必须是合法 JSON / org_codes 必须在 Organization sheet 中
- `phone` 格式 / `email` 格式
- 同一 `org_code` 至少 1 个 `role=admin` 或上级组织有 admin

## 四、导入与错误反馈

### 4.1 系统导入流程

1. cici / 物资公司管理员登录系统 → 数据导入页面
2. 选择类型："组织与人员"
3. 上传 Excel（含 3 sheet：Organization / User / Role）
4. 系统校验（< 10s 完成 / 1000 行内）
5. 显示结果：
   - 成功导入行数
   - 校验失败行数 + 错误明细
   - 重复数据警告
6. 错误明细下载（Excel / 含错误原因列）

### 4.2 错误反馈格式

| 行号 | sheet | 字段 | 错误类型 | 错误描述 | 修复建议 |
|---|---|---|---|---|---|
| 5 | User | employee_no | 重复 | FK0001 已存在 / 行 3 | 检查是否同人 / 修改 employee_no |
| 8 | User | org_code | 不存在 | FK999 不在 Organization sheet | 添加 Organization 行 或 修正 org_code |
| 12 | User | role | 枚举不合 | "operator" 不在枚举 | 改为 admin/物资/财务/仓储/审批/浏览 |

### 4.3 数据问题台账自动记录

- 重复 / 编码冲突 / 必填缺失 / 枚举不合 / 关联不存在 → 自动记录到 `data_issue_log` 表
- 责任部门：导入用户所属组织 + 物资公司
- 状态：待处理 → 处理中 → 已解决 / 已忽略
