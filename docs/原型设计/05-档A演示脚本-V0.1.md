## 一、演示前准备

### 1.1 环境

- 浏览器：Chrome / Edge / Safari 现代版本
- 屏幕：建议 1920×1080；演示前关闭多余标签页
- 数据：首次打开任一引擎接入页，会自动 seed 主数据（M-01/02/05/09/12 + F-13 + P-01/02/03 业务种子）
- 打开两个标签页：一个用于演示主线，另一个开 `prototype/_engine-test.html` 或 `prototype/alert-rules.html` 验证跨页 BroadcastChannel

### 1.2 重置数据（每场演示前）

打开 `prototype/_engine-test.html` → 点 **"重置 (reset)"** 按钮 → 确认 → LocalStorage 全部清空回 seed 状态。

### 1.3 角色切换

顶部 header 右侧"当前角色"下拉，6 个角色：采购员 / 计划员 / 物资主管 / 财务 / IT / 集团委员会。
切换后页面 reload，菜单可见性 + 数据范围按角色过滤。

## 四、场景 3：NC 失败 → 自动重推 → 成功（治理闭环）

**目标**：演示 F-01 推送失败的自动重推机制 + F-08 异常台账升级。

| 步 | 角色 | 操作 | 预期 |
|---|---|---|---|
| 1-19 | 同场景 1 步骤 1-19 | —— | 直到生成 S-05 草稿 |
| 20 | 仓管员 | 进入 `purchase-receipt.html` → S-05 → "提交审核" | S-05 待审 |
| 21 | 仓管员 | 提前打开 `nc-interface.html` 在另一标签页 | 准备观察 F-01 |
| 22 | 仓管员 | （回 purchase-receipt 标签页）调高 mock 失败率：浏览器控制台输入 `SC.nc.setConfig({ failureRate: 0.8 })` | 失败率改 80% |
| 23 | 仓管员 | S-05 → "审核通过" | F-01 创建 → 推送中 → 1-2 秒后大概率失败 |
| 24 | 任一角色 | nc-interface 看 F-01 推送失败 + 错误码 | 自动 ALR-INT-001 预警；30 秒后 linkage 自动重推（retry_count + 1）|
| 25 | 任一角色 | 等 ~90 秒看连续重推 3 次失败 | F-08 异常台账自动创建（升级） |
| 26 | IT | 切到 `nc-interface-detail.html` F-08 引擎面板 | 看到异常 + 高敏感处置选项（编辑映射 / 重推 / 关闭）|
| 27 | IT | 选 "重推" → 二次确认 → 标记已处置 | F-08 → 已处置 |
| 28 | 仓管员 | 把失败率调回：`SC.nc.setConfig({ failureRate: 0.05 })` | 恢复默认 |

**演示焦点**：
- 第 23 步 失败 → 第 24 步 自动预警 + 自动重推（30 秒延迟）
- 第 25 步 升级 F-08 异常台账
- 第 26-27 步 高敏感处置 + 二次确认（合规闭环）

## 六、口径补充

### 6.1 演示中常被问的问题

- **"数据是真的吗？"** — 全部 mock，存 LocalStorage，单机演示用，刷新页面不丢
- **"NC 真的接通了吗？"** — mock-nc.js 模拟，1-2 秒延迟 + 5% 失败率，可调
- **"角色权限是怎么实现的？"** — roles.js 6 角色 + capabilities，菜单可见性 + 数据范围 mock；正式系统用 A-06 数据权限表
- **"详设里的字段都有吗？"** — 核心字段约 60% 子集（详设字段表完整版以详设 V1.x 为准）
- **"出库 / 暂估 / 付款链路在哪？"** — 一期不演示，移二阶段 A2 / A8 / A4，详见 03-档A二阶段扩展规划-V0.2.md

### 6.2 重置数据

任何时候演示卡了 → 打开 `_engine-test.html` → "重置 (reset)" → 重新开始。

### 6.3 不在范围内的明确事项

- 真实多用户并发（单机演示）
- 真实权限 / 真实 SSO（mock 角色）
- 真实 NC / Nova / 招采平台（全 mock）
- 性能 / 信创合规（不是正式系统）
- 移动端响应式（桌面浏览器）

如需以上，应升级到档 B（MVP）或档 C（正式开发）。
