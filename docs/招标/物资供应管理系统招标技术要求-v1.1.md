## 第一章 项目概况

### 1.1 项目名称

物资供应管理系统建设项目

### 1.2 建设单位

阜矿集团（以下简称"招标方"）

### 1.3 项目背景

招标方为辽宁能源集团所属二级集团，下辖多个煤矿、洗煤厂及辅助单位，物资品类涵盖火工品、支护材料、备品备件、机电材料、油脂燃料、钢材、建筑材料、通防材料、化工材料、劳保用品等十余个大类。

当前物资管理主要存在以下问题：

- 审批与业务流转线上化程度不足，线下流转较多，整体效率有待提升。
- 主数据、单据和统计口径不统一，跨环节追溯和分析能力不足。
- 采购、库存、设备、合同、资金计划及财务衔接不够紧密，业财协同能力不足。
- 基层区队和仓库使用环节覆盖不充分，存在管理断点。
- 现有系统在信创适配、统一集成和后续扩展方面存在不足。

结合当前管理要求，本次建设需重点满足**无纸化审批、多端协同、流程优化、数据赋能、基层延伸**等核心诉求。

### 1.4 项目定位

本系统定位为：

可以将本系统理解为一个以物资流转为主线、强化设备管理并保留资产关联能力、以财务接口为落点、以权限审批和报表预警为支撑的业务平台。

- 物资业务管理平台（实物流转全过程管理）
- 库存实物流转管理平台
- 财务核算前置业务数据源（与用友NC财务系统协同）

本系统**不直接替代**以下系统或职能：

- 用友NC财务总账、应付、成本核算
- 独立合同法务管理平台
- 集团级BI决策平台

## 第二章 建设目标与总体原则

### 2.1 建设目标

**（一）统一标准**

统一物料分类、物料编码、设备分类、资产分类、计量单位、组织、仓库、成本中心等基础数据标准，形成集团统一口径。

**（二）闭环管理**

打通需求提出、采购执行、到货验收、质检、入库、领料、退料、调拨、盘点、废旧处置，以及设备建档、领用、调拨、维保、停启用、报废处置、资产关联留痕等核心业务流程，实现业务全过程留痕。

**（三）业财协同**

明确物资系统与NC财务系统的主数据映射关系、业务触发规则和凭证推送边界，支撑业务与财务协同。

**（四）集团可视**

形成库存、采购、消耗、盘点、呆滞、专项物资，以及设备状态分布、设备租赁执行、资产关联变动留痕等核心管理视图，为管理层提供真实、及时、可追溯的数据支撑。

**（五）流程提效**

实现无纸化审批、多端协同、电子签章衔接、审批提醒催办，大幅提升业务流转效率。

**（六）基层延伸**

系统覆盖到基层区队和仓库使用环节，消除管理断层，实现"最后一公里"数字化。

综上，本项目核心目标是把物资、设备及与设备链路相关的资产关联业务从“分散处理”变成“统一标准、流程闭环、业财协同、集团可视”。

### 2.2 总体原则

- 一物一码，统一标准
- 编码稳定，只增不改
- 业务闭环，单据留痕
- 业务驱动财务，不反向篡改业务
- 集团统一规则，矿厂按权限执行
- 先规范业务，再实现系统
- 以可落地为优先，不追求一步到位
- 遵循集团统一建设原则（统一权限、统一数据、统一接口、统一AI入口、统一运维）
- 独立部署，与集团平台低耦合（系统独立建设与部署，通过标准 API 与集团统一平台对接）

## 第三章 建设范围与边界

### 3.1 一期建设范围

一期建设坚持"先跑通核心闭环、强化设备管理、设备租赁独立纳入并保留资产关联能力"的原则，重点覆盖以下内容：

**（一）基础档案管理**

组织架构、仓库档案、物料分类、物料主数据、编码规则、计量单位、供应商基础档案、成本中心对照、NC存货映射关系。

**（二）物资主数据管理**

物料新增/变更/停用流程、一物一码管理、分类与属性维护、批次/保质期管理规则、条码/二维码基础支持、历史物料与NC编码映射初始化，以及与辽宁能源未来统一编码体系对接的扩展预留能力。

**（三）需求提报与采购计划管理**

使用单位/区队需求提报、需求汇总与去重、月度/季度采购计划编制、计划分级审批、采购任务分解下达、计划执行跟踪、调整追加和计划外紧急采购留痕。

**（四）招投标管理**

招标申请与立项、采购方式分流（公开招标/邀请招标/竞争性谈判/单一来源/询比价/定向直采等）、采购文件生成与审批、采购文件导出/上传至能源集团招采平台、评标结果与中标通知书导入/附件回传归档、与采购计划和合同管理模块的联动、中标/成交价格回写物料主数据。一期实施边界按"协同而非替代"原则，以导出/上传和结果导入的协同方式为主；与能源集团招采平台的深度直连、自动同步和流程穿透纳入二期深化范围。

**（五）采购执行、到货与质量控制**

采购申请、采购订单、到货验收、质检处理、不合格退货与让步接收、火工品等特殊物资重点管控、安全专项物资标识与台账。

**（六）核心库存业务**

采购入库、采购退货、领料出库、退料入库、仓库调拨、盘点盘盈盘亏、废旧物资处置。

**（七）设备管理**

设备分类分级、设备台账建档、设备权属分级与权属变更审批、设备来源关联与资产标识留痕、设备权属归属信息维护、设备状态管理、设备领用/投用/停用/启用、基础调拨与位置变更、跨组织调拨审批与交接确认、维修保养点检留痕、检修申请/执行/完成轻量闭环、设备报废处置、关联资产处置留痕，以及设备生命周期基础过程追踪。

**（八）设备租赁管理**

设备租赁申请与审批、租赁合同与设备档案关联、起租登记、在租状态管理、使用单位/地点和责任人维护、续租停租退租与交接确认、租金标准与计费周期维护、月度租赁费用汇总，以及与资金计划管理模块的联动。

**（九）合同管理**

合同网上审批会签、合同基础信息登记、合同与采购计划/采购订单/入库单关联、付款条款与付款节点定义、履行进度查询、到期提醒、合同风险点管理、合同变更与终止留痕、合同全生命周期台账。

**（十）资金计划管理**

按合同付款条款自动生成付款计划、月度预支付汇总、付款申请审批、付款条件校验、付款进度台账、付款到期提醒、NC 实际付款结果回写与对账留痕。

**（十一）供应商管理**

供应商分类管理、资质文件管理、资质过期提醒、准入控制、考评机制、黑名单管理和供应商全品类台账。

**（十二）财务接口与对账**

主数据同步与映射、业务单据财务触发规则、委托加工等扩展业务场景财务触发、NC凭证推送、异常重推、月末对账与差异处理。

**（十三）报表预警与查询**

库存余额表、收发存报表、采购/领料/调拨汇总、盘点差异、废旧处置台账、接口状态、暂估未冲销、呆滞库存、低库存/超储/临期预警。

**（十四）系统管理与基础支撑**

用户/角色/岗位管理、数据权限控制、审批流程配置、操作日志、接口日志、通知提醒、无纸化审批、多端协同、电子签章衔接能力。

**（十五）数据迁移、并行运行与平稳切换**

历史数据清洗与迁移、新旧系统并行运行、关键数据比对、上线切换、回退预案和护航保障。

### 3.2 本次招标范围边界说明

本次招标范围仅限一期建设内容。投标方应按照本文件所列一期功能范围进行方案设计、报价、实施和交付，凡未明确列入一期建设范围的功能，不作为本次投标报价、建设实施和验收依据。

投标方在一期建设中应充分考虑后续扩展需要，整体架构、数据模型、流程配置、接口机制和权限体系应具备良好的可扩展性，以支持后续阶段功能深化和外部平台协同对接。

### 4.3 一期业务衔接说明

一期业务主链路为"需求提报→采购计划→招投标/询比价→合同/采购执行→到货入库→设备建档与投用/设备租赁执行→资金计划管理→生命周期管理→财务联动"。招投标管理已纳入一期建设范围，作为采购计划审批通过后、合同签订前的核心环节由系统统一管理：

- 采购计划审批通过后，进入招投标模块按采购方式（公开招标、邀请招标、竞争性谈判、询价、竞价、单一来源等）自动分流处理
- 本系统生成采购文件，支持导出或上传至能源集团招采平台；公告发布、投标、开标、评标、结果公示等过程在能源集团招采平台完成
- 中标/成交结果导入本系统，或将评标报告、结果公示材料等作为附件回传归档；结果回传后由招投标模块推送至合同管理模块，合同编制时自动带入招投标结果和采购计划数据
- 中标/成交价格自动回写至物料主数据，便于后续价格分析和历史比对
- "计划→招采平台→结果回传→合同→入库/设备租赁执行→资金计划管理→财务"全链路数据贯通，每个环节可追溯

一期至少应支持采购文件导出/上传、结果导入/附件归档的协同方式；与能源集团电子商务平台的深度对接、自动上传和自动回传不作为本次招标范围及验收依据。投标方应在系统架构中预留平台对接扩展能力。

### 4.4 设备、设备租赁、合同、资金计划、供应商模块特别说明

根据业务沟通结果，设备管理、设备租赁管理、合同管理、资金计划管理和供应商管理已作为一期独立模块纳入建设范围。资产相关能力不再单列独立模块，具体按以下边界执行：

- 设备管理一期须覆盖台账建立、权属分级与变更审批、来源关联与资产标识留痕、权属归属信息维护、状态变更、业务流转、基础调拨、跨组织调拨审批与交接确认、基础维修保养留痕、检修申请/执行/完成轻量闭环、处置报废、关联资产处置留痕、全过程留痕和统计分析等基础增强管理要求
- 设备租赁管理一期须覆盖租赁申请、合同关联、起租登记、在租跟踪、续租停租退租、月度租赁费用汇总和付款联动等基础闭环能力
- 资产相关能力本期仅保留来源关联、权属归属和处置留痕等关联信息，不再单列独立模块和独立验收对象
- 合同模块须负责审批会签、履行跟踪、付款条款定义、到期提醒、合同风险点管理、变更留痕和合同台账
- 资金计划管理模块须负责付款计划生成、月度预支付汇总、付款申请审批、付款条件校验、付款进度跟踪和实付结果回写
- 供应商分类管理、资质提醒、准入控制、基础考评和黑名单须一期实现
- 合同模板库、条款库、电子签约、复杂评分模型及其他未明确列入一期范围的深化能力，不作为本次招标范围和验收依据

## 第五章 主数据与编码管理要求

### 5.1 物料编码体系

投标方应支持招标方已制定的物料编码规范，主要特征：

- 编码结构：大类（2位字母）+ 分类（2位数字）+ 流水号（6位数字），总长10位（含分隔符12位）
- 一物一码，全集团唯一
- 编码只增不改，停用不删除不复用
- 编码不含组织信息，库存归属通过组织/仓库字段区分
- 两级分类，更细属性通过字段实现
- 跨大类边界模糊物料（如支护用钢、含化学添加剂的液压介质、防爆成套电气等）须按附件六 §5.16《归类裁决规则》的五级优先级（专业管控 → 终端用途 → 主要功能 → 整机/部件 → 物料形态）统一处理；系统须提供大类自动推荐、跨类申报提示、归类评审流程和案例库维护能力
- 一期须预留与辽宁能源未来统一编码体系对接的扩展能力，包括集团编码字段、映射关系和接口扩展空间；在上位标准未明确前，不要求一期直接启用集团统一编码，也不得以此为由要求重编现有物料编码

### 5.2 物料主数据管理

- 支持物料新增、变更、停用全流程审批
- 支持名称+规格+型号查重校验
- 支持物料属性维护（规格、品牌、材质、批次、保质期、危险品等级、损耗定额等）
- 支持资质文件管理（安标、防爆合格证等），支持上传、存储、查询，资质过期自动提醒
- 支持采购方式、技术要求等扩展属性维护
- 支持采购价格管理（中标/成交结果、采购价格自动回写，按价格来源时间、采购方式分类统计）
- 支持条码/二维码自动生成与绑定
- 支持物料分类体系维护与调整（需审批）
- 支持多维分类标签能力（material_tag）：在不改动主编码的前提下为同一物料挂接财务科目、煤矿专业系统（采掘运提通机洗辅）、行业标准编码（如 MT/T 1090）、自定义管理属性等多维视角，维度可运行态扩展，支持按任一维度聚合统计与一致性校验；具体预留要求见附件六 §11.5

### 5.3 与NC存货编码映射

- 支持N:1映射（多个物料规格对应一个NC存货编码）
- 支持默认映射 + 组织级映射（同一物料在不同矿厂可能映射到不同NC存货）
- 支持映射的建立、修改、停用、追溯
- 支持映射完整性日检（nc_inv_code为空告警）
- 支持上线前历史映射批量导入

### 5.4 其他主数据

- 组织架构与仓库档案
- 计量单位字典（须与NC统一）
- 供应商基础档案
- 成本中心/使用单位对照
- 物料分类与NC存货分类对照

### 5.5 主数据治理要求

投标方应配合招标方在上线前完成一次主数据治理，包括：

- 历史物料梳理与清洗
- 重复编码清理
- 单位统一
- 分类补齐
- NC映射建立与校验

### 5.6 权威数据源与扩展预留

- 组织架构、人员信息、SSO 身份以集团统一平台（Nova Platform）为权威来源，本系统通过平台 API 引用，不重复维护
- 记账科目以用友NC财务系统为权威来源
- 物料编码由本系统作为权威源，并预留对齐集团未来统一编码体系的字段、映射关系和接口扩展能力
- 供应商、合同、资产等编码在集团权威来源明确前由本系统自行维护，须预留后续对齐集团统一权威来源的字段、映射关系和接口扩展能力；上位权威来源确立后，本系统以映射或引用方式对齐，不反向主导集团编码

## 第六章 业务流程与单据功能要求

### 6.1 需求提报与采购计划

投标方应至少支持以下业务流程和单据：

**单据：** 物资需求提报单、月度/季度/年度/专项/调整采购计划、新品计划申报单、采购任务单

**流程要求：**

- 支持使用单位/区队按周期或临时提报物资需求（品名、规格、数量、期望到货时间、用途）
- 支持各矿物资管理部门归集、合并、去重汇总需求，核对库存余量
- 支持月度、季度、年度、专项、调整计划等多类型计划的分类编制和管理
- 支持新品计划线上申报，新品信息与物资信息库自动关联
- 采购计划按金额/类别分级审批
- 审批通过后支持计划分解为采购任务，下达至采购人员
- 支持计划执行跟踪（完成率、到货率、预算消耗）
- 支持计划调整和追加，须说明原因并留痕
- 支持计划外紧急采购流程，须补充审批并纳入计划台账
- 库存充裕物料在汇总阶段应自动提示，避免重复采购
- 计划的取消、变更操作全程记录留痕，保留变更原因、审批意见

### 6.1-1 招投标管理功能要求

招投标管理已纳入一期建设范围，投标方应至少支持以下功能：

**单据：** 招标申请单、非招标申请单、公开询比申请单、不足三家处置单、采购文件、招采结果回传/归档单、询价/竞价单

评标报告、结果公示材料等由能源集团招采平台形成后，可作为附件回传归档，不要求本系统内独立形成评标过程单据。

**采购方式分三条路径（按项目属性自动分流）：**

1. **招标路径**（满足招标条件且采用招标方式）：招标申请经物资公司内部逐级汇签（业务员 → 业务中心主任 → 分管业务经理 → 企划中心主任 → 公司负责人），加盖招标专用章及公章后报集团生产技术部审批。**涉安标项目须由生产技术部、安监部分别审批**。最终 PDF 提报招投标管理中心。
2. **非招标路径**（满足招标条件但走非招标）：在上述审批基础上增加招投标管理中心审批；**金额超过 100 万元的须由集团分管领导签字**；**涉安标项目生产技术部与安监部一同审批**。
3. **公开询比路径**（不满足招标条件）：物资公司自行编制公开采购文件（含资格、技术、商务、评标办法），经公司领导审批后提报招投标管理中心，**须提前两周**提交以便合理安排开标日期。

**流程要求：**

- 支持招标申请与立项，关联上游采购计划，按采购方式自动分流
- 支持公开招标、邀请招标、竞争性谈判、询价、竞价、单一来源等主要采购方式
- 支持按采购方式（招标 / 非招标 / 公开询比）配置差异化审批链与审批节点
- 支持招标文件编制（按标包自动生成采购文件）
- 支持采购文件导出，或按条件上传至能源集团招采平台
- 支持能源集团招采平台结果导入，或将评标报告、结果公示材料等作为附件回传归档
- 支持**公示期配置**（默认公开询比公示期 5 天）及开标日期自动校验
- 支持**评标专家抽取**辅助登记（默认 5 名专家，可按项目配置）及评标结果登记
- **不足三家处置全链路**：
  - 一次公示报名单位不足三家 → 自动生成"不足三家处置单"，登记流标原因，提报招投标管理中心后发起二次公示
  - 招标路径二次公示仍不足三家 → 可提报**集团供应链管理委员会**登记决议结果，支持"调整采购方式 / 付款方式后三次招标"或"转为非招标采购"两类处置
  - 公开询比路径二次公示后按报名家数自动分流：**= 2 家转竞价采购**、**= 1 家转直接采购**，均须按询比文件规定执行
  - 全流程决议与转换动作须完整留痕，支持审计回放
- 支持招投标结果与采购计划、合同管理模块联动（中标/成交结果回传后自动推送至合同模块）
- 支持询价、竞价等非招标采购方式的结果回传与留痕管理，与供应商管理模块联动（仅合格供应商可参与）
- 中标/成交价格自动回写物料主数据
- 招投标全流程数据留存，满足审计和监管追溯要求

一期至少应支持采购文件导出/上传、结果导入/附件归档的协同方式；与能源集团电子商务平台的深度对接、自动上传和自动回传不作为本次招标范围和验收依据。投标方应在系统架构中预留平台对接扩展能力。

### 6.2 采购到货与入库

**流程要求：**

- 支持合格入库、部分合格、让步接收、全部退货
- 支持部分到货、部分入库

- 支持入库照片拍照保存，操作留痕可追溯

  - 设备到矿后，使用单位出具**《设备直达到货验收单》**（含设备及随机部件清点、外观检查、资料核验）→ 触发开票通知、挂账、退还履约保证金

- 支持预付款、进度款、到货款、验收款、质保金、尾款等分期支付跟踪
- 支持付款进度、已付未付余额和剩余可付金额台账
- 支持付款到期提醒、超期预警和超节点金额控制
- 支持 NC 或等效财务系统实际付款结果回写，并同步更新合同台账

### 6.9 供应商管理

**单据：** 供应商档案、供应商资质文件、供应商考评记录、黑名单管理记录

**流程要求：**

- 支持潜在供应商、合格供应商、负面供应商分类管理
- 支持供应商基础信息管理（名称、资质、联系方式、经营范围、合作记录）
- 支持供应商资质文件上传、存储、查询，资质过期自动提醒
- 合格供应商与采购环节联动，仅允许合格供应商参与采购活动
- 支持供应商考评（履约进度、产品质量、服务水平等维度），考评结果与采购联动
- 支持负面供应商（黑名单）管理，限制参与集团所有采购活动，解除限制需走审批流程
- 供应商全品类台账

### 6.10 单据设计统一要求

- 每张单据均有唯一单号
- 均有明确状态流转（草稿→待审→已审→已作废等）
- 影响库存的单据必须明确生效时点
- 触发财务的单据必须明确推送时点
- 已推财务的单据不得随意修改
- 作废、冲红、重推均需留痕

## 第七章 财务与NC接口要求

### 7.1 接口建设原则

- 业务主导：物资系统是业务发生方和数据源头
- 主数据先行：接口依赖主数据统一，须先完成映射
- NC侧零改造：物资系统主动适配NC
- 同口径：计量单位、计价方法、成本中心须两系统一致
- 可追溯：所有推送、重推、失败、差异调整可查询可审计

### 7.2 接口范围

一期须实现 **29项接口**（详见附件二《接口清单及报文示例》），包括：

- 主数据同步接口 5项
- 业务单据推送接口 19项
- 对账与监控接口 5项

委托加工业务场景的相关财务触发和接口映射应纳入上述接口清单统一管理，不得脱离接口台账单独处理。

### 7.3 关键接口要求

**（一）凭证推送**

- 所有业务单据审核通过后方可推送NC
- 推送报文采用JSON格式，包含来源单号（`sourceBillNo`）等幂等关键字段用于幂等校验，具体幂等键组合以配套《物资管理与财务接口规范》为准
- 火工品出入库须**实时**推送，其他业务**准实时**推送
- 支持委托加工相关业务场景的财务触发和凭证推送
- 暂估和暂估冲销为**月末/月初批量**处理

**（二）异常处理**

- NC接口超时自动重推3次，间隔30秒
- NC返回业务错误记录日志，人工处理后手动重推
- 重复推送由NC幂等校验处理
- 接口状态应清晰可见（待推送/推送中/成功/失败/已重推/已作废）

**（三）对账机制**

- 日对账：当日出入库笔数及金额 vs NC当日凭证（系统自动）
- 周对账：库存余额 vs NC存货科目余额
- 月对账：全量数量/金额对账，出具对账报告
- 年对账：配合财务审计

**（四）月末暂估**

- 月末扫描已入库+发票未到单据，按订单价暂估
- 次月初红字冲销上月全部暂估
- 发票到后正式入账，与暂估差异按财务政策处理

### 7.4 接口责任边界


| 责任方     | 职责                 |
| ------- | ------------------ |
| 物资系统开发商 | 业务数据准备、校验、推送、重推、日志 |
| NC侧     | 接收、处理、回执、幂等校验      |
| 招标方网信办  | 统筹协调、联调组织、异常升级     |
| 招标方财务部  | 口径确认、对账、月结确认       |


### 7.5 投标方响应要求

投标方须在技术响应中说明：

- 接口实现方式（REST API / 消息队列 / 混合）
- 与NC联调的经验和方案
- 接口异常处理和补偿机制
- 对账功能实现方案
- 暂估/冲销/正式入账自动化程度

## 第八章 角色权限与审批要求

### 8.1 主要角色

系统至少应支持以下角色：

1. 集团领导/管理层（只读看板）
2. 网信办/信息化管理
3. 集团物资管理
4. 各矿物资管理
5. 仓库管理员
6. 使用单位/区队申请人
7. 采购人员
8. 质检人员
9. 财务人员
10. 安全部门
11. 系统管理员

### 8.2 权限设计要求

- 支持基于组织+角色+岗位的权限体系
- 支持数据权限（按组织/仓库/物料范围控制数据可见性）
- 人员调动后权限应能及时调整
- 离岗/离职人员应能及时禁用
- 高敏感操作（月结/反结/火工品/盘亏等）须设置加强审批

### 8.3 集团平台对接要求

**（一）统一登录（SSO）**

- **须对接集团统一平台 SSO 单点登录**，不另行建设独立用户认证体系
- 用户登录集团平台后，可直接访问物资系统，无需二次登录

**（二）组织与人员数据**

- 组织架构和人员基础数据以集团平台为权威数据源
- 物资系统通过平台 API 获取并本地缓存，不另起一套组织人员管理
- 物料编码等物资领域专属主数据由本系统作为权威源

**（三）数据权限**

- 数据隔离以二级集团为边界，与集团平台隔离机制保持一致
- 物资系统内部的业务角色和业务权限由本系统自行维护

**（四）审批兼容**

- 审批流程中组织角色解析应与集团平台体系兼容
- 可通过平台 API 获取审批人信息，或按等效规则（角色+组织过滤+层级递进）自行实现
- 审批过程和结果须完整留痕

### 8.4 审批流程要求

- 支持可配置的多级审批流程
- 支持按金额、物料类别、组织等条件差异化审批路径
- 支持手机端审批
- 支持审批超时催办
- 支持审批全过程留痕

## 第十章 技术与部署要求

### 10.1 部署方式

- 采用**私有化部署**，部署在招标方指定的数据中心或机房
- 不接受纯SaaS公有云部署

### 10.2 信创与国产化要求

- 数据库须采用 PostgreSQL 兼容的国产信创数据库（如瀚高 HGDB、人大金仓 KingbaseES 等），独立部署，不与集团平台共用数据库实例
- 操作系统须支持国产Linux（银河麒麟/统信UOS等至少一种）
- 中间件须支持国产中间件或开源中间件
- 浏览器须兼容主流国产浏览器

### 10.3 多端支持

- PC端：B/S架构，浏览器直接访问
- 移动端：APP或小程序（至少覆盖领料申请、审批、库存查询、扫码）
- 数据实时同步
- 各终端操作记录统一归集、同步更新

### 10.4 操作便捷性要求

- 优化系统界面与操作流程，简化复杂操作步骤
- 提供操作指引与提示功能，降低新用户学习成本
- 适配不同岗位业务人员的操作习惯

### 10.5 性能要求

- 支持并发用户数不低于 **200人**（待根据实际用户规模确认）
- 单据操作响应时间 ≤ **3秒**
- 报表查询响应时间 ≤ **5秒**
- 接口推送延迟 ≤ **30秒**（准实时场景）
- 系统可用性 ≥ **99.5%**

### 10.6 安全要求

- 支持HTTPS传输加密
- 支持操作日志完整记录
- 支持数据备份与恢复
- 支持防篡改、防泄露等基础数据安全控制
- 支持敏感数据脱敏（如价格、合同金额按角色控制可见性）
- 支持按角色/岗位/组织控制数据导出权限，敏感数据导出须单独留痕
- 重要导出文件应支持水印、导出日志或等效追踪机制
- 接口调用须具备鉴权机制

### 10.7 接口标准

- 系统间集成采用API方式
- 接口报文统一使用JSON格式
- 接口应具备版本管理能力
- 接口应支持幂等调用

### 10.8 技术栈与平台集成

- 开发语言不做强制限定；若投标方采用 .NET 技术栈，可直接引用集团平台模块，降低集成成本；若采用其他技术栈，须通过标准 API 完成全部平台对接
- 本系统须与集团统一平台完成对接（详见第八章 8.3 节），包括：SSO 统一登录、组织与人员数据同步、数据权限对齐、审批体系兼容
- 投标方须在技术方案中明确说明与集团平台的集成方案和技术路径
