## 第一章 项目概况

### 1.1 项目名称

物资供应管理系统建设项目

### 1.2 建设单位

阜矿集团（以下简称"招标方"）

### 1.3 项目背景

招标方为辽宁能源集团所属二级集团，下辖多个煤矿、洗煤厂及辅助单位，物资品类涵盖火工品、支护材料、备品备件、机电材料、油脂燃料、钢材、建筑材料、通防材料、化工材料、劳保用品等十余个大类。

当前物资管理存在以下突出问题：

**（一）流程效率低下**

- 缺少无纸化审批能力，审批流转依赖线下人工传递
- 电子签章能力缺失，审批周期长、滞后明显
- 人工跑流程成本高，业务办理效率难以满足生产节奏

**（二）数据管理薄弱**

- 数据更新不够实时，难以及时反映业务变化
- 缺少跨环节、跨单据的关联分析能力
- 人员岗位调整后系统权限不能同步更新
- 系统未有效延伸至基层单位和区队，造成管理断层

**（三）功能覆盖不足**

- 缺少合同账期提醒等管理功能
- 缺少物资全生命周期管理能力
- 关键业务功能和管理辅助能力尚未覆盖

**（四）合规性不达标**

- 现有系统在技术路线和部署适配方面不符合国产化信创要求
- 难以满足后续规范建设、统一运维和持续扩展的要求

前期通过对系统开发端问题清单梳理及 11 名关键岗位使用人员调研，业务人员对**无纸化审批、手机/PC多端同步、流程优化、数据赋能、基层延伸**的需求最为集中和迫切。

### 1.4 项目定位

本系统定位为：

可以将本系统理解为一个以物资流转为主线、兼顾设备与资产全生命周期管理、以财务接口为落点、以权限审批和报表预警为支撑的业务平台。

- 物资业务管理平台（实物流转全过程管理）
- 库存实物流转管理平台
- 财务核算前置业务数据源（与用友NC财务系统协同）

本系统**不直接替代**以下系统或职能：

- 用友NC财务总账、应付、成本核算
- 独立合同法务管理平台
- 集团级BI决策平台

## 第二章 建设目标与总体原则

### 2.1 建设目标

**（一）统一标准**

统一物料分类、物料编码、设备分类、资产分类、计量单位、组织、仓库、成本中心等基础数据标准，形成集团统一口径。

**（二）闭环管理**

打通需求提出、采购执行、到货验收、质检、入库、领料、退料、调拨、盘点、废旧处置，以及设备、资产建档、领用、调拨、维保、停启用、报废处置等核心业务流程，实现业务全过程留痕。

**（三）业财协同**

明确物资系统与NC财务系统的主数据映射关系、业务触发规则和凭证推送边界，支撑业务与财务协同。

**（四）集团可视**

形成库存、采购、消耗、盘点、呆滞、专项物资，以及设备、资产状态分布、变动情况和生命周期过程等核心管理视图，为管理层提供真实、及时、可追溯的数据支撑。

**（五）流程提效**

实现无纸化审批、多端协同、电子签章衔接、审批提醒催办，大幅提升业务流转效率。

**（六）基层延伸**

系统覆盖到基层区队和仓库使用环节，消除管理断层，实现"最后一公里"数字化。

综上，本项目核心目标是把物资、设备、资产相关业务从“分散处理”变成“统一标准、流程闭环、业财协同、集团可视”。

### 2.2 总体原则

- 一物一码，统一标准
- 编码稳定，只增不改
- 业务闭环，单据留痕
- 业务驱动财务，不反向篡改业务
- 集团统一规则，矿厂按权限执行
- 先规范业务，再实现系统
- 以可落地为优先，不追求一步到位
- 遵循集团统一建设原则（统一权限、统一数据、统一接口、统一AI入口、统一运维）
- 独立部署，与集团平台低耦合（系统独立建设与部署，通过标准 API 与集团统一平台对接）

## 第三章 建设范围与边界

### 3.1 一期建设范围

一期建设坚持"先跑通核心闭环、同步纳入设备与资产管理"的原则，重点覆盖以下内容：

**（一）基础档案管理**

组织架构、仓库档案、物料分类、物料主数据、编码规则、计量单位、供应商基础档案、成本中心对照、NC存货映射关系。

**（二）物资主数据管理**

物料新增/变更/停用流程、一物一码管理、分类与属性维护、批次/保质期管理规则、条码/二维码基础支持、历史物料与NC编码映射初始化，以及与辽宁能源未来统一编码体系对接的扩展预留能力。

**（三）需求提报与采购计划管理**

使用单位/区队需求提报、需求汇总与去重、月度/季度采购计划编制、计划分级审批、采购任务分解下达、计划执行跟踪、调整追加和计划外紧急采购留痕。

**（四）采购执行、到货与质量控制**

采购申请、采购订单、到货验收、质检处理、不合格退货与让步接收、火工品等特殊物资重点管控、安全专项物资标识与台账。

**（五）核心库存业务**

采购入库、采购退货、领料出库、退料入库、仓库调拨、盘点盘盈盘亏、废旧物资处置。

**（六）设备管理**

设备分类分级、设备台账建档、设备状态管理、设备领用/投用/停用/启用、设备调拨与位置变更、维修保养点检留痕、设备报废处置，以及设备全生命周期过程追踪。

**（七）资产管理**

资产分类与资产台账、资产登记与来源关联、资产领用/配置/调拨、资产盘点、资产状态变更、资产处置与报废，以及资产全生命周期过程追踪。

**（八）合同管理**

合同网上审批会签、合同基础信息登记、合同与采购计划/采购订单/入库单/付款记录关联、履行进度查询、到期及付款节点提醒、合同变更与终止留痕、合同全生命周期台账。

**（九）供应商管理**

供应商分类管理、资质文件管理、资质过期提醒、准入控制、考评机制、黑名单管理和供应商全品类台账。

**（十）财务接口与对账**

主数据同步与映射、业务单据财务触发规则、NC凭证推送、异常重推、月末对账与差异处理。

**（十一）报表预警与查询**

库存余额表、收发存报表、采购/领料/调拨汇总、盘点差异、废旧处置台账、接口状态、暂估未冲销、呆滞库存、低库存/超储/临期预警。

**（十二）系统管理与基础支撑**

用户/角色/岗位管理、数据权限控制、审批流程配置、操作日志、接口日志、通知提醒、无纸化审批、多端协同、电子签章衔接能力。

**（十三）数据迁移、并行运行与平稳切换**

历史数据清洗与迁移、新旧系统并行运行、关键数据比对、上线切换、回退预案和护航保障。

### 3.2 一期暂不纳入范围

- 设备租赁全流程管理（作为一期设备管理的深化增强内容）
- 检修计划深度联动
- 能源集团电子商务平台深度对接（待平台标准明确后启动）
- 合同模板库、条款库、电子签约、索赔等深化能力
- 复杂经营分析驾驶舱
- 全量PDA深度作业
- 与辽宁能源集团级平台正式对接（预留接口）
- 智能补货、智能预测等高级能力

### 3.3 本系统与现有系统的关系

| 系统 | 关系 | 说明 |
|------|------|------|
| 集团统一平台（Nova Platform） | 平台对接 | 组织架构、人员基础数据、SSO 单点登录、数据权限体系从集团平台获取；物资系统独立部署数据库，通过 API 与平台集成 |
| 用友NC财务系统 | 接口协同 | 物资系统为业务源头，NC为价值核算，通过标准接口推送凭证 |
| 人资系统 | 数据引用 | 人员、组织信息以集团平台为权威源（平台从人资系统同步），物资系统通过平台 API 获取 |
| 现有物资系统 | 替代 | 当前日常运行中，新系统上线后逐步替代，历史数据按方案迁移 |

### 4.3 一期业务衔接说明

一期业务主链路为"需求提报→采购计划→招投标/询比价→合同/采购执行→到货入库→库存流转→设备/资产建档与投用→生命周期管理→财务联动"。招投标管理已纳入一期建设范围，作为采购计划审批通过后、合同签订前的核心环节由系统统一管理：

- 采购计划审批通过后，进入招投标模块按采购方式（公开招标、邀请招标、竞争性谈判、询价、竞价、单一来源等）自动分流处理
- 招投标/询比价过程在系统内完成，评标、结果公示、结果确认等环节全程线上留痕
- 中标/成交结果由招投标模块自动推送至合同管理模块，合同编制时自动带入招投标结果和采购计划数据
- 中标/成交价格自动回写至物料主数据，便于后续价格分析和历史比对
- "计划→招投标→合同→入库→财务"全链路数据贯通，每个环节可追溯

与能源集团电子商务平台的深度对接纳入二期，待平台侧对接标准明确后启动；一期以本系统内建招投标能力为主。

### 4.4 合同与供应商模块特别说明

根据业务沟通结果，设备管理、资产管理、合同管理和供应商管理已作为一期独立模块纳入建设范围，不再按“轻量台账”处理。原则上：

- 设备管理、资产管理须覆盖台账建立、状态变更、业务流转、全过程留痕和统计分析等全生命周期管理要求，不限于基础信息登记
- 合同审批会签、履行跟踪、账期提醒、变更留痕和合同台账须一期实现
- 供应商分类管理、资质提醒、准入控制、基础考评和黑名单须一期实现
- 合同模板库、条款库、电子签约和复杂评分模型可作为后续增强内容

## 第五章 主数据与编码管理要求

### 5.1 物料编码体系

投标方应支持招标方已制定的物料编码规范，主要特征：

- 编码结构：大类（2位字母）+ 分类（2位数字）+ 流水号（6位数字），总长10位（含分隔符12位）
- 一物一码，全集团唯一
- 编码只增不改，停用不删除不复用
- 编码不含组织信息，库存归属通过组织/仓库字段区分
- 两级分类，更细属性通过字段实现
- 一期须预留与辽宁能源未来统一编码体系对接的扩展能力，包括集团编码字段、映射关系和接口扩展空间；在上位标准未明确前，不要求一期直接启用集团统一编码，也不得以此为由要求重编现有物料编码

### 5.2 物料主数据管理

- 支持物料新增、变更、停用全流程审批
- 支持名称+规格+型号查重校验
- 支持物料属性维护（规格、品牌、材质、批次、保质期、危险品等级、损耗定额等）
- 支持资质文件管理（安标、防爆合格证等），支持上传、存储、查询，资质过期自动提醒
- 支持采购方式、技术要求等扩展属性维护
- 支持采购价格管理（中标/成交结果、采购价格自动回写，按价格来源时间、采购方式分类统计）
- 支持条码/二维码自动生成与绑定
- 支持物料分类体系维护与调整（需审批）

### 5.3 与NC存货编码映射

- 支持N:1映射（多个物料规格对应一个NC存货编码）
- 支持默认映射 + 组织级映射（同一物料在不同矿厂可能映射到不同NC存货）
- 支持映射的建立、修改、停用、追溯
- 支持映射完整性日检（nc_inv_code为空告警）
- 支持上线前历史映射批量导入

### 5.4 其他主数据

- 组织架构与仓库档案
- 计量单位字典（须与NC统一）
- 供应商基础档案
- 成本中心/使用单位对照
- 物料分类与NC存货分类对照

### 5.5 主数据治理要求

投标方应配合招标方在上线前完成一次主数据治理，包括：

- 历史物料梳理与清洗
- 重复编码清理
- 单位统一
- 分类补齐
- NC映射建立与校验

## 第六章 业务流程与单据功能要求

### 6.1 需求提报与采购计划

投标方应至少支持以下业务流程和单据：

**单据：** 物资需求提报单、月度/季度/年度/专项/调整采购计划、新品计划申报单、采购任务单

**流程要求：**

- 支持使用单位/区队按周期或临时提报物资需求（品名、规格、数量、期望到货时间、用途）
- 支持各矿物资管理部门归集、合并、去重汇总需求，核对库存余量
- 支持月度、季度、年度、专项、调整计划等多类型计划的分类编制和管理
- 支持新品计划线上申报，新品信息与物资信息库自动关联
- 采购计划按金额/类别分级审批
- 审批通过后支持计划分解为采购任务，下达至采购人员
- 支持计划执行跟踪（完成率、到货率、预算消耗）
- 支持计划调整和追加，须说明原因并留痕
- 支持计划外紧急采购流程，须补充审批并纳入计划台账
- 库存充裕物料在汇总阶段应自动提示，避免重复采购
- 计划的取消、变更操作全程记录留痕，保留变更原因、审批意见

### 6.1-1 招投标管理功能要求

招投标管理已纳入一期建设范围，投标方应至少支持以下功能：

**单据：** 招标申请单、采购文件、评标记录、结果公示单、询价/竞价单

**流程要求：**

- 支持招标申请与立项，关联上游采购计划，按采购方式自动分流
- 支持公开招标、邀请招标、竞争性谈判、询价、竞价、单一来源等主要采购方式
- 支持招标文件编制（按标包自动生成采购文件）
- 支持评标管理（评标过程自动生成评标文件和排名报表）
- 支持结果公示与确认
- 支持招投标结果与采购计划、合同管理模块联动（中标/成交结果自动推送至合同模块）
- 支持询价、竞价等非招标采购方式的线上操作，与供应商管理模块联动（仅合格供应商可参与）
- 中标/成交价格自动回写物料主数据
- 招投标全流程数据留存，满足审计和监管追溯要求

与能源集团电子商务平台的深度对接纳入二期，一期以本系统内建能力为主；投标方应在系统架构中预留平台对接扩展能力。

### 6.2 采购到货与入库

投标方应至少支持以下业务流程和单据：

**单据：** 采购申请单、采购订单、到货验收单、质检单、采购入库单、采购退货单

**流程要求：**

- 支持到货登记、验收确认、质检分支处理
- 支持合格入库、部分合格、让步接收、全部退货
- 支持部分到货、部分入库
- 支持三单匹配（订单+入库+发票）辅助记录
- 支持暂估标识，月末暂估批处理
- 入库审核通过后方可正式影响库存
- 支持入库照片拍照保存，操作留痕可追溯
- 大型设备直达验收须在系统中单独记录验收信息
- 支持条码/二维码扫码入库

**财务触发：**

- 正式入库 → 推送NC入库凭证（BIZ-001）
- 发票未到月末 → 推送暂估凭证（BIZ-002）
- 退货 → 推送红字凭证（BIZ-004）
- 让步接收 → 推送降价入库凭证（BIZ-016）

### 6.3 领料出库与退料

**单据：** 领料申请单、领料出库单、退料入库单

**流程要求：**

- 支持领料申请→审批→仓库发料→系统出库
- 领料单须携带成本中心/用途信息
- 支持同一领料单多物料、多成本中心
- 支持退料关联原领料单
- 支持应急直发 + 事后补单（需受控审批）
- 批次物料领用遵循FIFO，系统自动推荐最早批次
- 火工品领用须实名制、按需领用

**财务触发：**

- 领料出库 → 推送费用/成本凭证（BIZ-005）
- 退料入库 → 推送红字冲销凭证（BIZ-006）
- 安全专项领用 → 推送专项储备凭证（BIZ-017）

### 6.4 调拨

**单据：** 调拨申请单、调拨单

**流程要求：**

- 支持同组织仓间调拨（不产生财务凭证）
- 支持跨组织调拨（须双方签收确认，产生财务凭证）
- 支持调拨差异登记（运输损耗等）
- 调拨差异超阈值须升级审批
- 支持物资移拨（物理位置转移但权属不变），仅记录保管状态变更
- 支持库存权属管理，区分权属归属和实际保管位置

**财务触发：**

- 跨组织调拨 → 推送内部往来凭证（BIZ-007）

### 6.5 盘点

**单据：** 盘点任务单、盘点单、盘盈处理单、盘亏处理单

**流程要求：**

- 支持定期和临时盘点任务
- 支持手机/PDA扫码盘点
- 系统自动比对账面与实盘差异
- 盘盈盘亏须审批，超阈值升级审批
- 火工品盘盈须上报安全部门，盘亏须附事故调查报告

**财务触发：**

- 盘盈 → 推送盘盈凭证（BIZ-008）
- 盘亏 → 推送盘亏凭证（BIZ-009）

### 6.6 废旧物资处置

**单据：** 废旧认定单、处置申请单、废旧出库单、处置结果登记单

**流程要求：**

- 支持报废、回收、变卖、销毁四类处置方式
- 危险品销毁须安全部门审批并全程留痕
- 处置结果（收入/损失）须登记
- 全过程可追溯

**财务触发：**

- 变卖出库 → 推送处置损失凭证（BIZ-010）
- 变卖收入 → 推送收入凭证（BIZ-011）
- 危险品销毁 → 推送销毁损失凭证（BIZ-012）

### 6.7 月末对账与结账

**流程要求：**

- 支持月末数据封存（停止当月补录）
- 支持暂估批处理、暂估冲销
- 支持物资系统与NC月度全量对账
- 支持对账差异清单生成和处理
- 支持月结确认和期间锁定
- 支持已结账期间受控反结（严格审批）

### 6.8 合同管理

**单据：** 合同审批会签单、合同登记单、合同变更单、合同终止单、合同台账

**流程要求：**

- 支持合同网上审批会签，按预设流程自动流转至相关部门/岗位，审批意见在线留痕
- 合同编制时自动带入上游采购计划、采购结果、物资信息库等相关数据
- 支持合同基础信息登记（合同号、供应商、物料范围、金额、有效期、付款条件）
- 支持合同与采购订单/入库单/付款记录关联
- 支持合同履行进度实时查询（已到货量/金额 vs 合同量/金额）
- 支持合同到期自动提醒（到期前 30 天、7 天通知）
- 支持付款节点/账期提醒（付款到期前 15 天推送财务）
- 支持未执行、即将到期、超期合同自动预警
- 支持合同变更、终止管理，变更须审批并留痕
- 合同执行数据与财务、库存模块联动，入库、付款信息同步更新至合同台账
- 支持合同全生命周期管理台账（签订、执行、变更、终止全环节数据）

### 6.9 供应商管理

**单据：** 供应商档案、供应商资质文件、供应商考评记录、黑名单管理记录

**流程要求：**

- 支持潜在供应商、合格供应商、负面供应商分类管理
- 支持供应商基础信息管理（名称、资质、联系方式、经营范围、合作记录）
- 支持供应商资质文件上传、存储、查询，资质过期自动提醒
- 合格供应商与采购环节联动，仅允许合格供应商参与采购活动
- 支持供应商考评（履约进度、产品质量、服务水平等维度），考评结果与采购联动
- 支持负面供应商（黑名单）管理，限制参与集团所有采购活动，解除限制需走审批流程
- 供应商全品类台账

### 6.10 单据设计统一要求

- 每张单据均有唯一单号
- 均有明确状态流转（草稿→待审→已审→已作废等）
- 影响库存的单据必须明确生效时点
- 触发财务的单据必须明确推送时点
- 已推财务的单据不得随意修改
- 作废、冲红、重推均需留痕

## 第七章 财务与NC接口要求

### 7.1 接口建设原则

- 业务主导：物资系统是业务发生方和数据源头
- 主数据先行：接口依赖主数据统一，须先完成映射
- NC侧零改造：物资系统主动适配NC
- 同口径：计量单位、计价方法、成本中心须两系统一致
- 可追溯：所有推送、重推、失败、差异调整可查询可审计

### 7.2 接口范围

一期须实现 **28项接口**（详见附件《接口清单》），包括：

- 主数据同步接口 5项
- 业务单据推送接口 18项
- 对账与监控接口 5项

### 7.3 关键接口要求

**（一）凭证推送**

- 所有业务单据审核通过后方可推送NC
- 推送报文采用JSON格式，包含来源单号（sourceBillNo）用于幂等校验
- 火工品出入库须**实时**推送，其他业务**准实时**推送
- 暂估和暂估冲销为**月末/月初批量**处理

**（二）异常处理**

- NC接口超时自动重推3次，间隔30秒
- NC返回业务错误记录日志，人工处理后手动重推
- 重复推送由NC幂等校验处理
- 接口状态应清晰可见（待推送/推送中/成功/失败/已重推/已作废）

**（三）对账机制**

- 日对账：当日出入库笔数及金额 vs NC当日凭证（系统自动）
- 周对账：库存余额 vs NC存货科目余额
- 月对账：全量数量/金额对账，出具对账报告
- 年对账：配合财务审计

**（四）月末暂估**

- 月末扫描已入库+发票未到单据，按订单价暂估
- 次月初红字冲销上月全部暂估
- 发票到后正式入账，与暂估差异按财务政策处理

### 7.4 接口责任边界

| 责任方 | 职责 |
|--------|------|
| 物资系统开发商 | 业务数据准备、校验、推送、重推、日志 |
| NC侧 | 接收、处理、回执、幂等校验 |
| 招标方网信办 | 统筹协调、联调组织、异常升级 |
| 招标方财务部 | 口径确认、对账、月结确认 |

### 7.5 投标方响应要求

投标方须在技术响应中说明：

- 接口实现方式（REST API / 消息队列 / 混合）
- 与NC联调的经验和方案
- 接口异常处理和补偿机制
- 对账功能实现方案
- 暂估/冲销/正式入账自动化程度

## 第八章 角色权限与审批要求

### 8.1 主要角色

系统至少应支持以下角色：

1. 集团领导/管理层（只读看板）
2. 网信办/信息化管理
3. 集团物资管理
4. 各矿物资管理
5. 仓库管理员
6. 使用单位/区队申请人
7. 采购人员
8. 质检人员
9. 财务人员
10. 安全部门
11. 系统管理员

### 8.2 权限设计要求

- 支持基于组织+角色+岗位的权限体系
- 支持数据权限（按组织/仓库/物料范围控制数据可见性）
- 人员调动后权限应能及时调整
- 离岗/离职人员应能及时禁用
- 高敏感操作（月结/反结/火工品/盘亏等）须设置加强审批

### 8.3 集团平台对接要求

**（一）统一登录（SSO）**

- **须对接集团统一平台 SSO 单点登录**，不另行建设独立用户认证体系
- 用户登录集团平台后，可直接访问物资系统，无需二次登录

**（二）组织与人员数据**

- 组织架构和人员基础数据以集团平台为权威数据源
- 物资系统通过平台 API 获取并本地缓存，不另起一套组织人员管理
- 物料编码等物资领域专属主数据由本系统作为权威源

**（三）数据权限**

- 数据隔离以二级集团为边界，与集团平台隔离机制保持一致
- 物资系统内部的业务角色和业务权限由本系统自行维护

**（四）审批兼容**

- 审批流程中组织角色解析应与集团平台体系兼容
- 可通过平台 API 获取审批人信息，或按等效规则（角色+组织过滤+层级递进）自行实现
- 审批过程和结果须完整留痕

### 8.4 审批流程要求

- 支持可配置的多级审批流程
- 支持按金额、物料类别、组织等条件差异化审批路径
- 支持手机端审批
- 支持审批超时催办
- 支持审批全过程留痕

## 第十章 技术与部署要求

### 10.1 部署方式

- 采用**私有化部署**，部署在招标方指定的数据中心或机房
- 不接受纯SaaS公有云部署

### 10.2 信创与国产化要求

- 数据库须采用 PostgreSQL 兼容的国产信创数据库（如瀚高 HGDB、人大金仓 KingbaseES 等），独立部署，不与集团平台共用数据库实例
- 操作系统须支持国产Linux（银河麒麟/统信UOS等至少一种）
- 中间件须支持国产中间件或开源中间件
- 浏览器须兼容主流国产浏览器

### 10.3 多端支持

- PC端：B/S架构，浏览器直接访问
- 移动端：APP或小程序（至少覆盖领料申请、审批、库存查询、扫码）
- 数据实时同步
- 各终端操作记录统一归集、同步更新

### 10.4 操作便捷性要求

- 优化系统界面与操作流程，简化复杂操作步骤
- 提供操作指引与提示功能，降低新用户学习成本
- 适配不同岗位业务人员的操作习惯

### 10.5 性能要求

- 支持并发用户数不低于 **200人**（待根据实际用户规模确认）
- 单据操作响应时间 ≤ **3秒**
- 报表查询响应时间 ≤ **5秒**
- 接口推送延迟 ≤ **30秒**（准实时场景）
- 系统可用性 ≥ **99.5%**

### 10.6 安全要求

- 支持HTTPS传输加密
- 支持操作日志完整记录
- 支持数据备份与恢复
- 支持防篡改、防泄露等基础数据安全控制
- 支持敏感数据脱敏（如价格、合同金额按角色控制可见性）
- 支持按角色/岗位/组织控制数据导出权限，敏感数据导出须单独留痕
- 重要导出文件应支持水印、导出日志或等效追踪机制
- 接口调用须具备鉴权机制

### 10.7 接口标准

- 系统间集成采用API方式
- 接口报文统一使用JSON格式
- 接口应具备版本管理能力
- 接口应支持幂等调用

### 10.8 技术栈与平台集成

- 开发语言不做强制限定；若投标方采用 .NET 技术栈，可直接引用集团平台模块，降低集成成本；若采用其他技术栈，须通过标准 API 完成全部平台对接
- 本系统须与集团统一平台完成对接（详见第八章 8.3 节），包括：SSO 统一登录、组织与人员数据同步、数据权限对齐、审批体系兼容
- 投标方须在技术方案中明确说明与集团平台的集成方案和技术路径

## 第十一章 统一AI入口接入要求

### 11.1 总体要求

本系统如涉及智能问答、智能查询、智能分析等AI能力建设，应以并入招标方网信办统一AI入口为前提进行设计和实施。投标方不得另行建设独立AI门户或独立智能问答入口。

### 11.2 能力开放要求

投标方应至少开放以下可供统一AI平台接入的业务能力：

1. 库存查询能力
2. 出入库业务查询能力
3. 采购与到货查询能力
4. 供应商查询能力
5. 领用与消耗分析能力
6. 预警与异常识别能力

### 11.3 接口与Tool支撑要求

- 开放标准化业务能力接口，支持结构化参数输入和结构化结果输出
- 便于注册为统一AI平台可调用的Tool
- 每项能力至少应明确功能说明、输入参数、输出结果、权限控制、错误码及典型调用示例

### 11.4 权限与安全要求

- AI查询结果须与本系统权限体系一致
- 统一AI平台不得直接访问底层数据库
- 应记录接口调用日志、权限校验日志和异常日志

### 11.5 联调配合要求

- 投标方应提供接口文档、字段说明、数据字典、示例报文
- 提供测试环境和测试账号
- 配合完成Tool注册验证和场景测试

> 详细能力清单、Tool定义、验收条款见附件。

## 第十二章 数据初始化与迁移要求

### 12.1 历史数据治理

- 投标方应配合招标方完成物料主数据清洗与整理
- 包括历史物料梳理、重复编码清理、单位统一、分类补齐

### 12.2 NC映射初始化

- 导出NC现有存货档案
- 按名称/规格匹配建立物料↔存货映射
- 无法匹配的单独列清单由业务确认
- 上线前必须完成映射完整性校验

### 12.3 期初数据导入

- 上线切换时须导入期初库存数据
- 期初数据须与NC存货余额核对一致
- 导入后须进行全量校验

### 12.4 试运行数据核对

- 试运行期间真实单据须跑通主要业务闭环
- 新旧系统应按既定范围并行运行，关键业务数据定期比对
- 接口推送结果须与NC核对
- 发现问题须在正式上线前完成整改

## 第十三章 实施、培训与上线支持要求

### 13.1 实施组织

- 投标方应指派专职项目经理
- 应组建包含业务顾问、技术开发、实施配置、培训人员的项目团队
- 重要阶段节点须到场支持

### 13.2 深化设计

- 中标后应开展需求复核和详细设计
- 应完成功能说明、原型说明、接口设计、权限配置方案
- 深化设计成果须经招标方确认后方可进入开发

### 13.3 培训要求

- 应覆盖管理员培训、业务操作培训、财务接口培训
- 培训应包含操作手册/使用指南
- 至少进行两轮培训（上线前 + 上线后）

### 13.4 上线支持

- 应提供上线切换方案
- 应提供新旧系统并行运行方案、退出条件和回退预案
- 上线首月提供现场驻场支持
- 上线后至少提供 **3个月** 运行保障期

## 第十四章 验收要求

### 14.1 功能验收

- 十四大模块核心功能均可正常运行
- 主要业务流程可闭环操作
- 审批流程可按配置执行

### 14.2 接口验收

- 28项接口均已联调通过
- 典型业务单据可稳定触发NC凭证
- 异常接口可提示、追踪、重推
- 月末暂估/冲销流程可执行

### 14.3 数据验收

- 主数据导入完整，映射校验通过
- 期初库存与NC核对一致
- 并行运行期间关键数据比对结果可追溯
- 试运行数据无重大偏差

### 14.4 报表验收

- 一期报表清单中的报表均可正常输出
- 预警功能可触发通知
- 数据口径正确

### 14.5 AI接入验收

- 核心业务能力已接入统一AI入口
- 返回结果结构化、权限正确
- 未新建独立AI门户

### 14.6 文档验收

- 用户手册/操作指南
- 系统管理手册
- 接口文档
- 培训材料
- 数据字典
