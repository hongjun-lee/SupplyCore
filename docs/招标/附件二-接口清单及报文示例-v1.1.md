## 一、附件说明

本附件为《物资供应管理系统招标技术要求-V1.1》第七章、第十四章 14.2 节的配套附件，汇总物资供应管理系统与用友 NC 财务系统之间**一期 29 项接口**的清单、技术规范和典型凭证报文示例，作为投标方进行接口设计、工作量评估和联调实施的统一依据。

本附件所列接口清单为一期建设的最小要求，凭证报文示例为代表性场景，完整字段口径、异常处理细则和联调规范以实施阶段由招标方与中标方联合确认的《接口联调规范》为准。投标方须在技术响应中承诺具备实现本附件全部 29 项接口的能力，不得以"分阶段实现""后续扩展"等方式缺项响应。

## 二、适用范围

本附件适用于：

- 主数据同步接口（物资主数据与 NC 存货档案的 N:1 映射同步）
- 业务单据推送接口（采购、领用、调拨、盘点、废旧、火工品、安全专项、委托加工等业务凭证推送）
- 对账与监控接口（日对账、周对账、月末对账、接口状态查询、映射完整性检查）

## 三、接口设计统一规范

### 3.1 正式集成标准


| 层面        | 统一要求             | 说明                                    |
| --------- | ---------------- | ------------------------------------- |
| 对外正式系统集成  | **API + JSON**   | 物资系统与 NC 的正式接口口径                      |
| 传输协议      | **HTTP/HTTPS**   | 须优先采用 HTTPS                           |
| 内部异步实现    | 允许采用消息机制         | 仅作为系统内部削峰、重试、解耦手段，不改变对外 API + JSON 标准 |
| 批量对账/定时任务 | 通过受控 API 或任务接口实现 | 不得以离线文件交换作为正式标准方案                     |


> **说明：** 如实施中确需采用消息机制，应明确其仅为内部技术实现方式，不单独改变招标和验收层面的正式接口标准。

### 3.2 统一四类状态


| 状态类别   | 统一状态                      | 说明             |
| ------ | ------------------------- | -------------- |
| 业务单据状态 | 草稿、待审、已审、已作废、已冲销          | 反映业务事实是否成立     |
| 接口推送状态 | 待推送、推送中、推送成功、推送失败、已重推、已关闭 | 反映接口执行过程       |
| 财务处理状态 | 未接收、已接收、已记账、已退回、已冲销       | 反映 NC 侧处理结果    |
| 期间状态   | 未结账、已结账、已反结               | 反映是否允许补录、重推和调整 |


### 3.3 最小回执字段

NC 侧返回接口回执时，至少须包含以下字段：


| 字段              | 类型       | 必填  | 说明                    |
| --------------- | -------- | --- | --------------------- |
| `interfaceId`   | String   | 是   | 接口编号（如 BIZ-001）       |
| `sourceBillNo`  | String   | 是   | 物资侧来源单号               |
| `orgCode`       | String   | 是   | 组织编码                  |
| `receiptStatus` | Enum     | 是   | 已接收 / 已记账 / 已退回 / 已冲销 |
| `receiptTime`   | DateTime | 是   | NC 侧处理时间              |
| `ncVoucherNo`   | String   | 否   | NC 侧凭证号（已记账时必填）       |
| `errorCode`     | String   | 否   | 错误编码（异常时必填）           |
| `errorMessage`  | String   | 否   | 错误描述（异常时必填）           |
| `retryNo`       | Integer  | 否   | 重试次数                  |


### 3.4 幂等规则

- **幂等键**：`interfaceId + sourceBillNo + orgCode` 组合
- 同一幂等键重复推送且报文一致时，NC 侧返回幂等成功，**不重复生成凭证**
- 同一幂等键重复推送但报文不一致时，应返回业务冲突并禁止覆盖
- **已结账期间**的单据不得通过重复推送规避正常审批与反结流程
- 投标方须在接口设计中提供**幂等校验日志**，可按幂等键查询和追溯

### 3.5 异常处理与重推


| 异常类型              | 处理方式                                   |
| ----------------- | -------------------------------------- |
| NC 接口超时           | 自动重推 3 次，间隔 30 秒（指数退避策略）               |
| NC 返回业务错误（如科目未配置） | 记录错误日志，人工处理后手动重推                       |
| 推送成功但 NC 未记账      | NC 侧确认，双方对账发现后补处理                      |
| 重复推送              | NC 接口幂等校验，`sourceBillNo` 唯一，重复推送直接返回成功 |
| 失败达阈值             | 触发告警，推送给运维和业务相关人员                      |


所有异常须进入**异常台账**，可查询、追踪、回溯，保留不少于 12 个月。

## 七、单价精度与尾差处理

### 7.1 精度统一规则


| 数据项     | 统一精度         | 说明                     |
| ------- | ------------ | ---------------------- |
| 移动平均单价  | **6 位小数**    | 中间计算保留 6 位，避免过早截断      |
| 入库/出库金额 | **2 位小数**    | 最终金额按 2 位小数四舍五入        |
| 数量      | **4 位小数**    | 适用于按重量（kg/t）计量的物料      |
| 税率      | 百分比整数（如 13%） | 税额 = 不含税金额 × 税率，四舍五入到分 |


> **关键约束：** 物资系统与 NC 的移动平均单价精度、金额舍入规则**必须完全一致**。投标方须在实施阶段与 NC 实施方联合确认并出具精度对照表。

### 7.2 尾差容忍与处理


| 规则      | 阈值       | 处理方式                   |
| ------- | -------- | ---------------------- |
| 单笔出库尾差  | ≤ 0.01 元 | 自动消化                   |
| 月度累计尾差  | ≤ 50 元   | 自动计入"管理费用-存货价差"        |
| 超月度累计阈值 | > 50 元   | 须查明原因（精度不一致 or 计算错误）   |
| 最后一笔出库  | 库存清零     | 金额 = 账面剩余金额（倒挤），避免残余金额 |

## 八、移动平均价保护机制

投标方须在接口设计中实现以下保护机制：


| 机制         | 说明                                  |
| ---------- | ----------------------------------- |
| **价格偏离告警** | 入库单价偏离当前移动平均 ≥ 30% 时，系统弹出告警，需人工确认   |
| **价格上限保护** | 入库单价超过当前移动平均 200% 时，系统拦截，须审批后放行     |
| **零价拦截**   | 入库单价为 0 时，系统拦截，须选择"赠品入库"类型          |
| **禁止负库存**  | 出库时检查可用库存，不足则拒绝；如业务时序问题须通过在途或预留机制处理 |

## 十二、术语表


| 术语     | 含义                                                            |
| ------ | ------------------------------------------------------------- |
| N:1 映射 | 多个物料规格对应一个 NC 存货的映射策略                                         |
| 幂等键    | 用于判定重复请求的唯一标识，本附件采用 `interfaceId + sourceBillNo + orgCode` 组合 |
| 暂估     | 月末物资已入库但发票未到的挂账操作，次月初冲销                                       |
| 反结     | 已结账期间的调整操作，须经财务审批后重新打开期间                                      |
| 滚动暂估   | 如发票跨月仍未到达，每月末重复执行"冲销 + 重新暂估"的循环                               |
| 尾差     | 因小数精度舍入产生的账面差异，按阈值自动消化或列示处理                                   |
| 三单匹配   | 采购订单 + 入库单 + 发票的三方匹配，是付款控制核心                                  |
| 让步接收   | 质检不完全合格但仍可用，供应商同意降价后的入库处理                                     |
