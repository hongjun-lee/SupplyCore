# 物资供应管理系统 — 物料编码规范

**版本：** V1.7  
**日期：** 2026-03-20  
**适用范围：** 集团型煤炭企业物资供应管理系统及与用友NC财务系统对接

---

## 一、编码目标

- **唯一性：** 一物一码，全集团唯一，不因组织不同而重复编码
- **可读性：** 大类采用2位拼音首字母，简洁且基层人员易识别
- **可扩展：** 预留充足号段，不因业务增长而重构
- **可对接：** 支持与用友NC存货编码N:1映射，物资系统主动适配NC，NC侧零改造

---

## 二、编码结构

### 格式

```
XX - XX - XXXXXX
 ↑    ↑     ↑
大类 分类  流水号
(2位)(2位)(6位)

示例：ZH-01-000001（支护材料-锚杆-第1号）
```

### 各段说明

| 段 | 位数 | 说明 |
|----|------|------|
| 大类代码 | 2位拼音首字母 | 物料一级分类，见第三章 |
| 分类代码 | 2位数字（01~99） | 物料二级分类 |
| 流水号 | 6位数字 | 按分类独立自增，从000001开始 |

**总长度：** 10位（含分隔符"-"共12位）

### 设计原则

- **编码是给系统用的，名称才是给人看的** — 大类拼音仅辅助粗识别
- **编码不含组织信息** — 库存归属通过"组织/仓库"字段区分
- **两级分类足够** — 更细的区分（规格、材质等）通过物料属性字段实现

## 六、编码规则约定

### 6.1 基本原则

1. **一物一码：** 同一物料不论采购来源、所属矿厂、存放仓库，全集团唯一编码
2. **只增不改：** 编码一经生效不得修改，停用物料标记"停用"状态
3. **不删不复用：** 已用编码永久保留，停用号不再分配给新物料
4. **编码不含业务属性：** 规格、供应商、组织等信息存入对应字段，不嵌入编码

### 6.2 组织与库存管理

物料编码不区分组织。库存归属通过记录字段区分：

```
物料编码：ZH-01-000001（左旋螺纹钢锚杆）

一号矿主仓库：库存 5,000 根
二号矿井下仓：库存 3,000 根
洗煤厂仓库：  库存 0
```

### 6.3 供应商物料号处理

供应商各自的货号存入**"供应商物料号"**字段，独立于本系统编码。

### 6.4 火工品特殊管控

火工品（HG）大类须执行：
- 独立台账管理，双人双锁
- 领用实名制，按需领用
- 库存变动实时上报
- 财务科目单独核算

### 6.5 新增物料流程

```
使用单位提申请（填写物料名称、规格、用途等）
        ↓
集团物资部审核（查重、归类、确定分类代码）
        ↓
系统自动生成编码（大类+分类+流水号）
        ↓
判断是否有对应NC存货？
    ├─ 有 → 填写 nc_inv_code，完成映射
    └─ 无 → 通知NC管理员创建存货档案，回填 nc_inv_code
        ↓
编码生效，各单位可用于业务单据
```

> 详细流程见第七章 7.5。

## 七、与用友NC对接规则

> **背景：** 物资系统与用友NC由不同开发商独立建设，存在历史存量数据，两系统编码体系无法统一，采用 **N:1映射** 进行关联——物资系统作为开发方主动适配NC，NC侧零改造。

---

### 7.1 核心设计理念

> **物料是存货的明细，存货是物料的财务轨迹。**

| 维度 | 物资系统（物料） | 用友NC（存货） |
|------|----------------|--------------|
| 管理对象 | 实物，按规格区分 | 资金，按科目归集 |
| 粒度 | 细（一规格一编码） | 粗（同类合并核算） |
| 典型例子 | 锚杆 Φ20×2400mm、锚杆 Φ22×2400mm | 锚杆（统一挂原材料科目） |
| 关系 | 存货的明细 | 物料的汇总财务轨迹 |

**映射关系：物资 N → NC 1**
多个物料规格共享同一NC存货编码，推送财务时按NC存货汇总。

---

### 7.2 数据结构设计

**方案：`nc_inv_code` 直接作为物料表字段**

> **注意：** 以下 SQL 为示意用途，采用 MySQL 语法。实际开发时须根据最终选定的数据库（如达梦、人大金仓等信创数据库）调整语法。

```sql
-- 物料主表新增字段
ALTER TABLE material ADD COLUMN nc_inv_code VARCHAR(50) COMMENT 'NC存货编码（N:1）';
ALTER TABLE material ADD COLUMN nc_inv_name VARCHAR(100) COMMENT 'NC存货名称（冗余，便于核对）';

-- 多组织补充表（当同一物料在不同矿/组织的NC存货编码不同时使用）
CREATE TABLE material_nc_org (
  id          BIGINT PRIMARY KEY,
  mat_code    VARCHAR(20)  NOT NULL COMMENT '物资物料编码',
  nc_org_code VARCHAR(20)  NOT NULL COMMENT 'NC核算组织编码',
  nc_inv_code VARCHAR(50)  NOT NULL COMMENT '该组织下的NC存货编码',
  status      TINYINT      DEFAULT 1 COMMENT '0=停用 1=有效',
  remark      VARCHAR(200),
  UNIQUE KEY uk_mat_org (mat_code, nc_org_code)
);
```

**查找优先级：**
1. 先查 `material_nc_org`（组织级精确匹配）
2. 无记录则取物料表的 `nc_inv_code`（默认映射）
3. 仍为空 → 拦截推送，触发告警

---

### 7.3 推送NC逻辑

```
物资出入库明细
    ↓
按 nc_inv_code 分组汇总（数量、金额）
    ↓
生成NC入库单/出库单（存货粒度）
    ↓
NC记账，挂对应会计科目
```

NC看到的永远是存货维度的汇总数据，感知不到物资侧的规格细分。

---

### 7.4 历史数据初始化

1. 导出NC现有存货档案（invcode + invname）
2. 人工或脚本按名称匹配，为物资物料表批量填写 `nc_inv_code`
3. 无法匹配的物料单独列清单，业务人员确认后补录
4. **上线前必做校验：** 检查是否存在NC存货比物资物料更细的反例，局部用 `material_nc_org` 补充处理

---

### 7.5 新增物料流程

```
使用单位在物资系统提申请
        ↓
集团物资部审批，生成物料编码
        ↓
判断是否有对应NC存货？
    ├─ 有 → 直接填写 nc_inv_code，完成
    └─ 无 → 通知NC管理员创建存货档案
              ↓
           NC建档完成，回填 nc_inv_code，完成
```

---

### 7.6 映射监控

| 频率 | 内容 |
|------|------|
| 每日 | 统计 nc_inv_code 为空的有效物料数，异常告警 |
| 每周 | 核对当周新增物料是否已完成映射 |
| 每月 | 全量对账：物资出入库记录 vs NC凭证，按nc_inv_code核对数量/金额差异 |

## 八、编码查重与校验

1. 新增物料时系统自动查重：编码唯一 + 名称规格组合唯一
2. 推送NC前校验存货编码是否已存在
3. 每月运行对账，比对两系统编码差异并告警

## 九、物料属性字段定义

> 编码不含业务属性，规格、材质等信息通过以下字段存储。

### 9.1 通用属性字段（所有物料必填）

| 字段名 | 字段说明 | 示例 |
|--------|----------|------|
| mat_name | 物料名称 | 左旋螺纹钢锚杆 |
| mat_spec | 规格型号 | Φ20×2400mm |
| unit | 计量单位 | 根 |
| mat_category | 大类代码 | ZH |
| mat_sub_category | 分类代码 | 01 |
| brand | 品牌/厂家 | （可选） |
| remark | 备注 | （可选） |

### 9.2 特殊属性字段

| 字段名 | 适用大类 | 说明 |
|--------|----------|------|
| shelf_life_days | HG、HX | 保质期（天），0表示无期限 |
| batch_required | HG、HX | 是否启用批次管理（Y/N） |
| hazard_level | HG | 危险品等级（1/2/3级） |
| barcode | 全部 | 条码/二维码内容（默认与物料编码一致，特殊情况可独立维护） |
| loss_rate | YZ、HG、HX | 合理损耗定额（%），0表示无损耗，入库时系统据此判断差异是否在定额内 |

### 9.3 批次管理规则（HG / HX）

- 启用批次管理的物料，入库时必须录入**生产日期**和**批次号**
- 系统自动计算**到期日**（生产日期 + 保质期）
- 到期前 **30天** 触发预警通知，到期当日自动冻结库存，禁止领用
- 领用时遵循 **先进先出（FIFO）** 原则，系统自动推荐最早批次

## 十一、条码 / 二维码集成规范

### 11.1 默认方案

- 物料编码（如 `ZH-01-000001`）直接作为条码内容，生成 **Code 128** 条形码或 **QR Code**
- 打印标签时同时打印：物料编码、物料名称、规格型号、计量单位

### 11.2 批次物料扫码规范

启用批次管理的物料（HG/HX），标签内容扩展为：

```
物料编码 + "-" + 批次号
示例：HG-01-000001-20260315A
```

扫码时系统自动拆分，识别物料编码和批次号。

### 11.3 手持终端要求

- 支持 Code 128、QR Code 扫码
- 出入库、盘点操作均应通过扫码完成，禁止手工录入编码

## 十四、与辽宁能源集团编码扩展预留

> **背景：** 阜矿集团为辽宁能源二级集团。辽宁能源尚未形成统一物料编码标准，但未来存在集团层面统一编码的可能。本章用于明确一期建设中必须满足的扩展预留要求，确保阜矿编码体系在大集团标准落地时可平滑对接，**无需重新编码**。

---

### 14.1 设计原则

**一期要求：**

- 本系统一期建设中，必须预留与辽宁能源集团未来统一编码体系对接的字段、映射关系和接口扩展能力
- 在辽宁能源统一编码标准尚未正式发布前，不要求一期直接启用集团统一编码
- 不得以未来集团统一编码为由，要求对阜矿现行物料编码体系整体推倒重建

> **阜矿编码是主体，不因上级标准而改动。**

与NC对接的逻辑完全一致：

| 层级 | 编码主体 | 对接方式 |
|------|----------|----------|
| 阜矿物资系统 | 本系统物料编码（XX-XX-XXXXXX） | 主体，永久不变 |
| 用友NC | 存货编码 | N:1 映射（`nc_inv_code`字段） |
| 辽宁能源集团 | 集团统一物料码（待定） | 待定映射（`group_mat_code`字段，预留M:N能力） |

无论辽宁能源标准如何制定，阜矿内部编码不受影响，原则上通过维护对应关系表完成适配。

---

### 14.2 数据结构预留

为满足一期招标和实施约束，数据模型至少应支持以下扩展预留：

```sql
-- 物料主表预留字段（暂时为空，等大集团标准落地后回填）
ALTER TABLE material ADD COLUMN group_mat_code VARCHAR(50) DEFAULT NULL COMMENT '辽宁能源集团物料编码（待对接）';
ALTER TABLE material ADD COLUMN group_mat_name VARCHAR(100) DEFAULT NULL COMMENT '集团物料名称（冗余，便于核对）';

> **注意：** 以下 SQL 为示意用途，采用 MySQL 语法。实际开发时须根据最终选定的数据库调整语法。
-- 跨集团映射表（预建，暂时不用）
CREATE TABLE material_group_map (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT,
  mat_code        VARCHAR(20)  NOT NULL COMMENT '阜矿物料编码',
  group_mat_code  VARCHAR(50)  NOT NULL COMMENT '辽宁能源集团物料编码',
  group_org_code  VARCHAR(20)  DEFAULT NULL COMMENT '集团核算组织（如有差异）',
  map_ratio       DECIMAL(10,4) DEFAULT 1.0000 COMMENT '换算比例（单位不同时使用）',
  status          TINYINT      DEFAULT 1 COMMENT '0=停用 1=有效',
  remark          VARCHAR(200),
  created_at      DATETIME,
  UNIQUE KEY uk_mat_group (mat_code, group_mat_code)
);
```

---

### 14.3 对接时的工作内容

当辽宁能源发布统一编码标准后，需完成以下工作（与NC初始化对接类似）：

1. 获取辽宁能源物料/存货档案
2. 按名称、规格进行人工或脚本匹配，填写 `group_mat_code`
3. 无法匹配的物料，向大集团申请新增或确认归类
4. 上线后定期与大集团平台数据对账

> **预计工作量：** 视大集团标准粒度而定。若大集团粒度比阜矿粗（类似NC），则映射工作量小；若大集团粒度更细，则需分析后决定是否拆分阜矿现有物料编码。

---

### 14.4 编码体系兼容性说明

阜矿当前编码设计已考虑兼容性：

- **大类代码** 采用行业通用拼音缩写，与煤炭行业惯用分类一致，和大集团标准冲突概率低
- **编码不含组织信息**，集团统一时无需剥离组织前缀
- **唯一性范围** 当前为阜矿集团内，若大集团要求全集团唯一，阜矿编码天然满足（集团内唯一 ⊆ 全集团唯一，只需确认流水号不与其他二级集团重复）
