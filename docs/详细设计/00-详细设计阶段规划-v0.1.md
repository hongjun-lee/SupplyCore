**版本：** V0.1
**日期：** 2026-05-01
**文档性质：** 详细设计阶段元文档/规划文件
**适用阶段：** 详细设计启动准备、详设过程治理、详设交付验收

---

## 一、文档目的

本文档用于在详细设计阶段正式启动前，先固化以下事项：

- 详设阶段的输入清单（来自概要设计 00-07、需求梳理、治理模板和详细规则层）
- 详设拆分原则与文档目录结构
- 每篇详设文档的深度边界（什么必须固化、什么留到原型/实施配置/环境准备）
- 与概要设计 `00-07` 的承接关系（细化 / 新做 / 不动）
- 业务确认轮未回执时各章节的占位策略
- 详设阶段的退出条件与交付物清单

本文档不替代任何具体详设文档，只定义详设阶段的"游戏规则"。具体业务实体、字段、接口、状态机和审批流应在后续详设分卷中固化。

## 三、详设拆分原则

### 3.1 沿用概要设计能力域归并

直接沿用概要设计的能力域结构（基础底座、计划与采购协同、合同与资金、库存实物流转、设备与租赁、业财与管理分析），但在每个能力域下细化到具体单据、字段、接口和状态机。

### 3.2 五层向下细化

每个模块至少要细化到以下五层：

| 层级 | 内容 | 必须固化 |
| --- | --- | --- |
| 模块 | 模块职责、上下游关系、对外能力 | 是 |
| 单据 | 主表、明细表、状态字段、关联键、业务规则 | 是 |
| 字段 | 字段名、类型、约束、索引建议、值域、注释 | 是 |
| 接口 | 请求/响应字段表、报文样例、错误码框架 | 是 |
| 配置 | 默认值、开关、阈值占位、审批维度 | 框架是、最终值否 |

### 3.3 数据库锁到逻辑模型，不锁厂商方言

详设固化逻辑数据模型（实体、字段、约束、索引建议）和默认 SQL 类型推荐；具体厂商方言、物理分区策略、信创数据库专有特性留到环境准备阶段。

### 3.4 业务确认未回执项采用占位策略

涉及 `12 附录 A` 业务确认轮事项的章节：保留章节框架与字段结构，具体阈值、默认值、初始数据标注"待业务确认（参见 13/14/15）"占位，便于回执到位后直接回填。

### 3.5 不重复概要设计已有内容

详设引用概要设计的边界、范围、能力域归并和总体架构，不重复展开；如发现概要设计存在缺漏或不准确，应回写概要设计而非在详设中独立另写。

## 五、详设深度边界

### 5.1 必须固化（每篇详设的输出物）

- 实体关系图（ERD），覆盖模块内全部主表、明细表、关联键
- 表结构（字段名、类型、长度、空值约束、默认值、唯一约束、外键、索引建议、注释）
- 状态机图（节点、迁移条件、保护条件、回退路径）和状态字段值域
- 接口字段表（请求/响应、报文样例、字段说明、错误码框架）
- 审批流框架（节点、维度、是否会签、是否升级、不锁阈值与最终人员）
- 报表字段口径定义（字段、来源、计算规则、统计周期、权限维度）
- 配置项默认值矩阵（参数名、默认值、可调范围、调整责任方）
- 单据生命周期与库存影响、财务影响、接口触发关系

### 5.2 不固化（留到原型 / 实施配置 / 环境准备）

| 不固化事项 | 留到的阶段 |
| --- | --- |
| 数据库厂商方言、SQL 优化、分区分库策略、特定 Hint | 环境准备阶段 |
| 接口最终错误码全集、特定异常补偿步骤 | 联调阶段 |
| 审批节点最终人员清单、最终金额阈值 | 实施配置阶段 |
| 页面交互细节、按钮位置、字体、配色 | 原型设计阶段 |
| 看板最终图表样式、字段刷新频率、主题排序 | 原型设计阶段（看板最小清单） |
| 验收用例最终参数、培训材料、用户手册 | 实施准备阶段 |
| 部署脚本、CI/CD 配置、运维手册 | 环境准备 + 运维交接阶段 |

## 六、业务确认未回执的占位策略

`12 附录 A` 中 6 项业务确认轮事项对详设的影响如下，应在详设对应章节按"占位策略"列处理。

| 业务确认事项 | 来源 | 影响详设位置 | 占位策略 |
| --- | --- | --- | --- |
| 计量单位字典与换算规则 | `15` 待业务确认 | `02 基础档案`、`03 物料主数据` | 字段结构与字典表固化；初始数据列"待业务确认（参见 15）" |
| 并行运行退出条件 | `13` 待业务确认 | `11 非功能` | 章节框架固化；具体阈值标注"待项目领导小组确认" |
| NC 存货映射与科目模板 | `14` 待业务确认 | `08 NC 接口`、`03 物料主数据` | 字段、状态机、开关固化；初始映射数据"待 NC 落地" |
| 基础管理看板最小清单 | 由原型阶段承接 | `09 报表 AI` | 看板章节先建框架；具体主题与指标"待原型确认" |
| 上线回退决策权与执行路径 | 决策权已写实，执行步骤后置 | `11 非功能` | 决策权写实"项目领导小组"；执行步骤"待上线方案承接" |
| 集团组织与 NC 核算组织正式映射 | 待 NC 账套落地 | `08 NC 接口`、`02 基础档案` | 字段、状态、配置入口固化；正式映射台账"待 NC 落地" |

占位文本统一使用 `[待业务确认 - 来源 XX]` 或 `[待 NC 落地]` 格式，便于后续批量检索回填。

## 七、详设阶段退出条件

详设结束转入开发实施前，应满足以下全部条件。

### 7.1 文档完整性

- 11 篇详设分卷（`01` ~ `11`）全部完成 V1.0 首版
- 每篇文档通过项目内部交叉评审，评审记录留痕
- 各篇之间无重复定义、无相互冲突，跨文档术语统一

### 7.2 可开发性

- 表结构能直接生成 DDL（不依赖单一厂商方言）
- 接口字段表能直接作为开发与联调输入
- 状态机能驱动业务单据全生命周期，无未定义状态迁移
- 配置项默认值矩阵可直接落到环境配置与初始化脚本

### 7.3 占位项可控

- 所有 `[待业务确认]`、`[待 NC 落地]` 占位项在文档中可被检索定位
- 占位项不阻塞开发启动（即开发可先按结构写代码，等回执回来再回填）
- 每个占位项明确对应业务确认轮中的哪一项

### 7.4 与上游一致

- 与招标 V1.1 范围一致，未隐藏扩项也未漏项
- 与概要设计 `00-07` 不冲突；差异点单独形成"概要设计调整建议清单"，并由项目内部评审通过后回写概要设计
- 与 `04` 节九承接映射保持一致，每条已确认结论在详设中找得到具体落实位置

## 八、本规划之外的事项

以下事项不在详设阶段、也不在本规划范围内：

- 具体编码（属实施开发阶段）
- 单元测试用例（属实施开发阶段）
- 部署脚本、环境配置（属环境准备阶段）
- 培训材料、用户手册（属实施准备阶段）
- 联调测试用例、联调脚本（属联调阶段）
- 运维手册、巡检方案（属运维交接阶段）

详设只为以上事项提供输入（字段、接口、状态、配置），不替代它们。

## 十、一句话结论

详设的核心不是把概要设计抄一遍，而是把"开发能直接动手、联调能直接对、运维能直接接"的字段、接口、状态、配置、流程模板逐项落实。本规划用于把详设的范围、深度、占位策略提前定下来，避免详设过程因边界不清反复返工，也避免业务确认未到位时强行锁死最终值。
