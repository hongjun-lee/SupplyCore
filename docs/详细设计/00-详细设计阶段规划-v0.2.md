**版本：** V0.2

**适用阶段：** 详细设计启动准备、详设过程治理、详设交付验收、后续调研回写与开发启动门治理

## 一、文档目的

本文档用于固化详细设计阶段的"游戏规则"，并在后续调研、政策解析、业务方答复和交叉评审持续进入后，作为详细设计基线治理依据。

本文档重点定义：

- 详设阶段的输入清单与输入层级
- 详设主分卷和配套治理文档的目录结构
- 每篇详设文档的深度边界
- 与概要设计、流程调研、政策解析、业务方答复和开发启动门的承接关系
- 占位项、外部依赖项、阶段性动作的登记口径
- 详设阶段退出条件与当前基线状态
- 后续新增调研内容是否需要回写本文档的判断规则

本文档不替代任何具体详设分卷。具体业务实体、字段、接口、状态机、审批流、报表口径和业务规则，应落实到 `01~11` 分卷及 10A 等配套文档中。

## 三、详设拆分原则

### 3.1 沿用概要设计能力域归并

直接沿用概要设计的能力域结构（基础底座、计划与采购协同、合同与资金、库存实物流转、设备与租赁、业财与管理分析），但在每个能力域下细化到具体单据、字段、接口和状态机。

### 3.2 五层向下细化

每个模块至少要细化到以下五层：

| 层级 | 内容 | 必须固化 |
| --- | --- | --- |
| 模块 | 模块职责、上下游关系、对外能力 | 是 |
| 单据 | 主表、明细表、状态字段、关联键、业务规则 | 是 |
| 字段 | 字段名、类型、约束、索引建议、值域、注释 | 是 |
| 接口 | 请求/响应字段表、报文样例、错误码框架 | 是 |
| 配置 | 默认值、开关、阈值占位、审批维度 | 框架是、最终值按配置/外部确认管理 |

### 3.3 数据库锁到逻辑模型，不锁厂商方言

详设固化逻辑数据模型（实体、字段、约束、索引建议）和默认 SQL 类型推荐；具体厂商方言、物理分区策略、信创数据库专有特性留到环境准备阶段。

### 3.4 占位与外部依赖分层管理

详设不再将所有未完全落地事项统称为"待业务确认"。后续统一分三类处理：

| 类型 | 处理方式 |
| --- | --- |
| 已有项目内决议 | 直接回写对应详设分卷，并在版本记录或业务依据中留痕 |
| 已有项目内口径、但依赖外部动作 | 登记到 `待外部确认事项汇总`，并在对应分卷标注触发时点、责任方和不阻塞口径 |
| 尚未形成口径 | 使用 `[待外部确认 - 责任方/触发时点]`、`[待 NC 落地]`、`[待联调验证]` 等明确占位，不使用泛化的"待确认" |

### 3.5 不重复概要设计已有内容

详设引用概要设计的边界、范围、能力域归并和总体架构，不重复展开；如发现概要设计存在缺漏或不准确，应形成概要设计调整建议并回写概要设计，而不是在详设中孤立另写一套口径。

### 3.6 新增调研按影响面回写

新增流程调研、政策解析、业务方答复或拍板材料进入后，应先判断影响面：

- 只影响单一业务规则、字段、接口、状态机或审批模板：回写对应 `01~11` 分卷或 10A，不必改本文档。
- 影响多个分卷之间的基线关系、开工条件、外部依赖或交付判断：同步更新交付总说明、启动门对照表、待外部确认事项汇总或交叉评审记录。
- 影响详设拆分、输入层级、深度边界、占位策略或退出条件：才需要更新本文档。

## 五、详设深度边界

### 5.1 必须固化（每篇详设的输出物）

- 实体关系图（ERD），覆盖模块内全部主表、明细表、关联键
- 表结构（字段名、类型、长度、空值约束、默认值、唯一约束、外键、索引建议、注释）
- 状态机图（节点、迁移条件、保护条件、回退路径）和状态字段值域
- 接口字段表（请求/响应、报文样例、字段说明、错误码框架）
- 审批流框架（节点、维度、是否会签、是否升级、不锁最终人员）
- 报表字段口径定义（字段、来源、计算规则、统计周期、权限维度）
- 配置项默认值矩阵（参数名、默认值、可调范围、调整责任方）
- 单据生命周期与库存影响、财务影响、接口触发关系
- 业务依据与决议来源（调研流程、政策解析、业务方答复、Spencer 拍板或交叉评审记录）
- 跨分卷影响说明（涉及多个分卷时，必须指出同步落点）

### 5.2 不固化（留到原型 / 实施配置 / 环境准备）

| 不固化事项 | 留到的阶段 |
| --- | --- |
| 数据库厂商方言、SQL 优化、分区分库策略、特定 Hint | 环境准备阶段 |
| 接口最终错误码全集、特定异常补偿步骤 | 联调阶段 |
| 审批节点最终人员清单、最终金额阈值 | 实施配置阶段 |
| 页面交互细节、按钮位置、字体、配色 | 原型设计阶段 |
| 看板最终图表样式、字段刷新频率、主题排序 | 原型设计阶段 |
| 验收用例最终参数、培训材料、用户手册 | 实施准备阶段 |
| 部署脚本、CI/CD 配置、运维手册 | 环境准备 + 运维交接阶段 |
| OAuth `client_id` / `client_secret` 等真实凭据 | 运维颁发，不进入公开文档 |
| NC 真实账套、正式科目映射和接口联调结果 | NC 落地与联调阶段 |

## 七、详设阶段退出条件与当前状态

### 7.1 文档完整性

- `01~11` 详设主分卷全部完成 V1.0+ 当前基线
- 10A 权限审计配套、角色权限矩阵、Catio 字段缺口清单已形成配套交付
- 交付总说明、开发实施引用索引、启动门对照表、外部确认事项汇总、交叉评审记录已形成
- 各篇之间无当前已知的版本引用、字段值域、角色权限或审批模板冲突

### 7.2 可开发性

- 表结构能直接生成 DDL（不依赖单一厂商方言）
- 接口字段表能直接作为开发与联调输入
- 状态机能驱动业务单据全生命周期，无当前已知未定义状态迁移
- 配置项默认值矩阵可直接落到环境配置与初始化脚本
- 权限矩阵和审批模板已具备种子数据与 Workflow 配置依据

### 7.3 占位项可控

- 当前剩余事项均已转为外部回函、凭据下发、NC 联调、消息联调、运维交接或 SLA 验证等阶段性动作
- 这些事项不阻塞当前详设交付和开发实施准备，但需要在触发时点回填
- 每个事项均应在 `待外部确认事项汇总` 中登记责任方和触发时点

### 7.4 与上游一致

- 与招标 V1.1 范围一致，未隐藏扩项也未漏项
- 与概要设计 `00-07` 不冲突；若后续发现概要设计需调整，应形成调整建议并回写

### 7.5 当前结论

截至 2026-05-10，详细设计文档层面已形成可交付基线；`开发启动门对照表-V1.0.md` 显示 11 个模块整体通过 6 项开工条件检查，无文档层面的阻塞项。

## 八、后续新增内容回写规则

### 8.1 需要更新本文档的情况

出现以下任一情况，应先更新本文档，再同步相关分卷或治理文档：

1. 详设主分卷数量、命名、拆分边界或顺序发生变化；
2. 新增一类长期输入源，例如新的政策附件体系、统一平台规范或外部系统集成规范；
3. 详设深度边界发生变化，例如原本留到实施配置的内容改为详设必须固化；
4. 占位策略或外部依赖登记口径发生变化；
5. 详设退出条件、开发启动门判断标准或交付包范围发生变化。

### 8.2 不需要更新本文档的情况

以下情况一般不更新本文档，直接回写对应分卷和治理文档即可：

1. 某个业务规则从 A 改成 B；
2. 某个字段、状态、接口、审批节点或报表口径调整；
3. 新增政策解析只作为某条规则的依据；
4. 业务方答复只影响一个或少数几个分卷；
5. 拍板材料只作为过程留痕，不改变详设拆分、深度或退出条件。

### 8.3 本轮更新判断

本轮新增内容不仅包含单点业务规则，还新增了流程调研 / 政策解析 / 业务方答复 / 拍板材料作为持续输入源，并新增交付总说明、启动门对照表、待外部确认事项汇总、交叉评审记录等治理文档。因此需要将本文档从 V0.1 启动规划更新为 V0.2 基线治理口径。

## 九、本规划之外的事项

以下事项不在详设阶段、也不在本规划范围内：

- 具体编码（属实施开发阶段）
- 单元测试用例（属实施开发阶段）
- 部署脚本、环境配置（属环境准备阶段）
- 培训材料、用户手册（属实施准备阶段）
- 联调测试用例、联调脚本（属联调阶段）
- 运维手册、巡检方案（属运维交接阶段）
- 真实账号、密钥、凭据和生产环境连接信息（不得写入本文档）

详设只为以上事项提供输入（字段、接口、状态、配置），不替代它们。
