# 阜矿集团 NC 财务接口能力内部底稿 V0.2

**文档目的：** 本文档是阜矿集团与用友 NC 财务系统对接前的**内部工作底稿**，用于梳理“需要 NC 团队确认 / 开放 / 配合提供哪些通用财务接口能力”。本文不作为直接外发函件；对外发文使用 [`08B-阜矿集团NC财务接口能力确认函-V0.1.md`](08B-阜矿集团NC财务接口能力确认函-V0.1.md)。

内部梳理口径：本需求代表阜矿集团对 NC 财务系统接口能力的通用性要求，不局限于物资管理系统。物资管理作为首个切入场景，用于梳理集团业务系统普遍需要的**主数据、统一凭证接入、回执/对账、账务明细查询**等能力。对外不要求 NC 团队逐项实现物资侧 20 个“制证功能”，而是要求 NC 提供可承接集团业务系统凭证数据的**统一凭证接入能力**（或 NC 现有接口平台的等价能力），并确认凭证模板、报文字段、回执、错误码和联调边界。物资侧内部仍可按详设 [`08-财务与NC接口详细设计-V1.1.md`](08-财务与NC接口详细设计-V1.1.md) §五管理 BIZ-001~BIZ-020 业务场景；对 NC 发函时按“能力 + 场景范围”表达，避免把内部业务拆分误写成 NC 必须交付的 20 个独立 endpoint。回函后联动详设 08 升 V1.2 + F-14 `interface_definition` 预置数据落库。

**建议外部回函 SLA：** T+10 工作日。NC 项目尚在不同就绪阶段（详设 08 §4.3 三段式），允许分阶段回函；最迟首笔正式凭证上线前 30 天前完成对照（即详设 08 §4.3 "NC 正式启用"前 30 天）。

**优先级口径：** P0 = 联调启动前必需（包括首批 MD-001 / MD-003 + **统一凭证接入能力** + 采购入库 / 领料出库 / 对厂矿销售出库等样例模板）；P1 = UAT 前补齐主要凭证模板、回执/对账和应收 / 应付等账务明细只读查询；P2 = 正式上线前补齐低频模板；P3 = 上线后可补。本清单按章节标记。

## 零、对接定位与沟通口径

本底稿用于支撑对用友 / NC 团队的发文，核心目的是：**以物资管理业务为首个切入点，提炼出阜矿集团需要 NC 团队确认、开放或配合提供的通用财务接口能力**。

形成外发函件时，沟通重点不是介绍物资系统内部怎么做，也不是要求 NC 团队理解全部物资业务流程，而是以阜矿集团口径请 NC 团队明确以下事项：

1. **当前 NC 能力现状**：账套、核算组织、科目、接口平台是否已具备联调条件；
2. **统一凭证接入方式**：是否提供统一凭证接收接口，或必须按 NC 凭证模板 / 交易类型拆分；
3. **凭证模板与字段**：采购入库、领料出库、对厂矿销售出库等典型场景分别对应哪些 NC 凭证模板、科目和辅助核算字段；
4. **回执与状态查询**：如何返回凭证号、记账状态、失败原因，采用同步返回、异步回调还是查询接口；
5. **主数据接口**：物料、计量单位、成本中心、核算组织等主数据由谁提供、怎么同步、是否支持增量；
6. **账务明细查询**：是否提供应收、应付等明细账只读查询接口，用于业务对账、付款 / 收款状态核验和凭证穿透；
7. **对账与期间控制**：是否提供日 / 月对账、凭证状态查询、期间状态查询、补推和反结相关接口；
8. **接口基础条件**：鉴权方式、环境地址、错误码、幂等规则、限流、日志保留、联调联系人和上线窗口。

对外沟通时应避免以下表述：

- ❌ “请 NC 团队实现 20 个物资制证功能”
- ❌ “请 NC 团队理解并落地全部物资业务流程”
- ❌ “请 NC 按物资侧 BIZ-001~BIZ-020 拆 20 个独立接口”

推荐表达为：

- ✅ “请 NC 团队确认统一凭证接入能力和凭证模板覆盖范围”
- ✅ “请 NC 团队确认主数据、凭证接入、回执状态、对账查询等接口功能”
- ✅ “物资管理场景仅作为首个切入点和举例，不作为 NC 独立功能拆分要求；相关能力应具备集团业务系统通用性”

本文中出现 BIZ-001 / BIZ-005 / BIZ-005A 等编码时，仅用于举例说明物资管理场景会产生哪些类型的凭证数据，便于 NC 团队判断模板、字段和接口能力是否覆盖；这些编码不限制后续其他业务系统复用同类 NC 财务接口能力。

## 三、NC 必须确认 / 提供的能力与接口清单（核心，P0~P2）

### 3.1 统一凭证接入能力（P0~P2）

> **写法原则：** 本节只提炼“请 NC 确认 / 提供什么”。BIZ 编码只作为业务场景举例，不要求 NC 按这些编码拆独立功能。

#### 3.1.1 NC 需要确认 / 提供的能力

| 能力项 | 需求描述 | 优先级 |
|--------|----------|--------|
| 统一凭证接收 | NC 是否支持一个统一凭证接入 endpoint（如 `POST /nc/vouchers`），由 `voucher_template_code` / `business_scene_code` / `transaction_type` 区分采购入库、领料出库、销售出库等场景；如 NC 只能按模板拆 endpoint，也请说明拆分规则 | **P0** |
| 凭证模板 / 交易类型 | 提供首批凭证模板编码、名称、适用场景、借贷科目规则、辅助核算维度要求 | **P0** |
| 报文字段 | 明确统一凭证请求字段：来源系统、来源单号、业务场景编码、核算组织、期间、币种、金额、税额、摘要、行明细、辅助核算、附件 / 追溯字段等 | **P0** |
| 回执机制 | 返回或回调 `nc_voucher_no`、`task_id`、`finance_state`、`accounted_time`、失败原因；支持同步返回或异步回执 / 轮询 | **P0** |
| 红冲 / 冲销 | 明确红冲走同一接口 + `reverse_flag`，还是单独红冲能力；明确原凭证号引用规则 | P1 |
| 幂等与重推 | 支持按物资侧 `idempotent_key` / 来源单号去重；明确重复推送、失败重推、已记账后重复推送的处理规则 | **P0** |
| 错误码 | 返回业务校验错误（科目不存在 / 期间关闭 / 辅助核算缺失）、系统错误、权限错误、重复请求错误等错误码 | **P0** |
| 批量能力 | 是否支持一批多张凭证 / 一张凭证多行明细；单次报文大小、行数、QPS、并发上限 | P1 |
| 附件与追溯 | 是否支持来源单据 URL / 附件 ID / 业务摘要 / 物资明细追溯字段写入 NC 凭证附件或摘要 | P2 |

> **特别说明（BIZ-005 / BIZ-005A）：** 详设 08 §5.2 已明确：对厂矿（集团内部用户单位）出库按外部销售实质开票（视同销售），凭证模板与自用出库不同。对外表达时不要求 NC 做两个独立 endpoint，但 NC 端**必须支持两套凭证模板 / 交易类型分别制证**，不能在统一接口后端按同一模板混合处理。

### 3.2 主数据类（MD-001 ~ MD-005，共 5 个，P0~P1）

| 接口编码 | 方向 | NC 必须实现 | 期望路径 | 优先级 |
|---------|------|----------|---------|--------|
| MD-001 | 物资 → NC | **接收端**：物料/存货档案推送 + 启停状态 | `POST /nc/masterdata/material` | **P0** |
| MD-002 | 物资 → NC | **接收端**：物料停用通知 + NC 端阻塞规则（如有未完结业务单据时 NC 端返回阻塞错误码） | `POST /nc/masterdata/material/disable` | P1 |
| MD-003 | 双向 | **读 + 写**：计量单位字典；明确以谁为源 | `GET/POST /nc/masterdata/unit` | **P0** |
| MD-004 | NC → 物资 | **NC 提供查询 endpoint**：成本中心全量 + 增量（`?since=yyyy-MM-ddTHH:mm:ss`）；支持订阅推送更佳 | `GET /nc/masterdata/cost-center` | P1 |
| MD-005 | NC → 物资 | **NC 提供查询 endpoint**：核算组织树 + 与集团二级组织（阜矿子树）映射 | `GET /nc/masterdata/accounting-org` | **P0** |

**MD 类必须回函的字段：**
- 各档案的 NC 主键字段（`material_id` / `cost_center_id` / `accounting_org_id`）+ 编码字段 + 名称字段 + 状态字段
- 增量拉取的 `since` 时间戳精度（秒/毫秒）+ 时区约定
- NC 主数据的码表（如物料类别、计量单位类型等）

### 3.3 回执 / 对账类（F-03 回执 + CHK-001 ~ CHK-004，共 5 个，P0~P2）

NC 必须实现以下接口的**任一种**（NC 端能力决定选择）：

| 接口 | 选项 A（推荐） | 选项 B（兜底） |
|------|-------------|-------------|
| F-03 凭证回执 | NC 主动回调物资侧 webhook：`POST {supplycore-base}/nc-callback/voucher-receipt` | NC 提供查询接口让物资轮询：`GET /nc/voucher/{task_id}/receipt` |
| CHK-001 日对账 | NC 主动推送日汇总到物资 webhook | NC 提供查询：`GET /nc/reconcile/daily?date=yyyy-MM-dd&org=___` |
| CHK-002 周库存余额 | — | NC 提供查询：`GET /nc/reconcile/inventory-balance?week=yyyy-Wnn` |
| CHK-003 月末全量对账 | — | NC 提供查询：`GET /nc/reconcile/month?period=yyyy-MM&org=___` |
| CHK-004 状态查询 | — | NC 提供查询：`GET /nc/voucher/query?task_id=___` 或 `?nc_voucher_no=___` 或 `?source_bill_no=___` |

**回函必须明确：**
- F-03 回执选 A 还是 B；选 A 时物资侧暴露的 webhook 路径会另发；选 B 时物资侧的轮询频率（建议 30 秒一次）
- 对账接口的返回字段：笔数、金额、凭证号清单、状态分布
- CHK-005 映射完整性属于物资内部接口，**不需要 NC 实现**

### 3.4 期间 / 反结（F-10 + F-11，共 2 个，P1）

| 接口 | NC 必须实现 | 期望路径 | 优先级 |
|------|----------|---------|--------|
| F-10 月结状态查询 | NC 当前期间状态（未结/已结/已反结）+ 上次结账时间 | `GET /nc/period/{yyyy-MM}/status` | P1 |
| F-11 反结申请 | 是否支持接口提交反结 / 还是仅 NC 系统内人工操作 | `POST /nc/period/{yyyy-MM}/reverse` | P2 |

**回函必须明确：** NC 端结账锁定多严？已结账期间能否补推？反结后允许补推的窗口期？

### 3.5 账务明细查询能力（应收 / 应付等，只读，P1）

> **定位：** 阜矿集团相关业务系统不直接读取 NC 数据库，也不要求 NC 暴露内部表结构；请 NC 团队提供正式的**账务明细只读查询接口**或接口平台能力，用于业务对账、付款 / 收款状态核验、凭证穿透和业务闭环查询。

请 NC 团队确认是否可提供以下查询能力：

| 查询能力 | 用途 | 建议路径 / 形态 | 优先级 |
|---------|------|----------------|--------|
| 应付明细账查询 | 查询供应商往来、应付发生、付款核销、余额，用于合同付款、三单匹配和对账 | `GET /nc/ledger/payables?org=___&period=___&supplier=___` | **P1** |
| 应收明细账查询 | 查询对厂矿 / 内部单位销售出库形成的应收、收款核销、余额 | `GET /nc/ledger/receivables?org=___&period=___&customer=___` | P1 |
| 凭证明细 / 分录查询 | 按 NC 凭证号、来源单号、期间查询凭证明细行 | `GET /nc/voucher/{nc_voucher_no}/lines` 或 `GET /nc/voucher-lines?source_bill_no=___` | **P1** |
| 往来余额查询 | 按供应商 / 客户 / 核算组织 / 期间查询期初、发生、核销、期末余额 | `GET /nc/ledger/balance?counterparty=___&period=___` | P1 |
| 核销 / 结算状态查询 | 查询某笔应付 / 应收是否已付款、已收款、已核销、剩余余额 | `GET /nc/settlement/status?source_bill_no=___` | P1 |

**建议查询条件：**

- 核算组织 / 账套：`accounting_org_code` / `book_code`
- 期间 / 日期范围：`period`、`date_from`、`date_to`
- 往来单位：`supplier_code` / `customer_code` / `counterparty_code`
- 来源信息：`source_system`、`source_bill_no`、`business_scene_code`
- 凭证信息：`nc_voucher_no`、`subject_code`
- 分页：`page_size`、`page_token` 或 `page_index`

**建议返回字段：**

- 基础字段：NC 明细 ID、凭证号、凭证日期、期间、摘要、科目编码 / 名称
- 往来字段：供应商 / 客户编码与名称、合同号、来源单号
- 金额字段：借方金额、贷方金额、发生额、已核销金额、未核销余额、期末余额
- 状态字段：记账状态、核销状态、付款 / 收款状态、红冲 / 冲销标识
- 追溯字段：来源系统、来源业务场景、来源单据号、NC 任务号 / 回执号

**权限与安全要求：**

- 只读查询，不写 NC 账务数据；
- 按账套 / 核算组织 / 期间进行权限控制；
- 支持分页，避免一次性拉取大批明细；
- 明确查询频率、QPS、最大时间跨度和日志保留期；
- 如应收 / 应付模块在 NC 中属于不同子系统，请分别说明接口路径、字段差异和负责人。

## 六、错误码与异常处理（P0）

### 6.1 错误码分层

NC 必须按以下三类返回错误码（详设 08 §F-08 `exception_record` 分类对齐）：

| 类型 | 含义 | 物资侧处理 | NC 必须给出码表 |
|------|------|----------|--------------|
| 业务校验失败 | 科目不存在、组织未启用、期间已封账等 | 写 F-08 异常台账，不自动重推；待人工处理 | ___ |
| 系统暂时不可用 | NC 实例宕机、网关超时、限流 | 物资按 §4.3 退避自动重推 | ___ |
| 数据冲突 | 幂等键已存在但报文不同 | 阻断（详设 08 §十一 `NC_IDEMPOTENT_CONFLICT_BLOCK=true`） | ___ |

### 6.2 退回 / 冲销

| 场景 | NC 端流程 | 物资侧需要的信息 |
|------|---------|---------------|
| NC 端凭证退回 | NC 通过 F-03 回执通知 `finance_state=已退回` | 退回原因（自由文本 + 错误码）+ NC 建议处理方式（红冲 / 修正 / 关闭） |
| NC 端冲销凭证 | NC 自行冲销后，物资侧如何感知？ | 是否通过 F-03 推送冲销事件？还是只能物资侧拉 CHK-001/CHK-004？ |

### 6.3 重复推送

- 物资侧因网络抖动可能对同一业务单据推送多次。NC 端**必须**按 `idempotent_key`（详设 08 §7.2 规则）去重，重复推送返回首次的凭证号 + 标记 `idempotent=true`。
