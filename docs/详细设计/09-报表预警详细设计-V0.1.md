## 一、概览

### 1.1 报表 vs 预警的边界

| 维度 | 报表 | 预警 |
| --- | --- | --- |
| 触发方式 | 用户主动查询 / 导出 | 系统被动扫描 / 业务事件触发 |
| 数据形态 | 多维聚合 / 明细列表 | 单条事件记录 + 状态机 |
| 存储载体 | 视图 / 物化查询（V1.2 §五）| `report_alert` 表（本版 §3） |
| 输出渠道 | 列表 / Excel / PDF / Print | 站内消息 / 待办 / 看板 / 邮件（未来）|
| 时效要求 | 用户查询时即查 | 每日 / 每小时调度 + 业务事件实时触发 |
| 闭环责任人 | 报表口径所有方 | 预警接收人 |

**本版重点：** 预警侧的"通用预警表 + 触发规则 + 调度框架"。报表侧已由 V1.2 §五"一期 21 张报表清单 + 字段口径"覆盖，本版不重写。

### 1.2 通用预警表 vs 业务领域表

V1.2 的 30+ 业务预警规则按"业务条线"在 ALR-XXX-NNN 编号中区分（库存/采购/合同/付款/设备/财务接口/合规等），但**落地存储统一在一张 `report_alert` 表**中。理由：

1. **状态机统一**：3 状态机（待处理 → 已处理 / 已忽略）所有预警通用，无需为每类业务建独立 state machine
2. **接收 / 处理 / 通知 UI 统一**：一个"我的预警"列表、一个"预警处理"页就能覆盖所有业务
3. **去重 / 抑制 / 升级规则统一**：在通用表上加索引和 last_alert_time 即可，避免每业务表重复实现
4. **简化 Sprint 7a 起步成本**：Sprint 7a 决策点 4B "仅 R-04" 落地 ~1 张表 + 1 个 Detector + 14 单测，工时 1.5 PD 内
5. **AI Tool 接入简化**：V1.2 §7.7 AI-006 预警 Tool 直接查 `report_alert`，无需多表 union

但**业务领域表**仍保留各自的「业务字段」（如 C-07 PaymentPlan 的 due_date、C-02 Contract 的 bond_release_date、M-15 BatchLot 的 expire_date），预警检测时由 Detector 服务通过 SourceBillType + SourceBillId 反查业务详情。

### 3.2 状态机

```
                    MarkHandled(handler, result)
                    ────────────────────────────►   已处理（终态）
待处理
（Pending）         MarkIgnored(handler, reason)
                    ────────────────────────────►   已忽略（终态）
```

**状态值（与代码 `ReportAlertStates` 常量对齐）：**

- `Pending = "待处理"`（初始状态，由 Detector 创建时设置）
- `Handled = "已处理"`（终态，业务部门确认处理完毕）
- `Ignored = "已忽略"`（终态，运维 / 业务判定无需处理）

**转换约束（`EnsureState` 守护）：**

- 仅 `待处理` 可 MarkHandled / MarkIgnored
- 终态不可二次转换；如需重开必须新建一条 ReportAlert
- MarkHandled 必传 handler_user_id + handle_result（业务规则强约束）
- MarkIgnored 必传 handler_user_id + reason

### 3.3 AlertContent JSON 结构约定

JSON 字符串承载业务上下文，按 AlertCode 不同结构不同。R-04 PaymentDueNear 示例（已落地）：

```json
{
  "planId": 123,
  "contractId": 45,
  "paymentNodeId": 67,
  "planAmount": 100000.00,
  "cumulativePaidAmount": 50000.00,
  "remainingAmount": 50000.00,
  "planState": "部分付款",
  "dueDate": "2026-05-20",
  "daysToDue": 3
}
```

**约定：**

- 字段名 `camelCase`（与 .NET 默认 JsonSerializer 一致）
- 金额字段 `decimal` → JSON `number`
- 日期字段 `DateOnly` → JSON `"YYYY-MM-DD"` 字符串
- 字段集合在 §4 各 R-XX 触发规则节给出（本版仅 R-04 详列；R-05~R-08 在 Sprint 8a 落地时按相同模式）

## 六、业务规则

### 6.1 去重 / 抑制

- **当前实施（Sprint 7a）：** 不去重，每次扫描全量新增（接受重复噪声）
- **Sprint 8a 接入：** 同一 (alert_code, source_bill_type, source_bill_id) 在 `INVENTORY_LOW_STOCK_DEDUP_HOURS`（默认 12 小时）窗口内仅生成 1 条
- **去重判定逻辑（Detector 实施）：**
  ```
  IF EXISTS (
    SELECT 1 FROM report_alert
    WHERE alert_code = @code
      AND source_bill_type = @type
      AND source_bill_id = @id
      AND alert_time > NOW() - INTERVAL '@dedupHours hours'
      AND alert_state = '待处理'
  ) THEN SKIP
  ELSE INSERT
  ```

### 6.2 升级机制（Sprint 8a+）

- 同一 source_bill 持续命中超过 `ALERT_ESCALATE_DAYS`（默认 7 天）且未处理 → 自动升级 alert_level（低→中→高）
- 升级时新增一条 alert（不修改原条），原条状态保持"待处理"
- 升级在 Hangfire 调度时统一处理

### 6.3 抑制周期

- 已 `MarkIgnored` 的预警：同一 source_bill 7 天内不再触发（实施在 Detector 查询时排除最近 ignored）
- 已 `MarkHandled` 的预警：业务问题已闭环，下次扫描如仍命中则再触发（不抑制 — 视为新问题）

### 6.4 多组织隔离

- AlertCount 查询、看板统计严格按 `org_id` + 数据权限过滤（沿用 A-06 data_permission 框架）
- SubGroupId 数据隔离守护单测自动覆盖（Sprint 1 D6 双轨钩子 + Sprint 7a Day 7 已加 ReportAlert）
