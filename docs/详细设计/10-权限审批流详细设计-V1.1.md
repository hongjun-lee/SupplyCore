# 权限审批流详细设计（V1.1）

**版本：** V1.1  
**日期：** 2026-05-05  
**文档性质：** 详细设计层 · 模块详设第十篇  
**适用阶段：** 详细设计执行、开发实施、实施配置

---

## 一、文档目的

本文档承接 `04-权限审批与审计概要设计-v0.1.md` 中的权限模型边界，以及 `01-数据库逻辑模型-V1.0.md` 中 A-01~A-20 骨架，把用户副本同步、业务角色定义、功能权限树、数据权限规则、审批流程模板、审批实例运行态、高敏感操作清单和日志字段固化到可开发、可配置、可联调的详设层。

本文档重点回答：

- A-01~A-20 各实体的完整字段、约束和索引建议
- 11 个一期业务角色的权限边界和互斥规则
- 各业务单据的审批流程模板（节点类型、路由条件、互斥规则）
- 高敏感操作清单（A-11）的完整列举和控制要求
- 日志实体的字段规范、保留策略和查询要求

## 三、实体关系（ERD 概要）

```
A-01 user_copy ←── Nova Platform 同步
  └─ A-03 user_role (N) → A-02 business_role
  └─ A-12 user_permission_log (N)
  └─ A-13 operation_log (N)
  └─ A-14 approval_log (N)
  └─ A-15 interface_operation_log (N)
  └─ A-16 ai_call_log (N)
  └─ A-17 data_import_log (N)
  └─ A-18 data_export_log (N)

A-02 business_role
  └─ A-05 role_function_permission (N) → A-04 function_permission
  └─ A-06 data_permission (N)
  └─ A-07 data_permission_rule (N)

A-08 approval_workflow → A-09 approval_node (N) → A-02
  └─ A-20 approval_instance (N)
       └─ A-10 approval_opinion (N)
       └─ A-14 approval_log (N)

A-11 sensitive_operation （独立维护的高敏感操作枚举表）
A-17 data_import_log (N) → A-19 data_import_failure_row
```

### 4.9 A-09 approval_node 审批节点

> **实施落点说明（V1.1）：** 本表 11 字段为**逻辑字段**，物理落到 Catio `Nova.Workflow.ProcessDefinitionVersion.StepsConfig` 的 V2 图结构 DSL（`WorkflowGraphDto: { nodes, edges, completionActions }`）。详细映射见下表 + 10A V0.6 节 5A.4。

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| node_id | bigint | PK | |
| workflow_id | bigint | NOT NULL FK→A-08 | |
| node_sequence | int | NOT NULL | 节点顺序 |
| node_type | varchar(20) | NOT NULL | 发起/初审/复核/会签/终审 |
| node_name | varchar(100) | NOT NULL | |
| approver_type | varchar(20) | NOT NULL | role/user/org_manager |
| approver_role_id | bigint | FK→A-02 | approver_type = role 时必填 |
| approver_org_id | bigint | FK→M-01 | 审批组织范围（空=发起人组织） |
| approve_condition | varchar(500) | | 进入本节点的条件（空=无条件） |
| is_parallel | boolean | NOT NULL DEFAULT false | 会签节点是否并行 |
| reject_goto_node | int | | 驳回时跳转到哪个 node_sequence；空=驳回到发起人 |
| node_status | varchar(20) | NOT NULL DEFAULT '启用' | |

**唯一约束：** `(workflow_id, node_sequence)` 复合唯一。

**逻辑字段 → Catio V2 DSL 落点映射（V1.1）：**

| 详设 10 字段 | Catio V2 DSL 落点 | 实施备注 |
| --- | --- | --- |
| node_sequence | 由 edges 拓扑推导 | 不落字段；如需线性序号在适配层做拓扑排序后编号 |
| node_type | `WorkflowNodeDto.type`（START/APPROVAL/CONDITION/FORK/JOIN/END/...）+ `approvalConfig.name`（业务名） | 详设 5 种业务值（发起/初审/复核/会签/终审）映射到技术节点 + 业务名标识 |
| node_name | `approvalConfig.name` | |
| approver_type | `ApproverRule.RuleType`（独立表） | 节点引用 `approvalConfig.approverRuleCode` |
| approver_role_id | `ApproverRule.Config jsonb` | 同上 |
| approver_org_id | `ApproverRule.Config jsonb` | 同上 |
| approve_condition | 独立 `CONDITION` 节点的 `conditionConfig.expression` | 高级表达式如 `amount * rate > 10000 && department == "FIN"` |
| is_parallel | `approvalConfig.signMode = COUNTERSIGN`（会签）/ `OR_SIGN`（或签）/ `SINGLE` | |
| reject_goto_node | `approvalConfig.rejectBehavior` + `allowedReturnStepKeys` | END / RETURN_TO_INITIATOR / RETURN_TO_STEP |

### 5.2 角色互斥规则

以下角色组合不允许同一用户同时拥有（同一业务单据的发起/审批/执行不应由同一人闭环）：

| 互斥场景 | 规则 |
| --- | --- |
| 仓库管理员 + 财务人员 | 不应由同一人既做仓库操作又做接口财务审核 |
| 采购人员 + 财务人员 | 付款申请发起人与财务审核人应分离 |
| 使用单位申请人 + 仓库管理员 | 同一人不应既发起领料申请又执行出库 |
| 仓库管理员 + 盘亏/报废高敏感终审 | 仓库管理员不应拥有盘亏/报废终审权 |

实现方式：A-03 授权时检查 A-02.role_category 的互斥配置；系统管理员授权高敏感组合须触发二次确认和 A-12 日志。

## 十三、一句话结论

10 详设的核心是把"谁能看、谁能做、谁来审、做了留痕"四件事用统一的角色—权限—数据范围—审批流—日志链串联起来：角色定义可配置、权限树可分层、审批节点可路由、高敏感操作必须强制留痕，使权限管控在开发阶段可实现、在实施阶段可配置、在运维阶段可审计。
