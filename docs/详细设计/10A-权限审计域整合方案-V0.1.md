# 10A 权限审计域整合方案 V0.1

## 一、文档目的

详设 10《权限审批流详细设计 V1.0》定义了 A-01~A-20 共 20 张权限/审批/审计表的**字段规范**，本文档定义其**实施层选择**：哪些表直接复用 Catio 已有模块（`Nova.Platform` 提供权限/角色/数据范围；`Nova.Workflow` 提供审批流；ABP Framework 提供身份/权限授予/审计日志），哪些表由 SupplyCores 自建。目标是**在不偏离详设 10 业务模型的前提下，把代码工作量从 20 张表压缩到 7 张表**，并明确两个项目的边界。

---

## 二、与详设 10 的关系

| 维度 | 详设 10 | 本文档（10A） |
| --- | --- | --- |
| 范围 | 字段、约束、索引、业务规则 | 实施载体（复用 vs 自建）、Phase 排期、对接缺口 |
| 是否新增/修改 A-01~A-20 设计 | 否 | 否 |
| 服务对象 | 业务方、所有同事 | 开发同事 + 项目领导 |
| 升级触发 | 业务模型调整 | Catio 模块结构变化 / Phase 推进 |

**强约束**：本文档不修改详设 10 的任何字段定义；如发现 Catio 等价物字段不全，需提案修改详设 10 后再整合，不允许"按 Catio 现状回切详设"。

---

## 三、整合策略原则

1. **能复用就不重建**：Catio 的 `Nova.Platform` + `Nova.Workflow` 在 Catio 主仓库长期演进，业务模块只 `[DependsOn]` 即可享受 SSO/数据范围/审批流，避免双份维护。
2. **业务专属才自建**：高敏感操作清单、NC 接口日志、AI 调用日志、数据进出口等是 SupplyCores 业务特征，不应进 Catio 平台。
3. **Schema 严格遵守 CLAUDE.md**：自建表全部落 schema `a`（auth/audit）；不与 Catio Platform 的 schema 冲突（Catio 框架表用默认 schema，业务表无 `a` 占用）。
4. **Phase 隔离风险**：Phase 2（自建 7 张表 + Seed）零外部依赖，先行；Phase 1（接 SSO + 挂 `[Authorize]`）需要 Catio AuthServer 配置，待业务方/运维确认后再做。

## 四、A-01~A-20 实施层映射表

| 编号 | 详设 10 表名 | 实施载体 | 决策 | 备注 |
| --- | --- | --- | --- | --- |
| A-01 | user_copy | `AbpUsers` (ABP Identity) + `Nova.Platform.UserOrganization` | **REUSE** | Nova 用户经 OAuth/OIDC 登录后落入 AbpUsers；不需要单独复制表 |

## 五、SupplyCores 自建 7 张表（schema `a`）

字段定义遵循详设 10 节 4.11 / 4.13 / 4.15~4.19，本文档不重述；以下仅列实施侧补充：

| 表名 | EF Core 实体 | 基类 | 说明 |
| --- | --- | --- | --- |
| `a.sensitive_operation` | `SensitiveOperation` | `FullAuditedAggregateRoot<long>` | 由 `SensitiveOperationDataSeedContributor` 灌入 19 条种子 |
| `a.operation_log` | `OperationLog` | `CreationAuditedAggregateRoot<long>` | append-only，无 LastModification/Deletion 字段 |
| `a.interface_operation_log` | `InterfaceOperationLog` | `CreationAuditedAggregateRoot<long>` | 同上 |
| `a.ai_call_log` | `AiCallLog` | `CreationAuditedAggregateRoot<long>` | 同上；input_params 须脱敏后写入 |
| `a.data_import_log` | `DataImportLog` | `CreationAuditedAggregateRoot<long>` | 同上 |
| `a.data_import_failure_row` | `DataImportFailureRow` | `Entity<long>` | 子明细，不带审计字段（追溯靠父表） |
| `a.data_export_log` | `DataExportLog` | `CreationAuditedAggregateRoot<long>` | 同上；含 watermark_content 字段 |

**JSONB 字段约定**：详设 10 中 `old_value / new_value / operation_details / input_params / output_summary / export_fields / raw_row_data / scope_value / approval_chain_snapshot` 等 jsonb 字段，EF Core 层用 `string?` + `HasColumnType("jsonb")` 映射，序列化/反序列化由应用代码负责。

**外键策略**：跨域引用全部走"软外键"（`long?` + 索引，无 DB 级 FOREIGN KEY 约束），与现有 Wave 1~5 一致。F-14 / F-01 / R-06 等未来落地的实体不在本期建立强约束。

## 六、Phase 排期

### Phase 2 — 自建 7 张审计表（先行，零外部依赖）

**预计 1 周**：
1. `Nova.SupplyCores.Domain/Auditing/` 落 7 个 Entity 类
2. `SupplyCoresDbContext` + `SupplyCoresDbContextModelCreatingExtensions` 配置（schema `a`）
3. EF Core migration 生成 + DbMigrator 应用
4. `SensitiveOperationDataSeedContributor` 灌 19 条高敏感操作
5. 提供 `IOperationAuditService` / `IInterfaceAuditService` / `IAiAuditService` / `IDataIoAuditService` 4 个接口（业务代码注入后调用，不在基类自动写入，避免与 Phase 1 的 `[Authorize]` 拦截器耦合）

### Phase 1 — 接 Catio 权限栈（依赖业务方/运维确认）

**预计 1 周（不含等待外部确认时间）**：
1. `SupplyCoresWebModule` 增加 `[DependsOn]`：`AbpAccountWebModule` + `AbpIdentityWebModule` + `AbpOpenIddictAspNetCoreModule` + `AbpPermissionManagementWebModule` + `NovaWorkflowApplicationModule` + `NovaWorkflowEntityFrameworkCoreModule`
2. `appsettings.json` 配置 OpenIddict（client mode 连 Catio AuthServer，或 server mode 自建）
3. `SupplyCoresPermissionDefinitionProvider` 补 11 个业务角色 + 功能权限树（详设 5.1 / 5.3）
4. `SupplyCoresDbMigrator` 增加角色种子（系统预置 11 角色，is_system=true）
5. 全部已落地 AppService（Unit / MaterialCategory / UnitConversion / Warehouse / WarehouseZone / StorageLocation）的方法挂 `[Authorize(SupplyCoresPermissions.X.Y)]`
6. 在 `SupplyCoresAppService` / `SupplyCoresCrudAppService` 基类植入"高敏感操作拦截 + operation_log 自动写入"（Phase 2 的接口在此被调用）

## 八、与详设 10 字段的差异说明

本文档**不修改任何字段定义**，但实施层有以下"等价转换"需在评审时显式确认：

| 详设 10 字段 | 本文档实施 | 差异说明 |
| --- | --- | --- |
| A-01 `user_id bigint PK` | `AbpUsers.Id` (Guid) | ABP Identity 用 Guid 作为 user PK；A-13/A-15/A-16/A-18 中所有 `user_id FK→A-01` 字段统一用 Guid |
| A-13 `created_at timestamptz` | EF Core `CreationTime` (DateTime) | ABP 基类自带，时区行为由 PostgreSQL `timestamptz` 保证 |
| A-13 `user_id` 与 ABP `CreatorId` | 都保留，前者是业务快照 | `user_id` 与 `CreatorId` 在正常场景一致；保留 `user_id` 列以便用户记录被删除后仍能追溯快照 |
| A-09 独立表 → JSON 节点 | Catio Workflow 把节点配置内嵌到 ProcessDefinitionVersion 的 JSON 字段中 | 业务读节点信息走 Catio API，不直接 SQL 查 A-09；查询能力等价 |

## 十、一句话结论

**详设 10 是"是什么"，本文档是"怎么落"——把 20 张表的代码工作量压到 7 张，靠的不是减少业务能力，而是把 Catio 已有的 `Nova.Platform` + `Nova.Workflow` 当作平台基座完整复用；自建的 7 张全是 SupplyCores 业务专属审计能力，复用与自建的边界由 Phase 2 / Phase 1 排期严格隔离风险。**
