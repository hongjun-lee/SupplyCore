# 10A 权限审计域整合方案 V0.3

## 一、文档目的

详设 10《权限审批流详细设计 V1.0》定义了 A-01~A-20 共 20 张权限/审批/审计表的**字段规范**，本文档定义其**实施层选择**：哪些表通过引用 Catio 仓库的 ABP 模块在本地库实现，哪些表由 SupplyCores 自建，并显式列出二者之间所需的"适配层"工作量。目标是**在不偏离详设 10 业务模型的前提下，把"自建表"的工作量压缩到 8 张**，同时把适配工作（ID 桥接、索引扩展、审批链快照映射、DataScope 拆口径、Properties 扩展、日志不可篡改防护等）作为可量化的开发 backlog 一并列出，而非用"几乎重合"等模糊措辞掩盖。

## 三、术语澄清

V0.1 评审中识别到 "Catio" 与 "Nova Platform" 在本文档语境内并非同一概念。V0.2 显式拆分如下三层：

| 名称 | 性质 | 在本系统的角色 |
| --- | --- | --- |
| **集团 Nova Platform**（业务/运维语境） | 集团已部署的统一身份系统（**Nova 项目** 的运行实例） | SSO 权威源、组织/人员主数据来源；SupplyCores 通过 OAuth/OIDC 与之集成（概设 01 节 6.1） |
| **Catio 仓库目录** | Nova 项目的源码目录（"Catio 是 Nova Platform 系统的目录"） | 包含 `Nova.Platform`、`Nova.Workflow` 等可复用 ABP 模块；SupplyCores 引用其代码 |
| **`Nova.Platform` / `Nova.Workflow` 模块** | Catio 仓库下的 ABP 模块命名空间（**不是特指集团 Nova Platform 系统**） | SupplyCores 通过 `[DependsOn]` 引用其代码、在**本地部署的独立数据库**中建相关表（AbpUsers / RoleOrganization / DataScopeGrant / ApprovalRecord / ProcessInstance 等） |

**与概设 01 部署约束的一致性**：
- 概设 01 第 28、150、208 行约束："物资系统应独立部署、不与 Nova Platform 共库、不直接访问 Nova 底层数据库"
- 本文档 "REUSE" 含义：**复用 Catio 仓库的 ABP 模块代码**，在本地库建相同结构的表；**不是访问集团 Nova Platform 的运行库**
- 业务/运维层与集团 Nova Platform 通过 OAuth/OIDC + 接口调用集成；代码层与 Catio 通过 `[DependsOn]` 复用；部署层 SupplyCores 独立库

## 四、整合策略原则

1. **能复用就不重建**：Catio 仓库的 `Nova.Platform` + `Nova.Workflow` 在 Nova 项目长期演进，业务模块只 `[DependsOn]` 即可享受 SSO/数据范围/审批流，避免双份维护。
2. **业务专属才自建**：高敏感操作清单、NC 接口日志、AI 调用日志、数据进出口等是 SupplyCores 业务特征，不进 Catio。
3. **复用即承认适配成本**：每张 REUSE 表都要登记"字段缺口"和所需"适配层"工作（见节五缺口列、节六适配层清单），不依赖"几乎重合"等模糊措辞。
4. **详设修订按字段性质分层**：L1 业务模型（实体存在、字段语义、状态值域、业务规则、引用关系）不可由 10A 单方面改；**L2 技术实施（PK 类型、ABP 基类、jsonb 映射、索引细节、长度/精度）允许 10A 提案修订详设**，走"提案 → 详设作者评估 → 详设修订 → 10A 合一"流程；L3 命名/格式实施层自决。详细分类与 L2 提案登记见节十一。
5. **Schema 严格遵守 CLAUDE.md**：自建表全部落 schema `a`（auth/audit）；Catio 模块表落默认 schema（`public`，由 ABP 模块决定）。
6. **阶段隔离风险**：阶段 A（自建表 + 接口契约）零外部依赖，先行；阶段 B（接 SSO + Workflow 引擎）需要业务方/运维确认；阶段 C（高敏感拦截 + 自动留痕）依赖 A 和 B 都完成后做。

### 6.3 Schema 与多租户

**Schema 落点：**

| 模块来源 | Schema | 说明 |
| --- | --- | --- |
| Catio `Nova.Platform` / `Nova.Workflow` 模块表 | 默认（`public`） | AbpUsers / AbpRoles / RoleOrganization / DataScopeGrant / ProcessDefinition / ProcessInstance / ApprovalRecord 等 |
| ABP 框架表（`__EFMigrationsHistory` 等） | 默认（`public`） | — |
| SupplyCores 业务表（Wave 1~5） | 各业务 schema（详设 02/03 等定义） | — |
| **SupplyCores 审计自建表（8 张）** | **`a`** | DbContext 中 `b.ToTable("...", "a")` 显式指定；迁移脚本须先 `CREATE SCHEMA IF NOT EXISTS a` |

**多租户：** 一期不启用（与 [01-数据库逻辑模型-V1.0.md:175](docs/详细设计/01-数据库逻辑模型-V1.0.md#L175) "tenant_id 一期不启用但预留" 对齐）。
- 8 张自建实体不实现 `IMultiTenant`
- Catio 模块如默认带 `TenantId` 列，配置 `MultiTenancyConsts.IsEnabled = false`，DataFilter 关闭后字段保留但不生效
- 未来开多租户时，需评估"REUSE 表带 TenantId、自建表预留 tenant_id 列"的迁移路径，作为长期 backlog（不在一期）

### 6.4 日志不可篡改三层防护

详设 10 节 8.2 要求"日志表禁止 DELETE 和 UPDATE"。`CreationAuditedAggregateRoot` 仅在 ORM 层去掉了修改字段，并不强制 DB 层不可篡改。V0.2 落实三层防护：

| 层级 | 措施 | 适用范围 |
| --- | --- | --- |
| 仓储层 | 6 张日志表的 `IRepository<T>` 改为只暴露 `InsertAsync` / `GetAsync` / `GetListAsync`，不暴露 `UpdateAsync` / `DeleteAsync`（编译期堵住） | 6 张 append-only 日志表 |
| DB 应用账号权限 | PostgreSQL `GRANT INSERT, SELECT ON a.<日志表> TO supplycores_app`（不授 UPDATE/DELETE/TRUNCATE） | 6 张 append-only 日志表 |
| DB 兜底 | 对 6 张日志表加 `BEFORE UPDATE OR DELETE` 触发器抛错；DBA 维护账号走独立 PostgreSQL role，操作时须 A-12 留痕 + 二次留痕 | 6 张 append-only 日志表 |

**例外**：`a.business_data_scope_grant` 和 `a.sensitive_operation` 是维度表（可维护），不挂三层防护，按 ABP 标准 `FullAuditedAggregateRoot` 行为处理。

**回写详设 10**：建议在详设 10 V1.1 节 8.2 加一句"具体落地见 10A 节 6.4"，避免详设说一套、实施另一套。

## 七、A-10/A-14 查询路径

A-10（按节点视角的审批意见）和 A-14（按链路视角的审批日志）双映射到 Catio `ApprovalRecord` 单表后，前端两个查询入口的 SQL 路径如下：

### 7.1 主路径（业务单据视角查询审批记录）

```sql
SELECT ar.*
FROM workflow.approval_record ar
JOIN workflow.process_instance pi ON ar.instance_id = pi.id
WHERE pi.business_type = :business_type
  AND pi.business_id   = :business_id
ORDER BY ar.created_at;
```

依赖 `process_instance.(business_type, business_id)` 二级索引（适配层 #2）。

### 7.2 性能/审计报表兜底

如审计报表 SLA 要求高（详设 11 节 5 要求报表 P95 ≤ 5s），主路径 JOIN 在审批量大时可能不达标。备选方案：

| 方案 | 实施 | 触发条件 |
| --- | --- | --- |
| 物化视图 | `CREATE MATERIALIZED VIEW a.approval_record_view AS SELECT ar.*, pi.business_type, pi.business_id, pi.business_bill_no FROM workflow.approval_record ar JOIN workflow.process_instance pi ...`，定时 REFRESH | 审计报表查询频度 > 100 次/天且 P95 不达标 |
| 投影表 | 自建 `a.approval_record_projection`，通过 EventBus 订阅 ApprovalRecord 写入事件同步 | 物化视图刷新延迟不可接受时 |

**当前不预建**，待审计报表 SLA 验收时根据实测决定（占位入节十）。

## 八、4 个审计接口契约

阶段 A 落地审计接口的方法签名（DTO 字段与详设 10 一一对应）：

```csharp
public interface IOperationAuditService
{
    Task LogAsync(OperationLogInput input);
    // input 字段对应 A-13：user_id/org_id/operation_type/business_bill_type/business_bill_id/
    //                     business_bill_no/old_value/new_value/change_reason/ip_address
}

public interface IInterfaceAuditService
{
    Task LogAsync(InterfaceOperationLogInput input);
    // input 字段对应 A-15：interface_id/task_id/operation_person_id/operation_type/
    //                     operation_details/operation_reason
}

public interface IAiAuditService
{
    Task LogAsync(AiCallLogInput input);
    // input 字段对应 A-16：ai_tool_id/caller_user_id/caller_org_id/tool_name/
    //                     input_params(已脱敏)/output_summary/data_range/
    //                     call_duration_ms/call_status/error_code
}

public interface IDataIoAuditService
{
    Task<long> LogImportAsync(DataImportLogInput input);    // 返回 import_log_id 供失败行写入
    Task LogImportFailureRowsAsync(long importLogId, IEnumerable<DataImportFailureRowInput> rows);
    Task LogExportAsync(DataExportLogInput input);
}
```

**脱敏责任划分**：调用方负责脱敏（业务上下文最清楚什么字段敏感），审计服务**不做二次脱敏**——但服务内部用 ABP `IDataFilter` 拦截已知敏感字段（如 `password`/`token`/`secret`/`id_card_no`），双重保险。

## 十一、与详设 10 字段的分层处理 + L2 提案登记

V0.3 引入 L1/L2/L3 分层（节四原则 #4）：**L1 业务模型不可由 10A 单方面改**；**L2 技术实施允许 10A 提案修订详设**；**L3 命名/格式实施层自决**。本节按此框架重组：先列分类原则（11.1），再列 L1 一致项（11.2，实施层等价转换），最后列 L2 提案待决项（11.3）。

### 11.1 L1/L2/L3 分类框架

| 层级 | 字段性质 | 例子 | 谁说了算 | 10A 能否提案修订 |
| --- | --- | --- | --- | --- |
| **L1** | 业务模型 | 实体的存在、字段语义、状态值域（如 `instance_state` 6 种值）、业务规则、引用关系（A-13 必须 FK→A-01） | 业务方 + 详设作者 | 不能单方面改，必须走业务评审 |
| **L2** | 技术实施 | **PK 类型**、ABP 基类、jsonb 映射、索引细节、长度/精度 | 详设作者 + 开发联合 | **允许 10A 提案修订**；走"提案 → 评估 → 详设修订 → 10A 合一"流程 |
| **L3** | 命名/格式 | 字段命名风格、字段顺序、注释格式 | 开发自决 | 不需要详设修订 |

### 11.2 L1 一致项（实施层等价转换，不修订详设）

| 详设 10 字段 | 本文档实施 | 说明 |
| --- | --- | --- |
| A-13 `created_at timestamptz` | EF Core `CreationTime` (DateTime) | ABP 基类自带，时区行为由 PostgreSQL `timestamptz` 保证 |
| A-13 `user_id` 与 ABP `CreatorId` 并存 | 都保留，前者是业务快照 | 在正常场景指向同一个人；保留业务快照列以便用户记录被删除后仍能追溯 |
| A-09 独立表 → JSON 节点 | Catio Workflow 把节点配置内嵌到 ProcessDefinitionVersion 的 JSON 字段 | 业务读节点信息走 Catio API；如有"按 approver_role_id 反查模板"查询，建投影表（节七） |
| A-06 `data_type` 5 种 | 拆口径：`org` 复用 Catio DataScopeGrant；`warehouse / material_category / supplier / contract` 走自建 `a.business_data_scope_grant` | `IDataScopeQueryPlanner` 实现路由 |
| 详设 10 所有 `jsonb` 字段 | EF Core 用 `string?` + `HasColumnType("jsonb")` 映射 | 序列化/反序列化由应用代码负责 |

### 11.3 L2 提案待决项

#### L2-提案-01：A-01 user_id PK 类型 — bigint vs Guid

**背景：** 详设 10 节 4.1 定义 A-01.user_id 为 `bigint PK`；ABP/Catio 默认 user 用 Guid。V0.2 选了"保留 bigint + 双层身份桥接"的稳妥路径，但承担了永久桥接成本。V0.3 重新打开评估。

**两条岔路对比：**

| 维度 | 岔路 1：保留 bigint（V0.2 现状） | 岔路 2：详设 10 V1.1 改 Guid |
| --- | --- | --- |
| 与 ABP/Catio 一致 | ✗ 永久维护双层身份（A-01 + AbpUsers） | ✓ AbpUsers 直接当 user_copy，少一套同步逻辑 |
| 风险 #7（双层同步延迟） | ✗ 永久风险 | ✓ 风险消除 |
| 适配层 #1（Guid↔long 桥接） | ✗ 永久存在，所有 AppService 入口转换 | ✓ 适配层 #1 消除 |
| 分布式/多机房 | ✗ 自增 ID 跨库冲突；改雪花 ID 又要再桥接 | ✓ 应用层生成，天然友好 |
| API 安全性 | ✗ ID 可枚举（`/api/users/123` 能扫）| ✓ 不可枚举 |
| 占空间 / JOIN 性能 | ✓ 8 字节 / 略快（PG btree 差距 <5%） | ✗ 16 字节 / 略慢（业务体量下可忽略） |
| SQL 可读性 | ✓ `WHERE user_id = 1024` | ✗ `WHERE user_id = '550e8400-...'` |
| 一次性修订成本 | — | ✗ 详设 10 V1.1 改 A-01 + 全部 `user_id FK→A-01` 字段；DB migration 重做；预估 0.5 周 |
| 长期成本 | ✗ 桥接 + 风险 #7 永久 | ✓ 一次修订后无永久负担 |

**当前状态：** 待决。在详设 10 V1.1 出台前，**10A 仍按"岔路 1（保留 bigint）"实施**——节五映射表、节六实体、节十二风险 #7 均按此版本；A-13/A-15/A-16/A-18 等所有 `user_id bigint FK→A-01` 字段保持 bigint。

**决策方：** 详设作者 + 业务方 + Catio/集团 Nova 团队联合。

**决策窗口：** 阶段 A 启动前（最迟阶段 A 第 1 周末）；超过此窗口默认锁定岔路 1，事后改动成本递增。

**若决定走岔路 2：** 详设 10 V1.1 + 10A V0.4 同步发布；适配层 #1 移除；风险 #7 移除；节十一本条转入 11.2 已合一项；A-01 实体退化为 AbpUsers Extension（保留 `user_code` / `nova_user_id` / `last_sync_time` 三个业务字段）。

### 11.4 修订流程

**L2 提案上交路径：**
1. 10A 在节 11.3 登记提案（背景 + 多条岔路 + 利弊 + 工作量 + 决策窗口）
2. 提案上交详设作者，必要时联合业务方/Catio 团队评估
3. 评估结论：
   - **采纳** → 详设 10 出小版本（V1.1/V1.2…）+ 10A 同步合一为已合一项（迁入 11.2）
   - **驳回** → 在 10A 显式记录"为什么承担此实施成本"的理由，避免后人重复纠结

**版本一致性原则（V0.3 修订）：** L1 字段不允许 10A 单方面修改；L2 提案需走上述流程，不得绕过详设作者直接落地实施侧的不同字段类型；L3 实施层自决无需登记。

## 十四、一句话结论

**详设 10 是"是什么"，本文档是"怎么落"——把"自建表"工作量压到 8 张靠的不是减少业务能力，而是把 Catio 仓库的 `Nova.Platform` + `Nova.Workflow` 模块代码作为本地复用基座；适配层（ID 桥接 / 索引扩展 / 审批链快照映射 / DataScope 拆口径 / Properties 扩展 / 日志不可篡改三层防护）作为可量化的开发 backlog 显式登记，而非用"几乎重合"等措辞掩盖；阶段 A 零依赖先行、阶段 B 接外部、阶段 C 收束自动化，三段排期严格隔离风险。**
