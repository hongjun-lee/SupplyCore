## 一、文档目的

详设 10 定义 A-01~A-20 的**字段规范**；本文档定义其**实施层选择**：哪些表通过引用 Catio 仓库的框架模块在本地库实现，哪些表由 SupplyCores 自建，并显式列出二者之间所需的"适配层"工作量。

目标三句话：
1. 在不偏离详设 10 业务模型的前提下，把"自建表"压缩到 8 张（净 7 张，剔除 2 处 ⚠ 待 Nova.Platform 团队回复）；
2. 适配层工作（DataScope 拆口径 / Properties 扩展 / 节点字段语义映射 / 状态值域映射 / 审批链快照映射 / 日志不可篡改防护）作为可量化的开发 backlog 一并登记；
3. 阶段 0 / A / B / C 严格隔离风险——0 完成 L2 决议（已完成 2026-05-04）、A 零业务依赖、B 接 SSO 与 Workflow、C 收束自动化。

## 三、术语澄清

V0.1 评审中识别到 "Catio" 与 "Nova Platform" 在本文档语境内并非同一概念。三层拆分如下：

| 名称 | 性质 | 在本系统的角色 |
| --- | --- | --- |
| **集团 Nova Platform**（业务/运维语境） | 集团已部署的统一身份系统（**Nova 项目** 的运行实例） | SSO 权威源、组织/人员主数据来源；SupplyCores 通过 OAuth/OIDC 与之集成（概设 01 节 6.1） |
| **Catio 仓库目录**（= Nova 项目代码仓库的目录名，历史代号，不是另一套系统） | Nova 项目的源码目录 | 包含 `Nova.Platform`、`Nova.Workflow` 等可复用模块；SupplyCores 引用其代码 |
| **`Nova.Platform` / `Nova.Workflow` 模块** | Catio 仓库下的模块命名空间（**不是特指集团 Nova Platform 系统**） | SupplyCores 通过 `[DependsOn]` 引用其代码、在**本地部署的独立数据库**中建相关表（AbpUsers / RoleOrganization / DataScopeGrant / ApprovalRecord / ProcessInstance 等） |

**与概设 01 部署约束的一致性：**
- 概设 01 第 28、150、208 行约束："物资系统应独立部署、不与 Nova Platform 共库、不直接访问 Nova 底层数据库"
- 本文档 "REUSE" 含义：**复用 Catio 仓库的模块代码**，在本地库建相同结构的表；**不是访问集团 Nova Platform 的运行库**

## 四、整合策略原则

1. **能复用就不重建**：Catio 仓库的 `Nova.Platform` + `Nova.Workflow` 在 Nova 项目长期演进，业务模块只 `[DependsOn]` 即可享受 SSO/数据范围/审批流，避免双份维护。
2. **业务专属才自建**：高敏感操作清单、NC 接口日志、AI 调用日志、数据进出口等是 SupplyCores 业务特征，不进 Catio。
3. **复用即承认适配成本**：每张 REUSE 表都要登记"字段缺口"和所需"适配层"工作（见节五缺口列、节 5A 字段映射详表、节 6.2 适配层清单），不依赖"几乎重合""字段几乎一一对应"等模糊措辞。
4. **详设修订按字段性质分层**：L1 业务模型不可由 10A 单方面改；L2 技术实施允许 10A 提案修订详设；L3 命名/格式实施层自决。详细分类与 L2 提案登记见节十一。
5. **Schema 规约**：自建表全部落 schema `a`（auth/audit）；Catio `Nova.Workflow` 模块表落 `workflow` schema（**Catio 团队 2026-05-05 回函确认硬编码不可配置**）；Nova.Platform 模块表 schema 待 Platform 团队补充；框架基础表落 `public`。schema 命名前缀、迁移版本号策略参见 SupplyCores 仓库 `CLAUDE.md`（外部依赖）。
6. **阶段隔离风险**：阶段 0（L2 提案决议，已于 2026-05-04 完成）；阶段 A（自建表 + 接口契约）零业务/运维确认依赖，先行；阶段 B 接 SSO + Workflow；阶段 C 拆 C1（自动留痕，依赖 A）+ C2（高敏感拦截，依赖 A+B）。

### 6.2.1 Variables._ 前缀字段清单（V0.6 新增）

**背景：** Catio `Nova.Workflow.ProcessInstance.Variables` 是通用 jsonb 容器；适配层在其上序列化"详设 10 逻辑字段"作为冻结快照或运行态扩展。无规约时存在被滥用风险（_snapshot / _chainPath / 业务字段全塞一起 + 跨适配点重名碰撞 + 无主膨胀）。本节作为**适配层规约**显式登记当前已用前缀字段，新增需评审。

**当前清单：**

| 字段名 | 用途 | 来源 | 写入时机 | 类型 |
| --- | --- | --- | --- | --- |
| `Variables._snapshot` | 冻结审批模板结构（A-20 approval_chain_snapshot 短期实施落点） | 适配层 #3（拷贝 `ProcessDefinitionVersion.StepsConfig`） | 创建 ProcessInstance 时一次性写入，之后只读 | jsonb |
| `Variables._chainPath` | 冻结审批链路快照（A-14 approval_chain_path 后备落点；当前不启用，**仅当审计场景需冻结时才序列化**） | 适配层 #8（实时拼装的归档版本） | 触发时点：审计冻结需求出现时 | jsonb |

**规约：**
1. **命名约束**：所有适配层在 `Variables` 下序列化的"详设 10 逻辑字段"**必须以 `_` 前缀**，与业务自由变量隔离；前缀不可省略。
2. **新增评审**：新增 `Variables._<名>` 字段须在本表登记 + 由 10A 维护者批复（避免无主膨胀）；批复理由作为新版本的版本记录条目。
3. **只读约束**：`_snapshot` 类冻结字段写入后**不允许 UPDATE**；适配层代码须显式只读，仓储层禁止暴露 update 路径。
4. **长期演进**：当 Catio 后续提供原生字段（如 `ProcessInstance.ProcessDefinitionVersionId` + `approval_chain_snapshot`）时，相应 `Variables._` 字段应从本表移除并迁移到原生字段；移除动作随版本升级登记到版本记录。

**与 6.4 不可篡改三层防护的关系：** 本节规约约束的是**适配层在 Catio Variables 容器内的字段命名+只读语义**；6.4 三层防护约束的是**自建 6 张主日志表的 DB 层不可篡改**。两者互不重叠，互为补充。

### 6.4 日志不可篡改三层防护

详设 10 节 8.2 要求"日志表禁止 DELETE 和 UPDATE"。`CreationAuditedAggregateRoot` 仅在 ORM 层去掉了修改字段，并不强制 DB 层不可篡改。落实三层防护：

**适用范围：5 张主日志表 + 1 张子明细日志（共 6 张）**

| 层级 | 措施 |
| --- | --- |
| 仓储层 | 6 张日志表的 `IRepository<T>` 改为只暴露 `InsertAsync` / `GetAsync` / `GetListAsync`（编译期堵住）。子明细 `data_import_failure_row` 写入仅由父表服务暴露 |
| DB 应用账号权限 | PostgreSQL `GRANT INSERT, SELECT ON a.<日志表> TO supplycores_app`（不授 UPDATE/DELETE/TRUNCATE） |
| DB 兜底 | 对 6 张日志表加 `BEFORE UPDATE OR DELETE` 触发器抛错；DBA 维护账号走独立 PostgreSQL role，操作时须 A-12 留痕 |

**例外：** `a.business_data_scope_grant` 和 `a.sensitive_operation` 是维度表（可维护），不挂三层防护。

**回写详设 10：** 建议在详设 10 V1.1 节 8.2 加一句"具体落地见 10A 节 6.4"，避免详设说一套、实施另一套。

### 6.5 三层防护验证策略

| 验证层 | 策略 | 触发频度 |
| --- | --- | --- |
| 单元测试 | 仓储层不暴露 Update/Delete——编译期校验；测试覆盖 6 张日志表的 InsertAsync 通过 + 维度表 2 张 CRUD 全通过 | 每次构建 |
| DB 集成测试 | 直连应用账号尝试 `UPDATE a.operation_log SET ... WHERE id = 1` 期望 `ERROR: permission denied`；尝试 `DELETE` 同 | 每次 CI |

## 七、A-10/A-14 查询路径

### 7.1 主路径（业务单据视角查询审批记录）

```sql
SELECT ar.*
FROM workflow.approval_records ar
JOIN workflow.process_instances pi ON ar.process_instance_id = pi.id
WHERE pi.business_type = :business_type
  AND pi.business_id   = :business_id
ORDER BY ar.operated_at;
```

依赖 `process_instances.(business_type, business_id)` 二级索引——**Catio 已有 `ix_process_instances_business`**（Q7.1 ✓），无需扩展。schema 前缀 `workflow.` 由 Catio 硬编码（Q9.1 回函）。

### 7.2 A-14 完整审批链路实时拼装

```sql
SELECT ar.id, ar.node_key, ar.approver_id, ar.action, ar.comment, ar.operated_at, ar.attachments
FROM workflow.approval_records ar
WHERE ar.process_instance_id = :process_instance_id
ORDER BY ar.operated_at;
```

依赖索引 `ix_approval_records_instance`（Catio 已有）。适配层 #8 包装为 `IApprovalLogQueryService.GetChainPathAsync(processInstanceId)` 方法。

### 7.3 性能/审计报表兜底

如审计报表 SLA 要求高（详设 11 节 5 要求报表 P95 ≤ 5s），主路径 JOIN 在审批量大时可能不达标。备选方案：

| 方案 | 实施 | 触发条件 |
| --- | --- | --- |
| 物化视图 | `CREATE MATERIALIZED VIEW a.approval_record_view AS SELECT ... FROM workflow.approval_records ar JOIN workflow.process_instances pi ...`，定时 REFRESH | 审计报表查询频度 > 100 次/天且 P95 不达标 |
| 投影表 | 自建 `a.approval_record_projection`，通过 EventBus 订阅 ApprovalRecord 写入事件同步 | 物化视图刷新延迟不可接受时 |

**当前不预建**，待审计报表 SLA 验收时根据实测决定。

### 7.4 A-09 节点投影（占位）

A-09 节点配置内嵌于 `ProcessDefinitionVersion.StepsConfig` 的 V2 DSL JSON。"按 `approver_role_id` 反查参与了哪些审批模板"等查询需扫 JSON 或建投影表。

**当前不预建**：一期审批运维场景未明确高频反查需求；如阶段 B2 验收发现该需求高频，再建 `a.approval_node_projection`，通过 EventBus 订阅 ProcessDefinition 变更事件同步。
