## 二、逐条修订建议

### 修订 #1：§4.1 A-01 user_id 类型 bigint → Guid（L1）

**触发：** L2-提案-01 决议岔路 2（详设 10 V1.1 改 Guid，AbpUsers 直接当 user_copy）

**当前详设 10 §4.1：**

```
| user_id | bigint | PK | 内部 ID |
| nova_user_id | varchar(64) | NOT NULL UQ | Nova Platform 原始 ID |
```

**建议改为：**

```
| user_id | uuid | PK | = AbpUsers.Id（ABP 标准，Guid） |
| nova_user_id | varchar(64) | NOT NULL UQ | 集团 Nova Platform 原始 ID |
```

**联动改动：** 全文所有 `user_id bigint FK→A-01` 改为 `user_id uuid FK→A-01`。
- §4.3 A-03 user_id：bigint → uuid
- §4.12 A-12 user_id / operator_id：bigint → uuid
- §4.13 A-13 user_id：bigint → uuid
- §4.15 A-15 operation_person_id：bigint → uuid
- §4.16 A-16 caller_user_id：bigint → uuid
- §4.21（如有）created_by/updated_by 等：bigint → uuid

**回写 10A 影响：** V0.5 节 6.2 适配层 #1（ID 桥接）删除；节十二风险 #7 删除；自建 8 张表 user_id 字段全部 long → Guid。

**触发：** L2-提案-02 决议岔路 2

**当前详设 10 §4.7：** A-07 data_permission_rule 作为正式表存在。

**回写 10A 影响：** V0.5 节五 A-07 行从"DEFER（待 L2-提案-02 批复）"改为"REUSE/ADD 都不适用，按详设 V1.1 标 [一期不实施]"；节 11.3 L2-提案-02 迁入 11.2 已合一项。

### 修订 #5：§4.9 A-09 11 节点字段改"逻辑字段"（L2）

**触发：** Catio Q8.1 + Q8.2 回函（V2 DSL 图结构 + 5 字段间接表达）

**当前详设 10 §4.9：** A-09 approval_node 11 字段（node_sequence/node_type/node_name/approver_type/approver_role_id/approver_org_id/approve_condition/is_parallel/reject_goto_node 等）。

**建议改为：在 §4.9 开头加注 + 11 字段表加"实施落点"列：**

> **实施落点说明：** 本表 11 字段为**逻辑字段**，物理落到 Catio `Nova.Workflow.ProcessDefinitionVersion.StepsConfig` 的 V2 图结构 DSL（`WorkflowGraphDto: { nodes, edges, completionActions }`）。详细映射见 10A V0.5 节 5A.4。

| 详设 10 字段 | Catio V2 DSL 落点 | 实施备注 |
| --- | --- | --- |
| node_sequence | 由 edges 拓扑推导 | 不落字段；如需线性序号在适配层做拓扑排序后编号 |
| node_type | `WorkflowNodeDto.type`（START/APPROVAL/CONDITION/FORK/JOIN/END/...）+ `approvalConfig.name`（业务名） | 详设 5 种业务值（发起/初审/复核/会签/终审）映射到技术节点 + 业务名标识 |
| node_name | `approvalConfig.name` | |
| approver_type | `ApproverRule.RuleType`（独立表） | 节点引用 `approvalConfig.approverRuleCode` |
| approver_role_id | `ApproverRule.Config jsonb` | 同上 |
| approver_org_id | `ApproverRule.Config jsonb` | 同上 |
| approve_condition | 独立 `CONDITION` 节点的 `conditionConfig.expression` | 高级表达式如 `amount * rate > 10000 && department == "FIN"` |
| is_parallel | `approvalConfig.signMode = COUNTERSIGN`（会签）/ `OR_SIGN`（或签）/ `SINGLE` | |
| reject_goto_node | `approvalConfig.rejectBehavior` + `allowedReturnStepKeys` | END / RETURN_TO_INITIATOR / RETURN_TO_STEP |

**回写 10A 影响：** V0.5 节 5A.4 字段映射详表 + 节 6.2 适配层新增 #6（节点字段语义映射 3~5 人天）。

### 修订 #8：§4.20 A-20 instance_state 6 → 5 状态（L2）

**触发：** Catio Q7.3 回函（无"草稿"对应）+ Q13a 决议（业务表自管草稿）+ Q11a 决议（已终止 = CANCELLED）

**当前详设 10 §4.20：**

```
| instance_state | varchar(20) | NOT NULL | 草稿/审批中/已通过/已驳回/已撤回/已终止 |
```

**建议改为：**

```
| instance_state | varchar(20) | NOT NULL | 审批中/已通过/已驳回/已撤回/已终止（5 状态；"草稿"由业务表 status 字段自管，提交后才创建本表实例） |
```

**Catio 状态值域映射：**

| 详设状态 | Catio ProcessInstanceStatus | Result | 备注 |
| --- | --- | --- | --- |
| 审批中 | RUNNING | — | |
| 已通过 | COMPLETED | APPROVED | |
| 已驳回 | REJECTED 或 RETURNED | REJECTED | RETURNED 含退回发起人 / 退回指定步骤（rejectBehavior 区分） |
| 已撤回 | CANCELLED | CANCELLED | 由发起人主动撤回 |
| 已终止 | CANCELLED | CANCELLED | 系统/管理员强制终止；与"已撤回"靠业务字段区分（如 cancel_reason） |

**业务规则补充：**
- 提交前业务表 status='草稿'，不创建 ProcessInstance；
- 提交后创建 ProcessInstance，业务表 status='审批中'，instance_state=审批中；
- "已撤回"与"已终止"在 Catio 都映射到 CANCELLED + Result=CANCELLED，详设层用 cancel_reason 或 cancel_actor 字段区分。

**回写 10A 影响：** V0.5 节 5A.6 字段映射详表 + 节 6.2 适配层 #7（状态值域映射 1 人天）。
