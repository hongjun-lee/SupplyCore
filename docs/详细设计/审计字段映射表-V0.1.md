## 一、命名约定（2026-05-11 拍板）

档 A 后端工程 `SupplyCores` 采用**方案 D**：
- 详设字段名保留 **snake_case** 业务风格（PostgreSQL 行业惯例 + DBA 友好）
- 与 **ABP** `FullAuditedAggregateRoot<TKey>` 概念对齐
- 工程层加 `EFCore.NamingConventions` → ABP 实体属性自动映射 DB 列名 snake_case
- 所有业务实体继承自定义抽象基类 **`SupplyCoresFullAuditedAggregateRoot<TKey>`**（ABP 基类 + `created_org_id` + `sub_group_id` + `delete_reason` 三个详设独有字段；`sub_group_id` 于 V0.2 增补，详见节 2.2）

## 三、抽象基类（自定义）

所有业务实体**必须继承** `SupplyCoresFullAuditedAggregateRoot<TKey>`（不是直接继承 ABP `FullAuditedAggregateRoot<TKey>`）：

```csharp
// Nova.SupplyCores.Domain.Shared/Entities/Auditing/SupplyCoresFullAuditedAggregateRoot.cs

using Volo.Abp.Domain.Entities.Auditing;

namespace Nova.SupplyCores.Entities.Auditing;

/// <summary>
/// SupplyCores 业务实体基类（详设 01 §4.2 + §4.3 + 审计字段映射表 V0.2）。
/// 在 ABP FullAuditedAggregateRoot 基础上扩展三个详设独有字段：
///   - CreatedOrgId 创建人所属组织（细粒度过滤辅助）
///   - SubGroupId   数据所属二级集团（A-06 数据范围"以二级集团为隔离边界"一刀切主用字段，V0.2 新增）
///   - DeleteReason 软删除原因（详设 11 §13 合规留痕要求）
/// </summary>
public abstract class SupplyCoresFullAuditedAggregateRoot<TKey>
    : FullAuditedAggregateRoot<TKey>
{
    /// <summary>详设 01 §4.2：创建人所属组织（FK→M-01）</summary>
    public virtual long? CreatedOrgId { get; protected set; }

    /// <summary>
    /// 详设 01 §4.2（V1.1 新增）：数据所属二级集团（FK→M-01）。
    /// 写入时由 CreatedOrgId 回溯 M-01.sub_group_id 确定；集团级共享主数据可为 NULL。
    /// A-06 数据范围过滤的主用字段；详见
    /// 评审留痕/数据隔离边界sub_group_id修订建议清单-V0.1.md 修订 #2 + 原则 1-4。
    /// </summary>
    public virtual long? SubGroupId { get; protected set; }

    /// <summary>详设 01 §4.3：软删除原因（高敏感操作必填）</summary>
    public virtual string? DeleteReason { get; protected set; }

    protected SupplyCoresFullAuditedAggregateRoot() { }
    protected SupplyCoresFullAuditedAggregateRoot(TKey id) : base(id) { }

    /// <summary>
    /// 软删除时显式设置原因（不直接暴露 setter）。
    /// 由 IRepository.DeleteAsync 钩子或 Domain Service 调用。
    /// </summary>
    public virtual void MarkAsDeleted(string reason)
    {
        DeleteReason = reason;
        // IsDeleted 由 ABP 框架在 DeleteAsync 时自动设置
    }
}
```

## 四、EF Core 配置要点

```csharp
// SupplyCoresDbContextFactory.cs / OnConfiguring
optionsBuilder.UseSnakeCaseNamingConvention();
```

效果：
- ABP 属性 `CreationTime` → DB 列 `creation_time`（自动）
- 自定义属性 `CreatedOrgId` → DB 列 `created_org_id`（自动）
- 自定义属性 `SubGroupId` → DB 列 `sub_group_id`（自动）
- 表名 `Material` → `material`（自动）
- ABP 框架表（`AbpAuditLog`）也走 snake_case → `abp_audit_log`

完整接入步骤见 [`Sprint-0.5-任务卡-V0.1.md`](../Sprint/Sprint-0.5-任务卡-V0.1.md)。

## 五、适用范围

| 实体类型 | 处理方式 |
|---|---|
| **业务实体**（M / P / C / S / E / F / R 域） | 继承 `SupplyCoresFullAuditedAggregateRoot<TKey>`，含全部审计字段 |
| **操作日志类**（A-13 OperationLog 等 append-only）| 继承 `CreationAuditedAggregateRoot<TKey>`（只有创建审计，无修改/软删除）|
| **审计实体本身**（A-13 / A-14 / A-15 等）| 不继承业务基类（避免循环审计；与 ABP AbpAuditLog 互补设计）|
| **配置 / 枚举类**（SY-XX）| 视具体情况，多数继承 `FullAuditedAggregateRoot<TKey>` |

## 六、与 ABP AbpAuditLog 的关系

**互补，不是替代**：

| 实现 | 索引视角 | 用途 | 字段维度 |
|---|---|---|---|
| **ABP AbpAuditLog**（默认）| HTTP 请求 + 方法调用 | 技术运维追溯 | TenantId / UserId / Url / Exception / Duration / ... |
| **A-13 operation_log**（详设独有）| 业务单据（business_bill_type/id/no）| 业务追溯"某单据被谁改过什么" | UserId / BusinessBillType / OldValue / NewValue（JSONB）|

两者并存，由 Sprint 0 Day 5-6 接通的自定义 `IAuditingStore` 统一驱动。

## 七、违反约定的检测（CI/Review 检查项）

| 检查项 | 命令 |
|---|---|
| 详设文档不应有旧字段术语 | `grep -rE "\bcreated_at\b\|\bversion_no\b\|\bcreated_by\b\|..." docs/详细设计/*.md` 应零结果（F-14 接口表 `version_no` 例外，是 API 版本号不是审计字段）|
| Domain 实体不应直接继承 `FullAuditedAggregateRoot` | `grep -r ": FullAuditedAggregateRoot" modules/nova.supplycores/src/Nova.SupplyCores.Domain/` 应零结果（应继承 `SupplyCoresFullAuditedAggregateRoot`）|
| DB 列名应 snake_case | `SELECT column_name FROM information_schema.columns WHERE table_schema IN ('m','p','c','s','e','f','r','a','sy') AND column_name ~ '[A-Z]'` 应零结果 |
| 业务表必须有 `sub_group_id` 列（**V0.2 新增**）| `SELECT table_schema, table_name FROM information_schema.tables WHERE table_schema IN ('m','p','c','s','e','f','r') AND table_name NOT IN (<白名单>) AND NOT EXISTS (SELECT 1 FROM information_schema.columns c WHERE c.table_schema=tables.table_schema AND c.table_name=tables.table_name AND c.column_name='sub_group_id')` 应零结果；白名单见 `评审留痕/数据隔离边界sub_group_id修订建议清单-V0.1.md` 原则 1 例外项 |
| 业务 SQL 不应递归 org_path 做二级集团过滤（**V0.2 新增**）| Code Review 项：检查 `org_path LIKE` 配合"二级集团范围"语义的过滤代码，应改用 `WHERE sub_group_id = ?` |
