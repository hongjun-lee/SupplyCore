## 一、背景与问题判断

当前物资系统中已经出现“平台基础”“功能管理”“用户权限管理”等菜单和页面。从页面表达和部分 mock 口径看，其中一部分能力容易被理解为从 Nova Platform 复制而来，例如跨集团 RBAC V2/V3、多租户 Host 管理员、Feature toggle、Nova 演进路线等。

这类设计在早期快速搭建独立 Host、验证 ABP 权限、联调 Nova 组织/用户/SSO 时有一定合理性，但如果进入正式菜单和正式页面，会带来三个问题：

1. **职责边界不清**：外部用户会认为物资系统重复建设了 Nova 平台后台。
2. **实施责任不清**：用户、组织、平台角色、平台权限到底由 Nova 管还是物资系统管，容易产生双维护。
3. **产品观感不稳**：正式业务系统出现 mock、placeholder、RBAC 演进、Host/Tenant 等研发语言，会削弱系统专业度。

因此，本方案的核心判断是：

> 复用 Nova 的技术模式是合理的，但复制 Nova 的平台管理模块不合理。物资系统应收敛为“物资业务权限配置”，而不是“平台功能管理”。

---

## 二、总体定位

物资系统不建设一套新的 Nova 平台后台。Nova Platform 作为上位平台，负责统一身份、组织、人员、平台角色、平台权限和统一登录；物资系统负责本系统内的业务角色、业务功能权限、数据范围、审批规则、审计和业务配置。

建议将相关菜单从“平台基础 / 功能管理”调整为：

- **权限与审批**
- **系统配置**
- **业务角色授权**
- **菜单权限映射**
- **数据范围授权**
- **组织同步台账**

其中 Nova 相关内容只作为同步状态、只读查看、联调诊断或跳转入口，不作为物资系统自身的主数据维护能力。

## 三、整改目标

1. **消除重复建设感**  
   正式页面不呈现 Nova 平台后台式功能，不让用户误解为物资系统维护统一用户、统一组织和平台权限。

2. **保留业务权限能力**  
   物资系统仍需维护自身 `SupplyCore.*` 菜单、按钮、业务角色、数据范围和高敏感操作规则。

3. **统一业务语言**  
   将技术语言、研发语言、原型语言改为业务方能理解的物资管理语言。

4. **支持 Nova 对接**  
   保留 Nova 组织、人员、权限、SSO 的同步状态、接口健康、异常诊断和跳转入口。

5. **形成可验收标准**  
   正式菜单、正式页面、正式文档均能通过“无 Nova 平台后台复制痕迹、无 mock 原型文案”的检查。

## 四、职责边界

| 事项 | Nova Platform | 物资系统 |
| --- | --- | --- |
| 用户身份 | 权威源，负责统一账号、登录、人员基础信息 | 保存必要业务副本，只读引用 |
| 组织架构 | 权威源，负责集团、二级集团、厂矿、部门、班组等组织树 | 同步组织副本，补充仓库、业务属性和数据范围 |
| 平台角色 | 统一平台角色由 Nova 维护 | 不维护平台级角色 |
| 业务角色 | 不作为物资业务职责权威源 | 维护采购、仓库、财务、审计、设备等业务角色 |
| 平台权限 | Nova 维护平台菜单、平台功能、通用权限 | 不维护 Nova 平台权限 |
| 物资功能权限 | 可承载统一授权入口或权限同步 | 维护 `SupplyCore.*` 菜单、按钮、模块权限 |
| 数据范围 | 提供组织范围、人员归属、平台数据范围能力 | 结合仓库、物料类别、供应商、单据等业务维度配置 |
| 审批人解析 | 提供人员、组织、岗位、ORG_ROLE 等基础能力 | 按物资业务规则配置流程、节点、加签、会签 |
| 审计 | 平台审计保留统一身份和平台操作 | 记录业务操作、权限变更、接口调用、高敏感动作 |

边界原则：

- 物资系统可以消费 Nova 数据，但不反向成为 Nova 的权威维护入口。
- 物资系统可以维护业务角色，但不维护平台通用角色。
- 物资系统可以配置业务功能权限，但不维护其他系统或平台功能权限。
- 物资系统可以显示 Nova 同步状态，但正式业务菜单中不出现 Nova 平台后台管理语义。

## 五、菜单重构建议

### 5.1 一级菜单建议

建议正式菜单按业务域重构为以下结构：

1. **基础档案**
   - 组织同步台账
   - 仓库档案
   - 库区货位
   - 物料分类
   - 计量单位
   - 供应商档案
   - 成本中心 / NC 映射

2. **业务运营**
   - 需求计划
   - 采购申请
   - 招投标
   - 合同管理
   - 到货验收
   - 入库出库
   - 库存盘点
   - 废旧处置
   - 设备台账 / 租赁 / 维修

3. **财务与接口**
   - NC 接口状态
   - 凭证管理
   - 对账管理
   - 月结 / 反结
   - 接口日志

4. **权限与审批**
   - 业务角色授权
   - 菜单权限映射
   - 数据范围授权
   - 审批流程配置
   - 高敏感操作规则

5. **系统审计**
   - 操作日志
   - 审批日志
   - 权限变更日志
   - 接口调用日志
   - 安全审计

6. **系统联调诊断**（内部入口，默认不进业务菜单）
   - Nova 同步状态
   - SSO 接入状态
   - 外部接口健康检查
   - 权限同步诊断

7. **其它**（V0.1 内 c 版新增 / Spencer 2026-05-21 拍板兜底原则 ⭐）
   - 现有菜单项在新菜单体系中**不好归类**的，先归此处不阻塞推进
   - 阶段一盘点中归属模糊 / 跨业务域 / 暂未明确归属的，**默认归「其它」**
   - 阶段三业务化时**逐步细分**到上述 1-6 大类
   - 兜底原则参考物料分类基线（每大类含 `99 其他` 的同款思路）

> **「其它」与「系统联调诊断」的区别**：
> - **其它**（业务菜单兜底）：业务用户可见 / 暂未细分 / 待业务方明确归属
> - **系统联调诊断**（技术内部入口）：技术 / 运维 / 系统管理员可见 / 业务用户不可见

## 六、“功能管理”重构方案

“功能管理”不建议作为正式名称保留。若保留其能力，应重构为“菜单权限映射”或“业务功能权限配置”。

### 6.1 保留能力

1. 管理物资系统自身菜单、页面、按钮权限。
2. 权限编码统一采用 `SupplyCore.*` 范围，如 `SupplyCore.Materials.Create`、`SupplyCore.Inventory.View`。
3. 将业务角色绑定到菜单、按钮和操作权限。
4. 支持高敏感操作专项权限，如付款、反结、库存调整、废旧处置、黑名单解除、接口重推等。
5. 支持按业务域查看权限树，如采购、库存、合同、财务、设备、报表、系统审计。
6. 记录权限变更审计。

### 6.2 不保留能力

1. 不创建、编辑、删除 Nova 用户。
2. 不维护 Nova 组织主数据。
3. 不维护 Nova 平台角色。
4. 不维护 Nova 平台菜单。
5. 不维护其他业务系统的功能权限。
6. 不出现 ABP Host/Tenant、Feature toggle、RBAC 演进路线等平台后台概念。

### 6.3 页面表达建议

原“功能管理”页面建议拆为三个业务页面：

1. **菜单权限映射**  
   展示物资系统菜单、页面、按钮与 `SupplyCore.*` 权限编码的对应关系。

2. **业务角色授权**  
   将采购、仓库、财务、审计、设备等业务角色绑定到功能权限和操作权限。

3. **数据范围授权**  
   将业务角色或人员授权到组织、仓库、物料类别、供应商、单据范围等业务数据维度。

## 七、Nova 痕迹与 mock 文案清理要求

本项作为正式化整改 P0。

### 7.1 硬边界

> 物资系统正式界面不得呈现 Nova 平台后台、ABP 多租户、RBAC 演进、Feature Toggle、Mock Placeholder 等研发/平台原型文案；所有正式页面必须转换为物资业务语言和用户可理解的管理口径。

### 7.2 必须清理的表达

正式菜单、正式页面、正式文档中，清理或替换以下表达：

- “跨集团 RBAC V2/V3”
- “多租户 Host 管理员”
- “Feature toggle”
- “ABP Host / Tenant / Cross-tenant”
- “Nova 演进路线”
- “mock”
- “placeholder”
- “待 backend”
- “平台权限 V3”
- “Federation”
- “SSO 调试”
- “Nova RBAC tracker”
- “Q1 next year 多租户拍板”

### 7.3 替换口径

| 原表达 | 建议替换 |
| --- | --- |
| 跨集团 RBAC V2/V3 | 跨组织数据范围 / 集团级授权 |
| 多租户 Host 管理员 | 集团管理员 / 系统管理员 |
| Feature toggle | 功能启用状态 / 参数配置 |
| ABP Tenant | 组织范围 / 单位范围 |
| Federation | 跨系统协同 / 外部平台对接 |
| mock 数据 | 示例数据（仅演示环境）或移除 |
| 待 backend | 暂未接入正式服务（内部说明，不展示给业务用户） |
| Nova 演进路线 | 平台对接状态（内部诊断） |

### 7.4 页面处理原则

1. 属于 Nova 平台后台的页面：从物资系统正式菜单移除，必要时提供跳转 Nova 的入口。
2. 属于联调诊断的页面：放入“系统联调诊断”内部入口，不进入业务菜单。
3. 属于物资系统自己的权限能力：改成业务语言，保留并接真实接口。
4. mock 页面：正式环境不可展示 mock 口径；未接真实接口的页面不得作为正式功能上线。

## 八、技术落地清单

### 8.1 后端

1. 保留 `SupplyCoresPermissions` 和 `SupplyCoresPermissionDefinitionProvider`，作为物资系统业务权限定义源。
2. 保留 ABP Permission Management 作为本系统权限授权能力，但范围限定为 `SupplyCore.*`。
3. 业务角色、数据范围、审批规则、审计日志继续落在物资系统业务库。
4. 用户、组织、人员从 Nova 同步或通过 API 获取，本系统只保存必要业务副本。
5. 不新增与 Nova 平台权限组、平台菜单、平台角色重复的权威表。
6. 权限变更、数据范围变更、高敏感规则变更必须写审计。

### 8.2 前端

1. 正式菜单中移除或隐藏多租户、RBAC 演进、Nova tracker、Feature toggle 等原型页。
2. 将“功能管理”改为“菜单权限映射”或“业务功能权限配置”。
3. 将“用户权限管理”改为“业务角色授权”。
4. 将“组织管理”改为“组织同步台账”。
5. 清理页面上的 mock、placeholder、待 backend、V2/V3 演进等文案。
6. 对内部诊断页增加明确环境控制，仅开发/测试/管理员可见。

### 8.3 文档

1. 在权限审批详细设计中补充本方案边界。
2. 在菜单设计或原型设计中更新正式菜单名称。
3. 在开发任务卡中增加“正式页面禁用研发文案”的验收项。
4. 在 Nova 对接文档中明确“平台权威源 / 业务系统副本”的关系。

### 8.4 测试与验收

1. 增加正式菜单快照检查：不出现被禁用的 Nova/ABP/mock 词汇。
2. 增加权限页面 E2E：菜单权限映射、业务角色授权、数据范围授权可打开并显示业务化文案。
3. 增加后端权限测试：`SupplyCore.*` 权限定义和授权检查不回退。
4. 增加审计测试：业务角色授权、数据范围授权变更写入审计。

## 十、验收标准

1. 正式菜单中不再出现“平台基础”“功能管理”这类容易混淆平台职责的名称。
2. 正式页面中不出现“跨集团 RBAC V2/V3”“多租户 Host 管理员”“Feature toggle”“mock”“placeholder”“待 backend”等研发或平台原型文案。
3. 用户、组织、人员页面明确体现来自 Nova 或集团统一平台，本系统只做同步台账和业务副本。
4. 权限配置页面明确体现物资系统业务权限，不维护 Nova 平台权限。
5. `SupplyCore.*` 权限、业务角色、数据范围、审批规则、高敏感操作规则可以闭环配置。
6. 权限变更和数据范围变更有审计记录。
7. 内部诊断页不进入业务用户正式菜单，且有权限控制。

## 十二、推荐结论

建议采用以下结论作为整改方向：

> 物资系统保留业务权限配置能力，但不保留平台功能管理定位；将“平台基础 / 功能管理”调整为“权限与审批 / 菜单权限映射 / 业务角色授权 / 数据范围授权”。Nova 作为统一身份、组织、人员、平台权限的权威源，物资系统只消费、同步和诊断，不复制 Nova 平台后台。正式页面必须清理 Nova、ABP、多租户、RBAC 演进和 mock 原型文案，统一转为物资供应业务语言。
