# 开发实施引用索引（V0.2）

**版本：** V1.0
**日期：** 2026-05-02
**文档性质：** 开发实施引用索引 / 代码仓库启动前治理文件
**适用阶段：** 详细设计编制期、开发仓库筹备期、供应商实施交接前

---

## 一、文档目的

本文档用于在详细设计尚未全部完成前，先明确未来开发实施阶段的文档引用规则。

本文档重点回答：

- 将来写代码是否应另建项目仓库
- 代码仓库与当前文档仓库是什么关系
- 开发团队应优先参考哪些文档
- 不同文档之间出现冲突时以什么为准
- 哪些详细设计完成前不宜直接开工
- 需求变更、设计调整和代码实现之间如何回写闭环

本文档不替代详细设计分卷，不直接定义数据库表、接口字段、页面原型或代码结构。真正指导开发落地的直接依据，应以后续 `docs/详细设计/01-11` 分卷为主。

## 二、仓库定位

### 2.1 当前仓库定位

当前 `SupplyCore` 仓库定位为：

**物资供应管理系统需求、招标、概要设计、详细设计资料库。**

本仓库主要沉淀：

- 集团统一建设原则
- 项目需求与范围
- 招标技术要求
- 概要设计
- 详细设计
- 编码、接口、AI 接入等专项规则
- 决策、确认、回写和变更依据

### 2.2 未来代码仓库定位

未来正式开发时，建议另建独立代码仓库，不建议在当前文档仓库中混放业务代码。

建议命名可选：

| 建议仓库名 | 适用含义 |
| --- | --- |
| `SupplyCores` | 偏完整应用系统，适合前后端与部署一体管理 |
| `SupplyCore.Service` | 偏后端服务，适合前后端分仓 |
| `SupplyCore.System` | 偏项目整体交付，适合供应商实施总仓 |

最终命名可在供应商技术方案或内部开发立项时确定。无论命名如何，代码仓库均应把当前 `SupplyCore` 文档仓库作为上游设计依据库。

## 三、文档引用优先级

开发实施阶段原则上按以下优先级理解文档。

| 优先级 | 文档层级 | 主要用途 | 开发阶段作用 |
| --- | --- | --- | --- |
| 1 | 集团统筹层 | 统一权限、SSO、接口、AI、信创、运维审计等上位原则 | 不得突破的集团级约束 |
| 2 | 招标正文层 | 对外正式发出的建设范围与交付要求 | 供应商合同与验收边界 |
| 3 | 详细设计层 | 表结构、字段、接口、状态机、审批流、配置项 | 开发直接依据 |
| 4 | 概要设计层 | 架构、模块边界、数据主线、集成边界 | 理解设计背景和范围边界 |
| 5 | 项目主文档层 | 需求、流程、角色、决策、确认事项 | 业务口径追溯依据 |
| 6 | 详细规则与专项附件 | 编码、财务接口、AI Tool、验收条款等专题展开 | 专项实现和联调依据 |

### 排序语义说明

本节排序是**开发实施场景下的"参考强度"**，不是**编制收敛阶段的"内部优先级"**。两套排序在不同语境下都适用，但语义不同：

| 排序来源 | 语境 | 主要用途 | 排序理由 |
| --- | --- | --- | --- |
| 本节优先级（开发实施） | 开发人员日常编码、联调、调试 | 决定"先看哪份文档" | 详细设计是离实现最近的依据，所以排前 |
| `docs/需求梳理/00-文档分层与引用关系说明-V1.0.md` 优先级（编制收敛） | 文档编制、口径仲裁、范围扩缩 | 决定"上位文档约束下位文档" | 集团统筹 → 项目主文档 → 招标正文 → 概要 → 详设 → 专项附件 |

如出现需要做"上位约束"判断的冲突（例如详设是否扩了招标正文范围），应回到 `00 文档分层` 的编制收敛优先级；本节排序仅用于开发场景下"先看哪份"。

说明：

- 如果详细设计与概要设计冲突，应先判断是否属于概要设计需回写调整，不应在代码中直接自行选择。
- 如果详细设计与招标正文冲突，应提交项目内部评审，不能由开发团队直接扩大或缩小范围。
- 如果专项附件与主文档冲突，以主文档和招标正文为准；专项附件负责细化，不负责改范围。

## 五、开发启动门

### 5.1 不宜直接开工的情况

出现以下情况时，不建议进入正式编码：

- 相关模块尚无对应详细设计分卷
- 关键表结构、状态机、接口字段仍未定义
- 涉及 NC、权限、审批、库存影响的规则仍处于口头讨论状态
- 需求变更未回写到文档
- 与招标正文或集团统筹原则存在冲突但未评审

### 5.2 可先行准备的工作

在详细设计分卷未全部完成前，可以先行准备：

- 技术栈调研与供应商技术方案比选
- 代码仓库结构草案
- 开发规范、分支规范、提交规范
- CI/CD、测试环境、数据库选型调研
- 原型工具和需求跟踪工具准备
- 与 Nova Platform 的 SSO、组织、人员、权限接口对接方式调研

上述准备工作不得反向锁死业务规则和表结构。

### 5.3 模块级开工条件

单个模块进入编码前，至少应满足：

| 条件 | 要求 |
| --- | --- |
| 范围明确 | 模块边界与一期/二期边界清楚 |
| 数据明确 | 主表、明细表、关键字段、关联键已定义 |
| 状态明确 | 单据状态、状态迁移、回退路径已定义 |
| 权限明确 | 角色、数据范围、审批节点框架已定义 |
| 接口明确 | 外部接口、内部触发点、幂等键、错误处理框架已定义 |
| 配置明确 | 可配置项、默认值、待确认占位和责任方已标注 |

## 六、代码仓库建议结构

代码仓库的最终结构应结合供应商技术栈确定。以下仅作为开发实施阶段的组织建议。

```text
SupplyCore.App/
  docs/
    design-references.md
    change-log.md
  src/
    backend/
    frontend/
    integrations/
  database/
    migrations/
    seeds/
    docs/
  tests/
    unit/
    integration/
    e2e/
  deploy/
    env/
    scripts/
  README.md
```

建议代码仓库内保留一份 `docs/design-references.md`，只做引用，不复制当前仓库的大量正文。该文件应写明当前代码版本对应的文档版本，例如：

| 代码版本 | 对应文档基线 |
| --- | --- |
| 开发启动版 | 招标 V1.1 + 概要设计 V0.1 + 详细设计 V1.0 |
| 联调版 | 详细设计 V1.0 + NC 接口联调确认记录 |
| 验收版 | 招标 V1.1 + 详细设计定稿 + 验收条款 |

### 6.1 多技术栈 / 多仓布局适配

上述目录结构是单仓单技术栈的最简布局，实际仓库可结合供应商技术方案调整：

| 布局形态 | 适用情况 | 注意事项 |
| --- | --- | --- |
| 单仓 monorepo | 前后端统一管理、便于一体化部署 | 前后端目录边界清晰，CI 流水线分别触发 |
| 前后端分仓 | 前后端独立技术栈或独立交付 | 接口契约目录独立维护，避免双向同步漂移 |
| 多服务多仓（微服务） | 业务能力域级独立部署 | 须设公共契约仓和公共主数据仓，避免主数据各自落库 |

无论哪种布局，**接口契约目录** `contracts/` 都应作为独立目录维护：

- OpenAPI / Protobuf / JSON Schema 契约文件（NC 接口、Nova、招采平台、AI Tool）
- 契约版本与详细设计版本绑定
- 契约变更必须按节七变更评审流程走

### 6.2 AI Tool 调用的开发约束

按 `docs/概要设计/06-报表预警与AI能力概要设计-v0.1.md` 与 `docs/集团统筹/AI接入专题/` 系列附件，AI 平台调用本系统的开发实施必须遵守：

- **AI 平台不得直连本系统底层数据库**，必须通过 8 类受控 Tool 接口访问
- AI Tool 接口必须共享业务系统的权限骨架（角色、数据范围、敏感字段脱敏），不得绕开
- AI Tool 的调用必须留痕到 `A-16 ai_call_log`（详 `01-数据库逻辑模型-V1.0.md`），包括调用方、入参、出参摘要、数据范围
- 一期 AI 能力以只读为主，不进入审批、付款、反结、重推等执行场景
- AI Tool 的输入输出契约视为对外接口，按本节 6.1 `contracts/` 目录管理

代码仓库内不得通过任何方式向 AI 平台暴露数据库连接串、原始 SQL 接口或不受控的查询通道。

## 七、需求变更与回写机制

### 7.1 文档先行原则

开发过程中发现需求变化、规则冲突或实现不可行时，应先回到文档仓库处理，而不是直接在代码中静默调整。

### 7.2 变更发起方

变更可由以下角色发起：

- 业务部门：业务规则、流程、口径调整
- 财务部门：财务规则、接口、对账要求调整
- 网信办：技术架构、信创、安全、运维要求调整
- 项目领导小组：项目范围、阶段计划、回退决策调整
- 实施供应商：开发中发现的设计问题、可行性问题

实施供应商不得绕过项目方直接使文档变更生效；可通过变更申请、评审意见或受控 PR / 补丁提交修改建议，由项目联系人或对口部门评审后回写。

### 7.3 变更评审流程

| 步骤 | 责任方 | 产出物 |
| --- | --- | --- |
| 1 提交变更申请 | 发起方 | 变更申请单（背景、影响、替代方案） |
| 2 影响范围评估 | 项目联系人 + 对口部门 | 影响清单（影响哪些文档、模块、阶段） |
| 3 评审决策 | 简单变更：项目联系人 + 对口部门；复杂变更或涉及合同义务：项目领导小组 | 评审纪要 |
| 4 回写文档 | 对口部门 + 项目联系人 | 文档更新 + 变更日志 |
| 5 同步开发 | 项目联系人 → 实施供应商 | 实施变更通知 |
| 6 验收闭环 | 实施供应商 → 项目联系人 | 实施回执与验证记录 |

### 7.4 紧急变更特例

涉及生产事故、合规风险、合同违约的紧急变更：

- 可先在代码仓库执行紧急修复
- 但 5 个工作日内必须回写文档并提交变更申请补办评审
- 不允许"先修代码、后写文档"成为常态
- 紧急变更的处理决策权归项目领导小组

### 7.5 需要强制回写的事项

以下事项必须回写文档：

- 表结构、字段含义、字段约束变化
- 单据状态、审批流程、回退规则变化
- NC 接口字段、触发时点、对账规则变化
- 权限、数据范围、敏感操作范围变化
- 一期/二期范围变化
- 报表口径、AI Tool 能力范围变化
- 上线、回退、验收条件变化

### 7.6 不必回写的事项

以下事项一般不必回写当前文档仓库，可留在代码仓库：

- 代码重构
- 文件目录调整
- 单元测试补充
- 框架内部配置优化
- 不改变业务语义的性能优化

### 7.7 变更日志规范

每份被变更的文档应在文档末尾或独立变更日志小节记录：

- 变更日期、变更前版本、变更后版本
- 变更内容摘要
- 变更原因（链接到变更申请单或会议纪要）
- 变更决策方
- 对开发的影响范围

## 八、冲突处理规则

开发实施中如发现文档冲突，按以下顺序处理。

| 冲突类型 | 处理规则 |
| --- | --- |
| 集团统筹原则 vs 项目文档 | 以集团统筹原则为上限，项目文档不得突破 |
| 招标正文 vs 详细设计 | 以招标正文为外部交付边界；详细设计如需调整，必须形成评审记录 |
| 详细设计 vs 概要设计 | 详细设计应承接概要设计；如概要设计不准确，应回写概要设计 |
| 详细设计分卷之间冲突 | 以公共底座分卷和最新评审记录为准，并同步修改相关分卷 |
| 需求文档旧口径 vs 后续确认表 | 以已签字或项目内确认的最新确认表为准，并回写旧文档 |
| 专项附件 vs 主文档 | 以主文档为准，专项附件只负责细化 |

口头说明、即时聊天和会议讨论不能单独作为开发依据，必须回写为文档、会议纪要或变更记录后才能进入实现。

### 8.1 冲突上报路径

| 冲突影响范围 | 上报对象 | 决策权 |
| --- | --- | --- |
| 仅影响某一模块的字段、状态或接口细节 | 项目联系人 + 对口业务部门 | 项目联系人议定后回写 |
| 跨模块、跨域或影响接口边界 | 项目联系人 → 项目领导小组 | 项目领导小组评审 |
| 涉及合同义务变化（招标正文/验收范围） | 项目联系人 → 项目领导小组 + 采购法务 | 必须经法务复核后处理 |
| 涉及集团统筹层原则 | 项目联系人 → 网信办 → 集团相关部门 | 不得在项目内部消化，必须上报集团 |

### 8.2 冲突留痕方式

每次冲突的处理结果应留痕，最低留痕要求：

- 在 `memory/` 目录或项目周报中记录冲突内容、处理决定、责任人、处理日期
- 文档已修改的，在文档变更日志中标注"为解决 XX 冲突而调整"
- 涉及合同义务变化的，应在合同补充协议或正式会议纪要中记录
- 留痕产出物作为后续审计和复盘的依据，不得仅以微信、邮件等私人通讯方式留存

### 8.3 时效要求

- 简单冲突：5 个工作日内闭环
- 复杂冲突：项目领导小组评审通过后 10 个工作日内闭环
- 紧急冲突（涉及生产事故、合规风险）：按节 7.4 紧急变更特例处理，5 个工作日内补办评审与回写

## 九、当前 V0.x 使用方式

本 V0.x 阶段的使用边界如下：

- 可以作为未来代码仓库筹备、供应商沟通、开发计划拆分的参考
- 可以作为详细设计编写时的约束，提醒各分卷要面向开发落地
- 不作为任何模块直接编码的最终依据
- 不替代后续 `01-11` 详细设计分卷
- 待 `01-11` 完成后，应升级为 V1.0，并补充"代码模块 - 详设章节 - 接口/表/权限引用位置"映射表

## 十、后续升级为 V1.0 的条件

当以下条件满足后，本索引应升级为 V1.0：

- `docs/详细设计/01-11` 全部形成 V1.0 首版
- 数据库逻辑模型、核心单据状态机、NC 接口、权限审批和非功能约束完成交叉评审
- 业务确认轮正式签字或形成明确的项目内替代口径
- 代码仓库命名、技术栈和供应商实施方式已确定
- 每个代码模块能明确映射到至少一份详细设计分卷

## 十二、一句话结论

未来开发应另建代码仓库；当前 `SupplyCore` 仓库作为上游文档和设计依据库。开发真正直接看的是详细设计分卷，但在详细设计尚未完成前，本索引用于先把引用优先级、开工条件、冲突处理和回写机制定下来，避免后续代码、设计和需求各走各的路。
