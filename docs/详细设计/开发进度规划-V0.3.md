# 开发进度规划 V0.3（项目内评审版）

**日期：** 2026-05-05
**范围：** **仅一期**（不含二期）；**NC 联调 + Catio 阶段 B/C 算入总工期**（外部触发型模块作为关键路径节点登记）
**估算依据：** 详设 01~11 V1.0+ + 10A V0.6 + 角色权限矩阵 V1.0 + 开发启动门对照表 V1.0 + 行业经验值（以 .NET / ABP / Vue Web + 小程序双端作为估算模型；不作为技术栈强制要求）+ 详设复杂度系数
**评审定位：** **V0.3 内部评审版**——已根据 Spencer 对 DBA/数据迁移和前端人员配置的二次意见修订；仍不对外发布，后续与中标供应商技术方案校准后再升 V1.0
**浮动范围：** 实际人天会随技术选型 / 团队成熟度 / 一期边界细节 ±30% 浮动；本文档为**立项申请、预算编制、供应商招标技术问询**的输入参考

## 一、文档目的与口径

### 1.1 文档目的

回答四个问题：

1. 完成项目一期总共需要多少人天 / 人月？
2. 按团队规模差异，工期需多久？
3. 关键路径是什么？哪些环节可并行 / 哪些是阻塞？
4. 风险预算占多少？

### 1.2 不做的事

- ❌ 不替代供应商技术方案（供应商对实现细节有自己的工时模型）
- ❌ 不锁定具体上线日（按详设 11 §八 三级退出门驱动，不写固定日期）
- ❌ 不指定具体技术栈（保留 .NET / Java / 前端框架选型空间）
- ❌ 不做团队个体能力评估（按行业平均能力假设）

### 1.3 估算方法

**底层公式：**

```
模块开发工时 = Σ(实体数 × 实体复杂度系数) + Σ(接口数 × 接口复杂度系数)
             + Σ(业务流程数 × 流程复杂度系数) + Σ(前端页面数 × 页面复杂度系数)

总工时 = 模块开发工时合计
       + 测试工时（30%）
       + 项目管理协调（15%）
       + 风险缓冲（30%）

总工期 = 总人月 / 团队规模 + 关键路径串行约束（NC 联调 / OAuth 颁发等）
```

## 二、行业经验值基线（估算模型：.NET / ABP / Vue Web + 小程序双端）

### 2.1 后端工时基线

| 任务类型 | 单位工时 | 说明 |
| --- | --- | --- |
| 简单 CRUD 实体 | **2 人天** | 含 EF Core 模型 + Repository + AppService + 单元测试 |
| 复杂业务实体（含状态机 / 业务规则） | **3-5 人天** | 含状态迁移 / 跨表联动 |
| 外部接口（NC / Catio） | **3-5 人天/接口** | 含校验 + 错误处理 + 重试；不含联调时间 |
| 接口实际联调 | **2 人天/接口** | 与外部系统真实交互调试 |
| 复杂业务流程（跨模块） | **5-10 人天** | 如采购下单跨需求/合同/库存/财务 |
| 数据库 schema + migration | **0.5 人天/实体（基础设施层）** | DbContext 多 schema 配置 |

### 2.2 前端工时基线（Web + 小程序双端）

**Web 端（Vue 假设）：**

| 任务类型 | 单位工时 | 说明 |
| --- | --- | --- |
| 列表 + 表单（CRUD 页） | **2-3 人天/页** | 含查询 / 分页 / 表单校验 |

| 二次确认 / 高敏感弹窗 | **0.5 人天/类** | 复用模式 |

**小程序端（uni-app / Taro / 微信原生假设；技能栈与 Web 不同步）：**

| 任务类型 | 单位工时 | 说明 |

| 项 | 占开发工时比例 | 说明 |

## 五、阶段排期与关键路径

### 5.1 阶段拆分（参考 10A V0.6 §九 + 详设 11 §八）

```
立项 → 阶段 0（招标/选型）→ 阶段 A（基础设施 + 自建审计表）
       ↓
       阶段 B1（SSO + 角色权限）—— 等 OAuth 凭据
       ↓
       阶段 B2（Workflow + 业务模块开发并行）
       ↓
       阶段 C1（自动留痕，依赖 A）+ C2（高敏感拦截，依赖 A+B）
       ↓
       NC 联调（依赖业务方签字 + NC 厂商）
       ↓
       测试 + 验收 → 并行运行（三级退出门）→ 全量上线
```

> **小型团队（5 人）：** 不建议完整承接 Web+小程序双端；若必须 5 人推进，应压缩小程序到“审批 + 扫码 + 快询”最小集，工期按 23-25 个月估算，并要求业务方高频参与。
> **8 人团队：** 可作为压缩配置，但前端与 DBA/迁移资源均偏紧，需强组件复用或将小程序/数据迁移专项外包。
> **9 人团队：** 可行但不作为推荐配置；若 DBA 仍兼职或 Web 前端不足 2 人，计划风险会上升。
> **大型团队（12 人）：** 增 1-2 后端 + 1 测试/实施支持，工期压缩到 11-13 个月，但需匹配的项目管理强度防止“加人减效”。
