# 开发进度规划配套：预算测算说明 V0.1

**定位：** `开发进度规划-V0.4.md` 的配套测算底稿  
**主从关系：** 开发周期、团队规模、工作量口径均以 `开发进度规划-V0.4.md` 为主；本文只在此基础上折算费用区间、解释报价口径、费用边界和预算压缩口径。  
**适用范围：** 只测算一期软件开发实施费用，不含硬件、数据库授权、等保测评、安全加固、第三方厂商专项收费和长期运维费用。

**说明：** 这不是正式报送材料，也不是最终招标控制价，只是作为开发进度规划的配套材料，用于说明“预算大概应该放在哪个区间、如何在预算较低时压缩首期建设范围、为什么不能简单按人员工资估”。

## 2. B 档偏上的压缩原则

预算压缩不建议采用“压低供应商人月单价”的方式。单纯压价容易导致：

- 投入人员水平下降；
- 项目经理、测试、DBA 等角色被弱化；
- 外部接口联调拖延；
- 数据迁移质量下降；
- 后续通过变更追加费用。

更稳妥的方式是：

> **不缩总体规划，缩首期范围；不降核心质量，延后增强能力。**

也就是说，详细设计仍保持完整目标，首期招标和预算控制采用“核心闭环 + 选配扩展”的方式。

### 2.1 首期必须保留的内容

| 类型 | 首期保留内容 | 原因 |
| --- | --- | --- |
| 基础数据 | 组织、仓库、物料、供应商、基础字典 | 后续所有业务模块依赖 |
| 采购主线 | 需求计划、采购申请、采购订单、到货验收 | 物资供应主流程核心 |
| 合同资金 | 合同登记、付款计划、付款申请、三单匹配基础能力 | 管理和财务闭环必需 |
| 库存主线 | 入库、出库、调拨、盘点、库存账、期初库存 | 现场管理和账实一致核心 |
| 权限审批 | SSO 对接、角色权限、基础审批流、高敏感操作留痕 | 合规、安全和流程线上化必需 |
| 核心 NC | 关键凭证、付款、出入库、对账相关接口 | 财务闭环必需 |
| 基础报表 | 库存、采购、合同付款、供应商、预警等核心报表 | 管理层和业务部门日常使用必需 |
| 数据迁移 | 基础档案、期初库存、未结业务数据迁移 | 上线必要条件 |

### 2.2 首期可压缩 / 后置的内容

| 类型 | 压缩方式 | 说明 |
| --- | --- | --- |
| 小程序 | 做最小闭环 | 保留审批、扫码、库存查询、预警；不做完整后台管理 |
| 报表 | 先做核心报表 | 非高频报表、复杂穿透、订阅推送后置 |
| 看板 / 大屏 | 先做 3–4 类核心看板 | 完整 6 类管理看板和大屏展示可作为选配 |
| AI 能力 | 先做入口和基础快询 | 复杂智能分析、自动生成建议、深度多轮工具链后置 |
| NC 接口 | 分优先级联调 | 接口契约可全设计，实际联调按核心接口优先推进 |
| 招采平台对接 | 先做编排和归档 | 与能源集团招采平台的深度自动对接可作为选配 |
| 历史数据 | 迁移当前有效数据 | 长周期历史明细可冷归档或只读查询，不全部结构化迁入 |
| 高级审计分析 | 先保证留痕 | 审计分析报表、复杂追踪视图后续补充 |

## 3. B 档偏上的预算口径

### 3.1 首期预算建议

B 档偏上的首期目标是：在预算较低的情况下，仍保证系统可上线、可运行、可验收。

建议首期软件开发实施费按：

> **380–420 万元**

控制。

这个金额对应的前提是：

1. 供应商按中等实施团队报价，不按高配全国型厂商报价；
2. 小程序、AI、大屏、复杂报表等按最小可用范围建设；
3. NC 接口和招采平台对接分优先级实施；
4. 历史数据迁移控制范围，优先迁移有效数据和未结业务；
5. 详细设计中的完整能力保留，但部分能力作为选配或二期。

### 3.2 完整规划仍保留

B 档偏上不是否定完整规划。完整规划仍建议保留：

> **450–520 万元**

作为后续二期、选配包、预算追加或供应商扩展报价的依据。

可以理解为：

| 层级 | 金额 | 说明 |
| --- | ---: | --- |
| 首期核心包 | **380–420 万元** | 本次预算优先争取，保证核心业务闭环 |
| 选配 / 扩展包 | **约 70–100 万元** | AI、大屏、完整小程序、复杂报表、深度平台联调等 |
| 完整规划合计 | **450–520 万元** | 对应 `开发进度规划-V0.4.md` 的完整建设口径 |

## 4. 为什么不是简单按“10 个人 × 工资 × 月数”算

软件项目报价里的“人月单价”，不是开发人员工资。

供应商的人月报价里一般包含：

- 开发人员工资；
- 社保、公积金、奖金、福利等用工附加成本；
- 项目经理、售前、质量管理、行政管理等间接成本；
- 办公、设备、培训、人员空档期成本；
- 税费和财务成本；
- 企业利润；
- 项目风险缓冲。

所以一个人对外报价 **3.5–4.2 万 / 人月**，并不代表这个人每月工资就是 3.5–4.2 万。

通常员工实际税前工资大约占人月报价的 **45%–60%**。

| 对外人月报价 | 反推税前工资大致区间 |
| ---: | ---: |
| 3.0 万 / 人月 | 1.4–1.8 万 / 月 |
| 3.5 万 / 人月 | 1.6–2.1 万 / 月 |
| 4.0 万 / 人月 | 1.8–2.4 万 / 月 |
| 4.5 万 / 人月 | 2.0–2.7 万 / 月 |
| 5.0 万 / 人月 | 2.3–3.0 万 / 月 |

这个比例比较适合解释预算逻辑：

> 项目报价不是单纯发工资，而是供应商完整交付能力的成本。

## 5. 本项目为什么不能按普通团队低价估算

如果只是普通台账系统，沈阳本地普通实施团队报价可以更低。

但这个项目不是普通 CRUD 系统，复杂点比较多：

1. **NC 接口多**  
   详设里涉及 29 项 NC 接口，还包括推送、回执、重试、对账、异常处理、封账、反结等。

2. **集团平台对接多**  
   要对接 Nova / SSO / Workflow / 组织人员权限体系，不是孤立系统。

3. **双端建设**  
   既要 Web 管理端，也要小程序端。小程序还涉及扫码、现场作业、移动审批、AI 快询、预警触达。

4. **库存和财务逻辑复杂**  
   库存有批次账、库存事务、调拨、盘点、废旧、期初导入；财务有 NC、三单匹配、付款、对账。

5. **数据迁移不能低估**  
   老数据清洗、试迁移、正式迁移、切换演练、回退准备都需要专人负责。

6. **审批和审计要求高**  
   有统一权限、审批流、高敏感操作拦截、审计留痕等要求。

7. **报表、预警、AI 能力不是附带项**  
   这部分涉及报表口径、看板、预警规则、AI Tool 权限和审计。

因此，即使采用 B 档偏上，也不建议把预算压到普通台账系统价格。

## 9. 费用边界

### 9.1 包含在开发实施费里的内容

一般应包含：

- 系统设计深化；
- 后端开发；
- Web 前端开发；
- 小程序端开发；
- NC 等接口开发与联调配合；
- 权限、审批、审计开发；
- 报表、预警、AI Tool 开发；
- 数据迁移脚本与试迁移；
- 测试、缺陷修复、上线支持；
- 基础培训和交付文档。

但采用 B 档偏上后，上述内容会区分为：

- **首期必做**：核心业务闭环、核心接口、基础小程序、基础报表、基础数据迁移；
- **选配 / 后置**：完整小程序、复杂报表、大屏、AI 增强、深度平台联调、复杂审计分析。

### 9.2 不建议包含在开发实施费里的内容

这些建议单独列预算，避免开发费被稀释：

- 服务器、存储、网络、安全设备；
- 国产数据库、中间件、操作系统授权；
- 等保测评、密评、安全加固；
- NC 厂商专项配合费用；
- 能源集团招采平台专项接口费用；
- 长期驻场运维；
- 大规模业务培训；
- 上线后年度维护服务。

## 10. 可以给领导解释的几个要点

这部分只是沟通时可用的解释点，不按正式材料格式展开。

1. **预算不是按工资直接乘出来的。**  
   人月报价包含工资、社保、管理、税费、利润和风险，不等于开发人员实发工资。

2. **本项目不能按普通台账系统估价。**  
   它包含 NC、SSO、审批流、数据迁移、Web + 小程序、报表和 AI，复杂度明显高于普通管理系统。

3. **完整规划 450–520 万，首期可按 380–420 万控制。**  
   预算压缩靠范围分级和阶段化，不靠降低核心质量。

4. **如果预算压到 350 万以下，需要明显缩小首期范围。**  
   否则后期容易通过变更、延期或减少人员投入来平衡成本。

5. **比较稳的首期预算中位数是 400 万左右。**  
   这个数对应 B 档偏上，不是完整规划全部能力。
