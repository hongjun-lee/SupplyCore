## 三、文档分类与使用原则

### 3.1 主设计文档

`01~11` 是当前详细设计的主体依据。开发、测试、供应商实施和验收用例设计，应优先引用对应分卷。

### 3.2 实施配套文档

`10A`、角色权限矩阵、开发实施引用索引、开发启动门对照表用于说明“怎么落地、怎么开工、怎么避免冲突”。这些文档不替代主分卷，但在开发实施阶段具有较高参考优先级。

### 3.3 确认事项与外部动作

`待外部确认事项汇总-V1.0.md` 记录 Spencer 已确认的基础口径，以及 Platform/Catio、OAuth、NC、消息平台、冷归档、外委检修财务口径、直达目录治理等后续阶段性动作。

### 3.4 同步核查材料

`调研政策业务确认-详设同步核查-V0.1.md` 是 2026-05-10 从业务方 P0 答复、政策解析、流程调研反向核查详设 01~11 的补充清单。开发和交底时若遇到外委检修、直达例外、履约保证金、后评价自动联动等问题，应优先结合该核查清单与对应主分卷阅读。

### 3.5 评审留痕

`评审留痕/` 下文件只作为决策链路和评审依据保存。除非主文档发生争议，一般不作为日常开发入口。

## 五、当前已确认的重要口径

以下口径已由 Spencer 于 2026-05-05 项目内确认，并已回写至相关文档：

1. Nova.Platform Q2.1/Q3.2/Q6.1 继续等待 Platform/Catio 明确回复；
2. OAuth client_id / client_secret 由集团 Nova 运维颁发，凭据不写入文档；
3. A-10/A-14 一期不预建投影表，实时拼装，SLA 不达标再评估；
4. 日志冷归档存储类型与恢复 SLA 上线前运维交接阶段再定；
5. 低库存按物料档案 `safety_stock`，超储按储备资金预算 110% + 安全周转周期双维度；
6. 采购基准价以物料标准价 `standard_price` 为主；
7. 一期管理看板最小主题为库存、采购、合同付款、供应商、预警、设备/租赁；
8. 审批金额阈值不在详设写死，按现行制度实施配置；
9. 审批超时只提醒，不自动指定上级代审；
10. 一期不开放加签入口，扩链场景用 Delegate/转办替代；
11. Token 口径：Access Token 2 小时、空闲 30 分钟、Refresh Token 7 天；
12. RPO ≤ 1 小时，RTO ≤ 4 小时；
13. 税率预置 13%、9%、6%、0%，财务实施时确认适用场景；
14. 并行运行不写固定天数，按三级退出门评估；
15. 数据库只写 PostgreSQL 兼容国产信创原则，不指定具体品牌。

完整记录见 `待外部确认事项汇总-V1.0.md`。

2026-05-09 至 2026-05-10 的业务确认 / 政策回填又补充以下关键口径：

1. 采购方式按政策 04 五档矩阵执行，供应商不足走强约束 + WF-PUR-EXC-001；
2. 暂估按 6 个月窗口、D-90 / D-30 两段预警、D 日阻断；
3. 非直达目录物资走直达通道需 WF-DIR-001，≤100 万业务主管，>100 万叠加总经理；
4. 外委检修启用 WF-RPR-001，合同金额原则上不超过设备原值 40%，超限走 WF-CON-OVERLIMIT-001 与 SENS-CON-004；
5. 履约保证金没收纳入 SENS-CON-003，退还 SLA 为验收合格后 5 个工作日；
6. 防爆 / 煤安反馈由系统自动汇总，差评累计触发供应商重评估 WF-SUP-REASSESS-001。

## 七、交付结论

当前详细设计资料已经形成可交付基线：

- `01~11` 分卷已形成完整详细设计主体；
- 权限审批与审计域已通过 `10 V1.2 + 10A V0.6 + 角色矩阵 V1.0` 收口；
- 基础待确认事项和 2026-05-10 政策 / 流程补充项已完成项目内口径整理；
- 开发启动门对照表显示 11 个模块在文档层面无阻塞项；
- 剩余事项均为外部回函、凭据下发、联调或上线前运维动作，不影响当前详细设计交付。

**结论：** 当前资料可作为开发实施交底、供应商启动准备、外部联调准备和招标技术附件支撑依据。  
**边界：** 本结论仅代表文档层面具备交底和启动准备条件，不代表项目已正式进入编码；正式编码仍需立项、团队/供应商安排、开发环境、代码仓库和外部凭据等条件到位。
