## 用语约定

本文件采用以下三级用语表达要求的强制程度：


| 用语    | 含义   | 说明                   |
| ----- | ---- | -------------------- |
| 应（必须） | 强制要求 | 正常情况下必须遵守，经豁免审批后方可例外 |
| 宜（建议） | 推荐做法 | 条件允许时优先采用，可灵活调整      |
| 可（允许） | 参考做法 | 自行决定是否采用，不做强制要求      |

## 一、文档目的

本文档用于明确集团层面对各类业务系统建设、改造、集成与运维的统一要求，作为后续各业务系统需求编制、招标建设、系统集成、实施交付和验收管理的上位指导文件。

本文档重点解决以下问题：

- 各业务系统建设应遵循哪些统一原则
- 哪些能力应按集团统一口径进行规划和落地
- 各系统在权限、数据、接口、运维、安全等方面应满足哪些基本要求
- 如何避免系统重复建设、口径不一、接口割裂和运维分散等问题
- 各方职责分工与例外豁免机制

## 二、适用范围

本原则适用于集团及所属单位新建、改造、扩展、集成的各类业务系统，包括但不限于：采购供应类、财务类、人力资源类、生产经营类、仓储物流类、办公协同类、数据分析与管理驾驶类系统，以及其他需要接入集团统一信息化体系的业务系统。

对于已上线系统，应在后续升级改造过程中逐步对齐本文件要求。

## 三、编制背景

当前集团正在推进信息化管理制度体系建设，目标是逐步形成统一规划、统一标准、统一集成、统一运维、统一安全的建设管理格局。在实际建设过程中，业务系统若缺少统一约束，容易出现以下问题：

- 账号体系分散，登录方式不统一，权限管理割裂
- 主数据来源不一致，同类数据在多个系统重复维护
- 接口标准不统一，集成成本高，后期维护困难
- 系统之间协同能力不足，形成信息孤岛
- 运维、日志、审计、监控机制不统一，风险难以及时发现和处置
- 新系统上线后难以纳入集团统一治理体系

因此，有必要先形成集团层面的统一建设原则，再在各系统项目中具体落实。

## 四、总体建设原则

集团业务系统建设应遵循以下要求：

### 1. 统一规划原则

各系统建设应纳入集团整体信息化规划统筹考虑，避免重复建设、分散建设和各自为政。

### 2. 统一标准原则

权限、主数据、接口、运维、安全、日志等关键要素应采用集团统一标准，减少口径差异。

### 3. 权威来源原则

同一类核心数据应明确唯一权威来源系统，其他系统以引用、同步、映射为主，不应各自重复维护。

### 4. 集成协同原则

各业务系统应优先考虑跨系统协同和数据贯通能力，不应形成新的信息孤岛。

### 5. 安全可控原则

系统建设、部署、接口和运维应满足集团安全管理要求，确保权限受控、数据可控、操作可追溯。

### 6. 信创适配原则

各系统应结合集团国产化、信创建设要求进行规划和建设，满足后续统一适配和升级需要。

### 7. 独立部署、低耦合原则

集团统一公共服务平台作为上位系统，提供组织、人员、权限、SSO 等公共服务。各业务系统应独立部署数据库，通过平台 API 获取公共数据和服务，不与平台共用数据库实例。

> *当前集团统一公共服务平台为 Nova Platform，具体平台能力说明请参见专项规范文件。*

### 8. 适度前瞻原则

在满足当前业务需求的同时，应对统一门户、统一 AI 助理、统一数据治理等能力预留接口和扩展空间。

## 五、职责分工

为确保本原则有效落地，明确各方职责如下：


| 角色        | 职责说明                                 |
| --------- | ------------------------------------ |
| 集团信息化管理部门 | 制定、解释、监督本原则执行；统筹信息化规划；审批例外申请；组织合规性评审 |
| 业务系统主责部门  | 编制需求、组织招标、监督实施；确保本系统按本原则落地           |
| 供应商/实施方   | 按本原则和招标技术要求进行设计、开发、联调和交付             |
| 运维团队      | 系统上线后的统一监控、日志审计、故障处置和安全管理            |


## 六、统一权限管理要求

权限管理是集团统一治理的基础。各业务系统应纳入统一身份与访问管理体系。

### 1. 统一登录要求

- 各业务系统应支持集团统一登录机制
- 最低要求是实现 SSO（单点登录）
- 不应长期保留完全独立、彼此割裂的登录体系

### 2. 统一身份来源要求

- 用户身份信息应与集团统一身份源或权威人员数据源保持一致
- 组织架构、人员信息应来源于人资系统
- 对人员入职、离职、调岗、调部门等变化，应建立同步或联动机制

### 3. 权限模型要求

- 集团、所属单位、部门、岗位、角色、数据范围等权限维度应清晰定义
- 系统可维护本系统业务角色和权限，但不应脱离统一身份体系独立运行
- 高敏感操作应支持审批、确认、日志留痕等控制机制

### 4. 权限变更要求

- 人员岗位调整后，系统权限应能及时更新
- 不得长期依赖手工逐个系统修改权限的方式维持运行
- 对离岗、离职、停用人员应具备及时禁用和收回权限机制

## 七、统一数据标准与权威数据源要求

统一数据标准的核心是明确"谁是权威数据源"。各业务系统涉及的数据，应来源于对应权威系统。

### 1. 基本原则

- 同一类核心主数据应明确唯一权威系统
- 其他系统应优先通过引用、同步、映射方式使用权威数据
- 不允许多个系统各自维护同一类核心主数据并长期并存
- 如暂时未建成权威系统，应明确临时归口系统和后续迁移安排

### 2. 当前明确的权威来源


| 数据类别   | 权威来源     | 其他系统要求      |
| ------ | -------- | ----------- |
| 组织架构   | 人资系统     | 引用或同步，不重复维护 |
| 人员信息   | 人资系统     | 引用或同步，不重复维护 |
| 记账科目   | 财务系统     | 以财务系统口径为准   |
| 物料编码   | 物资供应管理系统 | 引用或映射       |
| 客户/供应商 | 待定       | 待权威系统确立后对齐  |
| 项目编码   | 待定       | 待权威系统确立后对齐  |
| 合同编码   | 待定       | 待权威系统确立后对齐  |
| 资产编码   | 待定       | 待权威系统确立后对齐  |


> *过渡说明：物资供应管理系统当前尚在建设阶段，在其上线前，物料编码权威来源暂由集团物资部门线下台账承担。*

### 3. 数据管理要求

- 各项目在立项、需求、招标、设计阶段，应明确本系统涉及的主数据及其权威来源
- 对需同步的数据，应明确同步方式、更新频率、责任边界和异常处理机制
- 对存在映射关系的数据，应建立稳定映射规则和台账管理机制
- 关键主数据变更应具备日志记录和可追溯能力

## 八、统一接口规范要求

各业务系统之间应采用标准化 API 方式通信，接口报文统一使用 JSON 格式。

### 1. 通信方式要求

- 系统间集成应优先采用 API 接口方式，报文统一使用 JSON 格式
- 接口调用宜基于 HTTP/HTTPS 协议
- 异步场景（大批量同步、事件通知等）宜采用消息队列方式，并明确消息格式与重试策略

### 2. 接口网关要求

集团宜设立统一 API 网关，作为各业务系统间接口调用的统一入口，实现统一鉴权、流量管控、日志记录和异常监控。

### 3. 接口标准要求

后续各系统建设应围绕以下内容形成统一规范：接口地址命名规则、请求方式与报文结构、字符编码规则、时间/状态/枚举等基础字段规则、返回码和错误码规范、版本管理规则、幂等控制要求、超时、重试与异常处理机制。

### 4. 数据同步方式指引


| 同步方式  | 适用场景            | 说明            |
| ----- | --------------- | ------------- |
| 实时推送  | 权限变更、人员变动等高时效场景 | 权威系统变更时主动推送   |
| 定时拉取  | 物料编码、科目等低频变更场景  | 下游系统定时拉取最新数据  |
| 全量+增量 | 初始化+日常变更        | 首次全量同步，后续增量更新 |


### 5. 安全与审计要求

- 接口调用应具备统一或可兼容的鉴权机制
- 应记录接口调用日志、错误日志和关键业务日志
- 关键写操作接口应具备可追溯能力
- 接口异常应支持查询、告警和问题闭环处理

### 6. 文档与测试要求

- 各系统对外提供接口时，应提供完整接口文档
- 应明确字段说明、示例报文、返回说明、错误码说明
- 应提供联调环境、测试数据或测试方案

## 九、统一 AI 入口与智能能力接入要求

集团各类业务系统如涉及 AI 能力建设，应遵循以下核心原则：

### 1. 统一入口与能力开放

- AI 能力应纳入集团统一 AI 入口或 AI 助理平台管理，不应各自建设独立 AI 门户
- 可供 AI 调用的业务能力应以标准化接口对统一 AI 平台开放
- 开放接口应支持结构化参数输入和结果输出，遵循集团统一接口规范

### 2. 权限与数据受控

- AI 查询和辅助办理必须与原业务系统权限体系保持一致
- AI 平台不得绕过业务系统直接访问底层数据库
- 高风险场景（写操作、审批、数据变更）应建立确认、审批和二次校验机制

### 3. 运维、审计与文档

- 接入 AI 入口后的能力应纳入集团统一日志、审计、监控和运维体系
- 提供 AI 接入能力的业务系统应提供完整接口文档和联调支持资料

> *本章仅保留集团层面通用原则。具体系统的 AI 接入范围、Tool 明细、联调和验收要求应另行编制专项规范。参考：`docs/集团统筹/AI接入专题/`*

## 十、统一运维与日志审计要求

### 1. 运维要求

- 系统应支持统一部署、统一监控、统一告警和统一问题管理
- 应具备运行状态监测、故障提示和问题排查能力
- 应具备数据备份、恢复和应急处理能力

### 2. 日志要求

系统应至少具备以下日志能力：用户登录日志、关键业务操作日志、审批流转日志、接口调用日志、异常与错误日志、数据变更留痕。

### 3. 审计要求

- 关键业务操作应可追溯到人、时间、动作和结果
- 敏感操作（权限变更、接口失败、数据修正等）应重点留痕
- 审计信息应满足检查、追责和问题复盘需要

## 十一、统一信创与安全可控要求

### 1. 信创适配要求

各系统应结合集团信创和国产化要求统筹考虑，包括部署环境适配、组件兼容性（数据库优先选用 PostgreSQL 兼容的国产信创数据库）、私有化部署能力和后续平滑迁移能力。

### 2. 安全可控要求

- 系统应满足集团账号安全、权限控制、数据安全和日志审计要求
- 不应引入无法管控、审计、持续运维的关键依赖
- 敏感数据、接口和高风险操作应有明确控制机制

## 十二、对项目建设、招标、实施和验收的约束要求

本文件作为集团层面的上位原则，应贯穿各系统建设全过程。

### 1. 需求与立项阶段

应明确：本系统遵循哪些集团统一原则、涉及哪些权威数据源、是否接入统一权限体系和运维体系。

### 2. 招标与采购阶段

应将本文件要求纳入招标技术要求、供应商响应要求和评审关注点，避免后续因理解不一致导致返工或争议。

### 3. 设计与开发阶段

应明确：统一权限接入方案、权威数据同步或映射方式、API/JSON 接口实现方案、日志审计监控落地方案、AI 助理入口支持或预留方案。

### 4. 实施与联调阶段

应验证：权限接入有效性、数据同步稳定性、接口标准符合性、日志和异常处理可用性、与集团现有体系协同能力。

### 5. 验收阶段

建议将最小合规要求清单中的各项纳入验收检查范围，并按所列验证方式执行。

### 6. 文档引用与分层要求

- 各系统项目应明确所遵循的集团层面文件
- 项目主文档作为建设主体依据，专项附件仅用于补充展开
- 建议各项目形成《文档分层与引用关系说明》，明确文件优先级关系

## 十三、最小合规要求清单

以下内容作为各业务系统的最小合规项：


| 序号  | 要求项     | 级别  | 要求口径              | 验证方式        |
| --- | ------- | --- | ----------------- | ----------- |
| 1   | 统一登录    | 应   | 接入统一权限体系，至少实现 SSO | 统一登录入口测试    |
| 2   | 权威数据源   | 应   | 明确主数据权威来源，不重复维护   | 主数据清单对照审查   |
| 3   | 统一接口    | 应   | 系统间 API 集成        | 接口文档审查+联调测试 |
| 4   | 报文格式    | 应   | 接口报文统一 JSON       | 报文格式抽查      |
| 5   | 日志审计    | 应   | 关键操作必须留痕          | 抽查操作日志完整性   |
| 6   | 运维可控    | 应   | 监控、告警、备份、恢复       | 运维能力清单核查    |
| 7   | 信创适配    | 应   | 国产化、私有化部署         | 部署环境与组件审查   |
| 8   | AI 助理入口 | 宜   | 支持或预留统一 AI 入口     | 接入测试或接口预留确认 |
| 9   | 独立部署    | 应   | 独立数据库，平台 API 低耦合  | 部署架构图审查     |


## 十四、例外与豁免机制

对于确实无法在当前阶段完全满足本原则要求的系统，应按以下机制申请例外：

- **申请主体：** 业务系统建设主责部门提出例外申请，说明无法满足的具体条款及原因
- **审批主体：** 集团信息化管理部门负责审批
- **豁免有效期：** 每次豁免应明确有效期（建议不超过 12 个月），到期后重新评估
- **整改计划：** 申请时应同时提交整改计划，明确后续对齐的时间节点和责任人
- **记录归档：** 所有例外审批记录应纳入集团信息化治理台账统一管理
