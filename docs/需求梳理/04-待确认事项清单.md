# 待确认事项清单（内部工作底稿）

## 一、文档目的

本文档用于汇总物资供应与设备资产管理系统建设过程中，当前尚未最终确定但会直接影响业务规则、系统范围、招标要求、接口实现和上线准备的关键事项。

本文档定位为内部工作底稿，用于需求治理、招标准备和实施前确认，不作为正式随标附件对外发放。

本文档主要用于：

- 组织管理层、业务部门、财务部门和信息化部门开展集中确认
- 作为招标前需求澄清和供应商答疑的重要依据
- 避免将重大分歧和模糊口径带入开发实施阶段
- 形成“谁来确认、何时确认、确认后影响什么”的管理闭环

## 二、使用原则

待确认事项不应只是简单罗列，而应做到：

- **按优先级分类**：先解决影响上线与招标的核心问题
- **按责任归口明确**：每项事项均应明确牵头部门或责任人
- **按影响范围说明**：明确对业务、财务、接口、实施的影响
- **按决策状态跟踪**：区分待确认、确认中、已确认、暂缓处理

建议后续在项目推进中定期更新本清单，作为需求治理台账长期维护。

## 三、优先级说明

### P0：不上线会卡死 / 不确认无法招标

此类事项如不明确，将直接影响：

- 功能边界确定
- 供应商工作量评估
- 开发设计口径统一
- 上线方案可执行性

### P1：影响核算准确性和实施效果

此类事项如不明确，短期内可能不阻断招标，但会影响：

- 财务处理准确性
- 业务执行一致性
- 实施细节落地效果

### P2：可预留、可二期优化

此类事项不建议拖慢一期建设节奏，可在方案中先说明原则、预留能力，后续再细化。

## 五、已明确的关键口径汇总

本批事项已按建议方向确认，关键口径如下：

### 1. 一期建设边界

主要口径：

- 已明确的一期主链路范围是否存在遗漏项或需补充的特殊场景
- 哪些业务纳入一期，哪些明确为二期或预留
- 哪些模块只做接口，不做系统内完整闭环

### 2. 财务核心口径

主要口径：

- 存货计价方法
- 暂估入库规则
- 运费归集方式
- 跨组织调拨财务处理
- 月结、补单、反结规则

### 3. 接口实施边界

主要口径：

- NC 接口方式
- 谁是主数据源
- 映射关系由谁维护
- 接口异常由谁处理
- 上线前历史数据如何初始化

### 4. 成本归集基础

主要口径：

- 成本中心与使用单位完整对应关系
- 哪些领料场景必须填写成本中心
- 安全专项、工程项目、维修项目等特殊归集口径

## 六、确认结果维护机制

为保证后续维护一致性，已确认事项继续按以下机制管理：

### 1. 建立“事项台账 + 会议确认”机制

每周或按阶段召开需求确认会议，逐项确认、逐项留痕。

### 2. 每项事项明确责任归口

避免“大家都知道，但没人拍板”的情况。每项事项至少明确：

- 牵头部门
- 配合部门
- 计划确认时间
- 确认结果记录位置

### 3. 已确认事项要同步回写正式文档

事项确认后，不应只停留在会议纪要中，而应同步更新：

- 项目目标与范围说明
- 业务流程与单据清单
- 主数据与编码规则说明
- 财务/NC 接口需求说明
- 招标需求文件

### 4. 对暂时无法确认的事项要明确处理原则

若短期内无法最终拍板，应明确：

- 是否影响一期招标
- 是否先按“建议方案”写入
- 是否保留系统参数化能力
- 是否推迟到实施阶段最终确认

## 七、建议后续增加的字段

后续为了便于持续维护，可在本清单中补充以下字段：

- 计划确认时间
- 实际确认时间
- 决策结论
- 是否需要更新招标文件
- 是否需要更新制度文件
- 是否需要更新接口规范
- 备注

## 八、一句话总结

待确认事项清单的价值，不在于“列问题”，而在于：

**把影响招标、实施、开发和上线的关键分歧提前显性化、责任化、闭环化。**
