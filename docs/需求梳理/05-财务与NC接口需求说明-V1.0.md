# 财务与 NC 接口需求说明（V1.0）

## 一、文档目的

本文档用于从需求沟通、招标说明和开发指导的角度，明确物资供应管理系统与 NC 财务系统之间的协同边界、接口对象、业务触发点、管理要求和实施重点。

本文档主要用于：

- 与财务部门统一业财协同原则
- 与业务部门明确哪些业务会触发财务动作
- 为招标阶段说明接口建设范围和责任边界
- 为后续开发、实施、联调和验收提供依据

> 说明：本文档为需求层接口说明，偏重“做什么、为什么、何时触发、由谁负责”。如需查看更细的凭证样例、会计分录和技术报文示例，可结合《物资管理与财务接口规范》使用。

> 文档定位：本文档作为财务接口需求主文档，主要面向需求梳理、招标说明和管理沟通；详细规则、凭证分录、报文样例和联调细节，统一以下层配套文档《物资管理与财务接口规范》展开。若当前阶段需要先落地映射字段、状态和科目规则模板，可配套使用《14-NC映射与科目配置模板-V1.0》。

## 二、接口建设背景

物资系统与 NC 财务系统分属不同业务定位：

- **物资系统**负责物料主数据、库存实物流转、业务单据过程管理，以及合同付款计划、付款申请、支付进度跟踪和业务留痕
- **NC 财务系统**负责财务凭证、应付处理、实际资金支付、成本归集和价值核算

两者之间不是替代关系，而是协同关系。

本项目接口建设的核心目标是：

- 让物资业务成为财务核算的可信来源
- 让财务核算结果能够反映真实业务活动
- 减少手工重复录入、手工对账和口径不一致问题

## 三、接口建设总体原则

### 1. 业务主导原则

物资系统是业务发生和实物流转的记录方，财务处理应以真实业务为基础，不应绕过业务系统直接形成脱节数据。

### 2. 主数据先行原则

接口建设必须建立在主数据统一基础上。若物料、单位、组织、成本中心、存货分类不统一，接口将无法稳定运行。

### 3. 映射协同原则

考虑 NC 已有历史存量和使用习惯，建议继续采用：

- 物资系统保持自身物料编码体系
- 与 NC 存货编码建立稳定映射关系
- 通过映射实现业务与财务协同，不强行要求 NC 全量重构

### 4. 同口径原则

以下关键口径必须两系统一致或有明确映射关系：

- 计量单位
- 存货分类
- 成本中心
- 组织口径
- 计价方法
- 暂估规则

### 5. 可追溯原则

所有推送、重推、失败、回执、差异调整都必须可查询、可追溯、可审计。

### 6. 支付执行边界原则

资金计划管理模块负责付款计划、付款申请、付款条件校验、支付进度跟踪和预警，不直接替代 NC 的应付处理和实际资金支付。

付款申请审批通过，不等于款项已经支付；实际支付结果应以 NC 或等效财务系统回写结果为准。

## 四、接口范围说明

一期建议纳入以下接口范围：

### 1. 主数据相关接口

- 物料与 NC 存货编码映射
- 计量单位对照
- 分类对照
- 成本中心对照
- 组织口径对照

### 2. 业务单据触发财务接口

- 采购入库
- 采购退货
- 领料出库
- 退料入库
- 跨组织调拨
- 盘盈盘亏
- 废旧物资处置
- 委托加工
- 月末暂估与冲销相关处理

### 3. 对账与异常处理接口

- 接口执行结果回执
- 失败重推
- 差异清单
- 对账结果清单
- 实际付款结果回写

## 五、主数据接口需求

### 1. 物料映射需求

物资系统中的物料编码与 NC 存货编码之间，应建立稳定映射关系。

需求重点包括：

- 映射关系可建立、可修改、可停用、可追溯
- 映射维护要有责任人和审批机制
- 若 NC 已落地且正式财务接口启用，历史物料上线前需完成初始化映射；若 NC 尚未落地，则先完成映射字段、状态和配置模型准备
- 对于一对多、多对一等特殊场景，需提前明确支持边界
- 若当前阶段 NC 尚未落地，则应先完成映射字段、映射状态、维护流程和配置模型设计，不以形成最终 NC 存货编码台账为前置条件

当前建议方向：

- 一期以 **N:1 映射** 作为主策略进行说明和建设
- 在招标文件中应明确写入，避免供应商理解偏差
- 在 NC 未落地阶段，系统应支持“待映射/未启用”状态，待 NC 账套、存货档案和映射规则确定后再启用正式财务推送

### 2. 单位与分类对照需求

应保证物资系统与 NC 的以下口径一致：

- 主计量单位
- 单位换算规则
- 物料大类/分类与 NC 存货分类关系
- 特殊物资属性与科目归属基础

### 3. 成本中心与组织映射需求

应明确：

- 物资系统使用单位如何对应 NC 成本中心
- 哪些业务单据必须携带成本中心信息
- 跨组织业务使用哪个财务组织口径推送

### 4. 权威数据源与同步边界

应进一步明确以下边界：

- **组织架构、人员信息**：以人资系统为权威源，物资系统与 NC 原则上引用或同步使用，不应各自长期维护组织和人员主档
- **财务核算组织、会计科目、成本中心**：以财务系统/NC 为权威源，物资系统侧负责引用、映射和使用
- **物料编码**：以供应系统为权威源，NC 侧维护映射关系，不反向主导物料编码
- **计量单位字典**：如集团暂无独立权威系统，应在项目启动阶段先形成统一单位字典并明确维护归口，上线后不得双向随意扩展
- **科目设计原则**：生成凭证时涉及的科目规则不应硬编码，应支持按业务类型、组织、物料分类等维度配置；若 NC 暂未落地，可先维护科目规则模板或占位值，待 NC 账套和科目体系确定后完成正式配置；当前项目可先按《14-NC映射与科目配置模板-V1.0》统一收口
- 对关键主数据变更，应同步明确生效时点、影响范围、异常处理方式及既有单据留痕规则

## 六、业务接口需求

### 1. 采购入库接口

**业务含义：** 物资到货并完成验收/质检后，形成入库事实。

**接口需求：**

- 支持推送采购入库相关财务处理
- 支持发票未到时的暂估处理
- 支持正式发票到达后的正式入账或冲销处理
- 支持按供应商、组织、物料、金额等关键要素传递

**重点确认：**

- 暂估价格来源
- 正式入账时差异处理机制
- 三单匹配控制边界

### 2. 采购退货接口

**业务含义：** 已入库物资因质量、数量等问题退回供应商。

**接口需求：**

- 支持红字冲销或退货相关财务处理
- 支持关联原采购入库记录
- 支持部分退货和跨期退货场景（如政策允许）

### 3. 领料出库接口

**业务含义：** 仓库将物资发出供生产、维修、管理等使用。

**接口需求：**

- 支持根据用途、成本中心、物料属性推送相应费用或成本凭证
- 支持一张领料单多行、多成本中心归集
- 支持安全专项物资等特殊标识传递

**重点确认：**

- 成本中心是否必填
- 出库计价方法
- 特殊用途领料的财务归集口径

### 4. 退料入库接口

**业务含义：** 已领出物资因未使用、剩余、计划变更等退回仓库。

**接口需求：**

- 支持对原出库财务结果进行冲销或调整
- 支持与原领料记录关联
- 支持批次物资退回的一致性控制

### 5. 调拨接口

**业务含义：** 物资在仓间或组织间转移。

**接口需求：**

- 同组织内调拨原则上以库存位置变化为主
- 跨组织调拨需根据集团财务政策决定是否形成内部结算或相关凭证
- 如存在在途、差异、损耗，应明确对应处理方式

### 6. 盘点接口

**业务含义：** 盘点后形成盘盈、盘亏或减值处理依据。

**接口需求：**

- 支持盘盈、盘亏的财务处理推送
- 支持差异审批后的正式生效
- 支持记录差异原因和责任信息（如制度要求）

### 7. 废旧物资处置接口

**业务含义：** 对废旧物资进行报废、回收、变卖或销毁。

**接口需求：**

- 支持处置收入、损失、销毁等相关财务处理
- 支持危险品销毁等特殊场景留痕
- 支持处置结果回写和对账

### 8. 月末暂估与冲销接口

**业务含义：** 月末对未到票已入库业务执行暂估，并在后续进行冲销或正式确认。

**接口需求：**

- 支持生成暂估处理清单
- 支持次月冲销或替换正式入账
- 支持与对账和月结流程联动

### 8-1. 委托加工接口

**业务含义：** 物料出库送委托加工，不按普通领料处理；加工费和加工完成入库需与财务核算口径联动。

**接口需求：**

- 支持委托加工物料发出时的财务触发，不按普通领料成本直接结转
- 支持委托加工在制、发出、回库等状态的财务映射
- 支持加工费计入存货成本或按财务政策处理
- 支持委托加工完成入库后的后续核算衔接

**重点确认：**

- 委托加工发出时对应的财务科目口径
- 加工费是单独推送还是并入回库成本
- 委托加工完成回库与普通采购入库的边界

### 9. 合同付款与支付结果回写接口

**业务含义：** 业务侧基于合同条款形成付款计划并发起付款申请，财务侧在 NC 中完成实际支付后，将支付结果回写业务系统。

**接口需求：**

- 支持付款申请与合同、入库、验收、发票等依据关联
- 支持将 NC 的实际付款结果回写至业务系统，更新已付金额、未付余额和支付状态
- 支持预付款、进度款、到货款、验收款、质保金、尾款等分期支付结果回写
- 支持回写付款日期、付款金额、支付单号或等效财务凭据标识
- 支持支付失败、退回、冲销、作废等异常状态回写和留痕

**重点确认：**

- 付款申请审批通过后，是否需要与财务支付申请单建立映射关系
- NC 回写的最小粒度是按付款申请、按合同节点还是按财务支付单
- 部分支付、跨期支付、冲销重付等场景的状态更新规则

## 七、接口时点与状态控制要求

接口建设不能只定义“传什么”，更要定义“什么时候传”。

建议重点明确以下原则：

### 1. 审核通过后才可推送

原则上只有业务事实已经成立的单据，才能推送 NC。

### 2. 已推送成功单据应受控修改

已形成财务处理依据的单据，不应允许任意修改；如需调整，应通过红冲、冲销、补差等受控方式实现。

### 3. 已结账期间单据需严格限制

对已结账期间的补录、反结、重推要有明确权限和审批机制。

### 4. 接口状态应清晰可见

建议至少区分以下状态：

- 待推送
- 推送中
- 推送成功
- 推送失败
- 已重推
- 已作废/已冲销

### 5. 建议建立统一状态矩阵与回执标准

建议至少形成以下四类状态，并建立对应关系：


| 状态类别   | 建议状态                      | 说明             |
| ------ | ------------------------- | -------------- |
| 业务单据状态 | 草稿、待审、已审、已作废、已冲销          | 反映业务事实是否成立     |
| 接口推送状态 | 待推送、推送中、推送成功、推送失败、已重推、已关闭 | 反映接口执行过程       |
| 财务处理状态 | 未接收、已接收、已记账、已退回、已冲销       | 反映 NC 侧处理结果    |
| 期间状态   | 未结账、已结账、已反结               | 反映是否允许补录、重推和调整 |


对于资金计划管理相关单据，建议补充以下支付状态：

- 待申请
- 审批中
- 已审批待支付
- 部分支付
- 已支付
- 支付退回/支付失败

同时建议统一最小回执字段，包括但不限于：

- `interfaceId`
- `sourceBillNo`
- `orgCode`
- `receiptStatus`
- `receiptTime`
- `ncVoucherNo`（如已生成）
- `errorCode`
- `errorMessage`
- `retryNo`

幂等控制建议至少以 `interfaceId + sourceBillNo + orgCode` 作为业务唯一键，并明确：

- 同一键值重复推送且报文一致时，返回幂等成功
- 同一键值重复推送但报文不一致时，返回业务冲突并禁止覆盖
- 已结账期间的单据不得通过“重复推送”规避正常审批与反结流程

## 八、异常处理与对账需求

接口真正难的往往不是“正常推送”，而是异常和差异处理。

一期建议明确以下能力：

### 1. 失败可查

应能查看：

- 哪张单据失败
- 失败时间
- 失败原因
- 返回信息
- 当前处理人

### 2. 失败可重推

应支持：

- 人工重推
- 必要场景下自动补偿重试
- 重推过程留痕

### 3. 差异可对账

应支持形成差异清单，便于物资和财务联合核对，包括：

- 金额差异
- 数量差异
- 映射缺失
- 单位不一致
- 状态不一致

### 4. 问题可闭环

对异常接口，应明确：

- 谁负责分析
- 谁负责修正主数据或业务单据
- 谁负责再次推送
- 谁确认问题已关闭

### 5. 对账与封账操作骨架

建议至少明确以下日常与月末操作责任：


| 环节       | 最低要求                              | 责任方建议         |
| -------- | --------------------------------- | ------------- |
| 每日接口巡检   | 检查前一日失败、超时、未回执接口并闭环               | 物资接口管理员       |
| 每日差异处理   | 对主数据缺失、金额差异、状态不一致逐条处理             | 物资 + 财务       |
| 月末对账前检查  | 未映射、未推送、失败重推、暂估清单、盘盈盘亏、手工调整台账全部核对 | 物资 + 财务       |
| 月结确认     | 对账完成后由双方确认是否封账                    | 物资负责人 + 财务负责人 |
| 封账后补单/反结 | 必须走申请、审批、反结/重推、留痕流程               | 财务 + 信息化      |


建议将以下内容作为封账前最低检查项：

- 主数据映射缺失已清零或已有书面说明
- 接口失败单据已处理完毕
- 暂估清单和冲销清单已确认
- NC 手工调整台账已对齐
- 重大差异已明确原因和责任人

后续建议据此单独形成《财务接口对账与异常处理 SOP》。

## 九、招标与实施阶段建议写入的重点要求

为避免后续供应商理解偏差，建议在招标或技术要求中明确写入以下内容：

### 1. 接口责任边界

需明确：

- 物资系统负责哪些数据准备和业务校验
- NC 侧负责哪些接收、处理和回执能力
- 双方开发商的联调责任如何划分

### 2. 接口实施交付要求

建议要求供应商提交：

- 接口清单
- 字段映射说明
- 状态矩阵与回执字段说明
- 错误码与错误处理机制
- 对账机制说明
- 联调方案
- 测试用例与验收标准

### 3. 初始化与迁移要求

建议明确：

- 历史映射如何导入
- 数据清洗由谁负责
- 上线前如何试跑与核对
- 首月运行期间如何保障业务和财务稳定

| 3   | 暂估与正式入账差异       | 月末暂估入库、次月初冲销；暂估价格与正式发票差异以正式发票调整为准                                                                  | 招标正文 6.7/7.3、04 已确认 9、物资管理与财务接口规范 4 章 |
| 4   | 接口状态矩阵、回执字段与幂等键 | 采用统一状态矩阵（待推送→推送中→成功/失败→人工复核）、统一回执字段格式；幂等键由"sourceBillNo + 业务类型 + 组织 + 期间"等字段组合构成，详见配套《物资管理与财务接口规范》 | 招标正文 7.3(一)、物资管理与财务接口规范               |
| 5   | 接口失败的处理策略       | 自动重试（按指数退避）+ 人工重推 + 异常台账 + 告警推送，失败类异常纳入运维看板                                                        | 招标正文 7.3(三)/10.9、物资管理与财务接口规范 5 章      |

### 10.2 实施阶段需进一步明确事项

以下 2 项属实施配置和主数据治理层面的具体规则，一期招标已约束原则方向，具体执行规则须在实施阶段由相关归口部门进一步明确：


| 序号  | 事项                | 实施阶段须明确内容                                     | 归口部门建议                 |
| --- | ----------------- | --------------------------------------------- | ---------------------- |
| 1   | 计量单位统一字典的维护归口     | 字典具体范围、维护责任人、变更审批流程、与 NC 计量单位的映射关系            | 集团物资管理部（主责）+ 财务部（映射确认） |
| 2   | 集团组织架构与财务核算组织映射规则 | 物资组织（矿厂/仓库）与 NC 核算组织的一一对应或多对一映射表、新增组织时的映射维护流程 | 财务部（主责）+ 网信办（系统配置）     |


> 上述 2 项不影响一期招标技术要求的完整性；供应商应在实施启动会后 2 周内提交配套的配置规则模板，由归口部门填写生效。

## 十一、一期验收关注点

一期验收时，除页面和功能外，应重点核查以下内容：

- 主数据映射完整性（物料 → NC 存货、组织 → NC 核算组织、成本中心对齐）
- 典型业务单据稳定触发财务处理（入库、出库、退料、调拨、合同付款）
- 异常接口具备提示、追踪、重推能力，异常台账可查询
- 对账清单可生成并支持按物料、组织、期间维度钻取
- 月末暂估与冲销流程可端到端执行
- 已推送与已结账单据具备受控机制（禁止直接修改、强制反结流程）

## 十二、一句话总结

财务与 NC 接口建设的核心，不是“做几个接口”，而是：

**建立一套可落地、可追溯、可对账、可招标、可实施的业财协同规则。**
