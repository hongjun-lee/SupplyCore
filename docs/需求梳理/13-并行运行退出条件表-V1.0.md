# 并行运行退出条件表（V1.0）

**版本：** V1.0
**日期：** 2026-04-24
**文档性质：** 项目主文档层实施准备模板
**适用阶段：** 试运行、并行运行、正式切换前评审

---

## 一、文档目的

本文档用于将“并行运行不写固定时长”的原则，转化为可执行、可评审、可留痕的退出条件表。

本文档主要解决：

- 核心批何时可以退出并行运行
- 扩展批何时可以退出并行运行
- 全量正式上线前应满足哪些硬性条件
- 哪些问题属于一票否决项，出现后不得退出并行或必须启动回退评估
- 在 `NC 尚未落地` 的情况下，财务接口相关条件如何替代判断

本文档不以“运行满多少天”作为唯一标准，而以阶段门、证据和问题闭环情况作为退出依据。

---

## 二、使用原则

### 2.1 先看条件，再看时长

并行运行时间可长可短，但退出并行必须满足本表中的必达条件。即使运行时间足够，若关键条件不达标，也不得切换。

### 2.2 核心批、扩展批、全量上线分层判定

核心批退出条件、扩展批退出条件和全量正式上线条件不能混为一套。扩展批的进入和退出，应建立在核心批稳定基础上。

### 2.3 必达项优先

每项条件按以下方式判定：

| 类型 | 含义 | 判定规则 |
| --- | --- | --- |
| 必达项 | 不满足不得退出并行 | 任一项不通过，则整体结论为“不通过” |
| 观察项 | 不一定阻断退出，但必须有整改计划 | 若未通过，应附整改责任人和时限 |

### 2.4 证据先于口头结论

每项退出条件应至少有一种证据支撑，如测试记录、比对记录、日志、报表截图、会议纪要、培训签到或问题闭环单。

### 2.5 NC 未落地场景单独处理

若当前阶段 NC 账套、存货档案和正式科目体系尚未落地，则 NC 相关项不按“正式推送成功”判断，而按“配置模型已就绪、接口开关受控、占位联调或模拟校验通过”判断。

---

## 三、判定结论说明

| 结论 | 说明 |
| --- | --- |
| 通过 | 必达项全部通过，观察项已通过或已形成不影响切换的整改计划 |
| 限制通过 | 必达项通过，但存在少量观察项未闭环，需附限制条件和整改时限 |
| 不通过 | 任一必达项不通过，或触发一票否决条件 |

---

## 四、一票否决条件

出现以下任一情况，不得退出并行运行；若已切换，应立即组织回退评估：

- 存在未关闭的 `P0` 缺陷，且影响核心业务闭环。
- 库存、期初、关键报表或关键接口存在无法解释的重大差异。
- 出现跨组织、跨仓库、跨角色的数据越权或敏感数据泄露风险。
- 上线切换方案、回退预案或护航机制未准备完成。
- 关键用户培训未完成，关键岗位无法独立操作。
- 若 NC 已落地：关键财务接口无法稳定推送，或存在连续失败且无可行补偿路径。
- 若 NC 未落地：接口开关、映射状态和人工替代流程未明确，导致上线后财务联动边界失控。

## 八、NC 未落地时的替代判定口径

当前若 `NC 尚未落地`，则并行运行退出和正式上线评审中，NC 相关项按以下替代口径判断：

| 项目 | 替代判定口径 |
| --- | --- |
| 存货映射 | 映射字段、映射状态、配置入口和接口开关已具备 |
| 科目规则 | 科目规则支持按业务类型、组织、物料分类等维度配置，不硬编码 |
| 接口任务 | 正式推送开关受控，可保持未启用状态 |
| 人工替代路径 | 财务侧临时处理方式、责任人和后续补联调计划已确认 |
| 后续联调条件 | 明确待 NC 落地后再完成正式映射台账、科目配置表和联调验证 |

这意味着：

- `不因为 NC 未落地而阻断当前详细设计和实施准备`
- `也不因为 NC 未落地而放弃配置模型、开关控制和后续联调准备`

## 十、一句话结论

并行运行退出条件表的核心作用，不是证明系统“已经跑了一段时间”，而是证明系统“已经具备从并行运行转入正式运行的条件”。对于当前“全阜矿推进、并行运行不写固定时长、NC 尚未落地”的项目口径，这份表就是后续实施和上线决策的硬边界。
